# 卷四架构设计草案 v1

日期：2026-04-07  
状态：approved in chat  
类型：guidebookv2 卷级结构设计

## 1. 卷四定位

卷四定为：

> **卷四：上下文与状态怎么维持系统持续工作**

它承接卷二和卷三已经建立起来的两层事实：

- 卷二已经解释了一次 agent turn 怎样跑起来
- 卷三已经解释了模型意图怎样被 execution layer 落成现实执行

卷四继续回答的，不再是“一轮怎么跑”，也不再是“能力怎样执行”，而是：

> **Claude Code 为什么不是一轮跑完就散，而是能把工作继续维持下去？**

因此，卷四不是上下文机制清单，也不是 compact 功能总汇，而是：

> **Claude Code 持续工作能力的主线卷。**

---

## 2. 卷四读完后，读者必须留下什么

卷四最重要的目标，不是让读者背下 `compact / projection / collapse / restore` 这些名字，而是让读者脑中留下：

> **持续工作总图。**

也就是说，读完卷四之后，读者至少要能复述：

- 为什么 Claude Code 不是“一轮跑完就重来”的系统
- `messages / context / system prompt / transcript / session` 为什么不是一回事
- 当前可工作的上下文是怎样被拼起来的
- 为什么系统不能把全部历史原样无限送模
- collapse / compaction / projection / restore 这条治理链分别在处理什么问题
- 系统治理的重点为什么不是“改历史本身”，而是“维持当前可工作的视图”
- 为什么经过治理之后，这条工作链仍然能够 restore、继续、接活

源码链路当然要有，但它是为了支撑这张“持续工作总图”，而不是把卷四写成长上下文机制百科。

---

## 3. 卷四的组织原则

### 3.1 先回答“为什么它能持续工作”，再讲机制如何分工

卷四第一篇不应该一上来就讲 compact，也不应该先讲 messages / prompt / transcript 的对象定义。

更好的开法是先回答：

> **为什么 Claude Code 不是一轮跑完就重来，而是一套持续工作的系统？**

只有这个问题先立住，后面的对象边界、上下文构造、治理链机制才不会退化成机制图鉴。

### 3.2 先总后分

卷四整体采用：

- 先立持续工作问题
- 再立核心对象边界
- 再立当前可工作的上下文构造
- 再立“为什么不能原样无限送模”的治理起点
- 再给 collapse / compaction / projection / restore 的总体关系图
- 最后分别拆关键治理与恢复机制

也就是说，卷四不是按旧文目录排列，而是按“持续工作为什么成立”的认知坡度排列。

### 3.3 旧文只能作为素材仓，不能决定新文骨架

卷四必须继续遵守已经写进 skill 的 skeleton-first 规则：

1. 先定这篇在新卷里的职责
2. 先定这篇不讲什么
3. 先定新的 3~5 段主骨架
4. 再回旧文抽素材
5. 最后才参考源码锚点和旧解释

旧文可以贡献：

- 证据
- 局部判断
- 例子
- 机制线索

但不能反向决定：

- 新文章的主骨架
- 章节展开顺序
- 读者的阅读节奏

### 3.4 重点不在“上下文对象很多”，而在“系统怎么一直活着”

卷四当然会涉及：

- messages
- system prompt
- additional contexts
- transcript
- session state
- compact / microcompact / projection / restore

但这些都不是卷四真正统一的对象。

卷四真正统一的问题是：

> **这套系统怎样维持连续工作，而不因为历史膨胀、信息漂移或状态断裂而失去工作能力。**

所以卷四不能把对象百科当终点，也不能把 compact 当主角本身。

### 3.5 治理链的重点是“工作视图”，不是“历史文本本体”

卷四后半的关键判断已经在 brainstorm 中固定：

> **系统治理的不是 transcript 本身，而是当前可工作的视图。**

因此，projection / collapse 一线在卷四里很重要，因为它们负责校正读者最容易产生的误解：

- 不是所有机制都在“改历史”
- 不是所有减负都等于“删消息”
- 系统更关心的是当前 turn 还能否继续在一个可工作的上下文面上推进

### 3.6 restore / session recovery 必须进入卷四主线

如果卷四只讲减负与治理，不讲 restore / recovery，整卷就会停在“怎么防膨胀”，而不是“怎么继续活”。

所以卷四后半必须把 restore / session recovery 拉进来，让读者看到：

- 治理不是单向压缩
- session 不是被动存档
- 系统最终要解决的是持续工作能力，而不是单次节流能力

### 3.7 卷尾必须压回持续工作总图

虽然当前 brainstorm 暂定 8 篇，且第 8 篇以 restore / session recovery 收尾，但卷四最后一篇必须承担卷尾收束职责。

也就是说，restore / recovery 那篇不能只是机制说明，而要同时完成：

- 把前文对象边界、构造链、治理链重新收回一张图
- 说明为什么这条线最终能够重新接活
- 自然导向卷五与卷六，而不抢它们的职责

## 4. 建议篇幅与粒度

卷四不宜过薄，也不宜过碎。

当前聊天中已经确认：

> **工作粒度按 8 篇处理。**

接受区间可以是：

- 8~9 篇

但当前最稳的落点是 8 篇，原因是：

- 少于 7 篇，会把对象边界、构造链、治理链、恢复链压得过糊
- 超过 9 篇，则很容易重新滑向机制目录或源码功能索引
- 8 篇正好可以同时保住：
  - 持续工作总问题
  - 核心对象边界
  - 当前可工作上下文的构造链
  - 治理链起点
  - 治理链总图
  - 视图治理判断
  - compact / compaction 主机制
  - restore / session recovery 收束

## 5. 建议的卷四 8 篇结构

## 01｜为什么 Claude Code 不是“一轮跑完就重来”的系统

### 这篇要解决的问题
- 为什么 Claude Code 不能被理解成一次请求、一份回复的短事务系统
- 为什么“持续工作”在这套系统里是底层能力，不是附加体验
- 为什么卷四必须从持续工作问题本身开篇

### 这篇的卷内作用
- 卷四总起篇
- 持续工作存在理由篇

### 这篇最该留下的判断
> Claude Code 不是一轮跑完就重来的系统；它的很多内部设计都服务于“把工作继续维持下去”。

### 这篇不讲什么
- 不先讲对象百科
- 不先讲 compact 细节
- 不把这一篇写成状态存储实现篇

---

## 02｜messages / context / system prompt / transcript / session 为什么不是一回事

### 这篇要解决的问题
- 卷四里的关键对象为什么不能混成一个“上下文”大桶
- 这些对象分别承担什么角色
- 为什么对象边界不清会让后文治理链全讲糊

### 这篇的卷内作用
- 核心对象边界篇
- 卷四语义校准篇

### 这篇最该留下的判断
> messages、context、system prompt、transcript、session 不是一回事；Claude Code 的持续工作能力正是建立在这些对象被分层对待之上。

### 这篇不讲什么
- 不展开 projection / compact 细节
- 不讲所有 UI 层状态
- 不让对象说明膨胀成术语手册

---

## 03｜当前可工作的上下文是怎么被拼起来的

### 这篇要解决的问题
- 当前 turn 真正送模的上下文面是怎么形成的
- system prompt、additional contexts、messages 等对象怎样一起变成当前工作面
- 为什么“当前可工作的上下文”不等于“全量历史”

### 这篇的卷内作用
- 上下文构造篇
- 从对象边界过渡到治理链的桥接篇

### 这篇最该留下的判断
> Claude Code 每一轮真正依赖的不是裸历史，而是一块被构造出来、可继续工作的当前上下文面。

### 这篇不讲什么
- 不提前把治理链写爆
- 不展开 recovery 逻辑
- 不写成 prompt 拼接流水账

---

## 04｜为什么系统不能把全部历史原样一直送模

### 这篇要解决的问题
- 为什么持续工作天然会遇到历史膨胀问题
- 为什么“把全部历史原样塞给模型”不是稳定方案
- 治理链为什么会在这里变成必要能力

### 这篇的卷内作用
- 治理链起点篇
- 从构造链进入治理问题的转折篇

### 这篇最该留下的判断
> Claude Code 之所以需要上下文治理，不是为了炫技，而是因为持续工作一旦成立，原样无限送模就会变成必然失效的路径。

### 这篇不讲什么
- 不提前深讲每个治理机制
- 不把这篇写成 token 节省技巧文
- 不抢第 5 篇总体关系图的角色

---

## 05｜collapse / compaction / projection / restore 的总体关系图

### 这篇要解决的问题
- 卷四后半几种关键机制彼此是什么关系
- 哪些在处理视图，哪些在处理压缩，哪些在处理恢复
- 为什么这些机制不能被写成一组并排功能点

### 这篇的卷内作用
- 治理链总图篇
- 后半卷锚点篇

### 这篇最该留下的判断
> collapse、compaction、projection、restore 不是并排功能，而是一条围绕“维持可工作上下文”展开的治理与恢复链。

### 这篇不讲什么
- 不在这里把单个机制正文讲完
- 不把 recovery 细节提前写掉
- 不把这篇写成功能目录

---

## 06｜projection / collapse：系统治理的不是 transcript 本身，而是当前可工作的视图

### 这篇要解决的问题
- 为什么卷四真正先要校正的是“治理对象”
- projection / collapse 分别怎样改变当前可工作的视图
- 为什么它们不等于“改历史文本本身”

### 这篇的卷内作用
- 视图治理篇
- 卷四后半最关键的认知校正篇

### 这篇最该留下的判断
> Claude Code 真正优先治理的不是 transcript 本身，而是当前 turn 可继续工作的上下文视图。

### 这篇不讲什么
- 不把 compact 细节抢掉
- 不写成实现细节目录
- 不把 session recovery 提前展开

---

## 07｜compact / compaction：主动减负机制本体

### 这篇要解决的问题
- compact / compaction 在治理链里到底承担什么职责
- 它为什么不是简单删历史
- 它怎样把系统从“越跑越胀”拉回可继续工作的状态

### 这篇的卷内作用
- 主减负机制篇
- 治理链本体篇

### 这篇最该留下的判断
> compact / compaction 的价值不是压短文本，而是主动重组当前工作条件，让系统继续维持可工作的上下文面。

### 这篇不讲什么
- 不和 projection / collapse 再混成一篇
- 不提前展开 restore / session recovery
- 不写成 compact 命令手册

---

## 08｜restore / session recovery：系统怎么把这条线重新接活

### 这篇要解决的问题
- 为什么治理之后还需要恢复与续接
- session recovery 怎样把工作线重新接起来
- 为什么卷四最后必须回到“持续工作为什么成立”而不是停在治理机制上

### 这篇的卷内作用
- 恢复与收束篇
- 卷四卷尾总图篇

### 这篇最该留下的判断
> 卷四真正要解释的不是系统怎样压缩历史，而是它怎样在治理、视图调整和状态续接之后，把工作线继续维持下去。

### 这篇不讲什么
- 不抢卷五的扩展能力正题
- 不抢卷六的命令 / 控制层入口
- 不写成 session 存档说明书

---

## 6. 卷四与相邻各卷的边界

### 6.1 和卷二的边界

卷二讲的是：

- 一次 agent turn 怎么跑起来
- 请求怎样进入 QueryEngine
- 当前 query 怎样组织
- tool_result 怎样回到当前 turn

卷四不重复解释“一轮如何形成”，而是继续解释：

- 为什么这条工作链能维持
- 为什么它不会在长工作中直接爆掉
- 当前工作面为什么总是在被重新构造、治理、续接

### 6.2 和卷三的边界

卷三讲的是：

- 模型意图怎样落成执行
- tool runtime / orchestration / execution / tool_result 怎样成立

卷四不再主讲执行能力，而是接住执行之后留下的一个更深问题：

- 结果、历史、状态、系统提示怎样继续维持工作

### 6.3 和卷五的边界

卷四可能会碰到 session memory、外部记忆、团队记忆等线索，但这些不是卷四主轴。

卷四只讲：

- 持续工作怎样成立
- 当前工作面怎样维持
- 恢复如何发生

卷五才详细展开：

- skills
- agents / subagents
- MCP
- 外部能力接入与扩展平台

### 6.4 和卷六的边界

卷四可能会提到 `compact` 命令、context 命令或 UI 呈现，但不把命令入口与产品控制面当成主角。

卷六才负责回答：

- 用户命令入口
- runtime interface
- command / tool / skill / agent 的产品层整合

## 7. 当前结论

卷四已经在聊天中确认了以下方向：

- 读者残留目标：**持续工作总图优先**，治理链为支撑
- 开篇方式：先回答“为什么 Claude Code 不是一轮跑完就重来”的系统
- 结构策略：**先总后分**
- 粒度：**8 篇**
- 中后段关键判断：**系统治理的不是 transcript 本身，而是当前可工作的视图**
- 卷尾必须回到 restore / recovery，并完成“持续工作总图”的收束

因此，卷四已经具备继续进入写作卡片与后续批量生产的结构前提。
