## 1. 卷五定位

卷五定为：

> **卷五：外部扩展与多代理能力**

它承接卷三和卷四已经建立起来的两层事实：

- 卷三已经解释了 Claude Code 怎样把模型意图落成真实执行
- 卷四已经解释了 Claude Code 怎样维持持续工作，而不是一轮跑完就散

卷五继续回答的，不再是“怎么执行”，也不再是“怎么持续维持”，而是：

> **当真实工作场景变得复杂、多变、不可预置时，Claude Code 怎样把扩展权交给用户，并把系统之外的能力源、工作方式和更多执行者接进 runtime？**

因此，卷五不是扩展名词清单，也不是功能入口目录，而是：

> **Claude Code 适应复杂场景的扩展层主线卷。**

---

## 2. 卷五读完后，读者必须留下什么

卷五最重要的目标，不是让读者背下 `skills / MCP / agents / subagents / hooks / plugins` 这些名字，而是让读者脑中留下：

> **扩展能力总图。**

也就是说，读完卷五之后，读者至少要能复述：

- 为什么复杂场景下，内置能力迟早不够
- 为什么 Claude Code 选择把扩展权交给用户，而不是只靠产品内建能力硬撑
- 这些扩展对象分别在解决什么问题
- 这些对象怎样接入 runtime，而不是平行漂在系统外面
- 为什么 `skills` 和 `agents` 会成为卷五的双核心
- `MCP / hooks / plugins / subagents` 分别怎样补足能力源、接缝层和协作结构
- 为什么这些对象最终共同构成的，不是一组插件名词，而是一层真正的平台能力

源码链路当然要有，但它是为了支撑这张“扩展能力总图”，而不是把卷五写成能力对象百科。

## 3. 卷五的组织原则

### 3.1 先回答“为什么必须有扩展层”，再展开对象主线

卷五第一篇不应该一上来就解释 skill 或 MCP，也不应该先做对象百科。

更好的开法是先回答：

> **复杂场景下，为什么内置能力迟早不够，而 Claude Code 需要把扩展权交给用户？**

只有这个总问题先立住，后面的 `skills / MCP / agents / subagents / hooks / plugins` 才不会退化成名词目录。

### 3.2 先总后分

卷五整体采用：

- 先立复杂场景为什么逼出扩展层
- 再立“为什么把扩展权交给用户”
- 再给一张扩展对象总地图
- 再回到对象主线，分别展开不同扩展机制
- 最后把这些对象重新压回统一的平台层总图

也就是说，卷五不是按旧文堆栈排列，而是按“系统怎样长出适应复杂场景的能力”的认知坡度排列。

### 3.3 前段按系统问题立骨，后段按对象主线展开

卷五前段总论固定为 3 篇：

1. 复杂场景为什么逼出扩展层
2. 为什么 Claude Code 把扩展权交给用户
3. 扩展对象总地图：以 runtime 接入关系为骨，以对象分工解释为面

在这 3 篇之后，主线重新回到对象展开。

### 3.4 后段按对象展开，但不做平均主义配额

卷五后段确认回到按对象展开，顺序固定为：

- `skills`
- `MCP`
- `agents`
- `subagents`
- `hooks`
- `plugins`

但这里不是均匀平均拆篇，而是按对象在平台层中的系统地位、材料厚度和卷内职责分配篇幅。

### 3.5 `skills` 和 `agents` 是双核心

当前 brainstorm 已确认：

> **`skills` 是“能力定制”重轴，`agents` 是“执行者扩展”重轴。**

这意味着卷五真正要解释的，是 Claude Code 最强的两层平台能力：

- 用户怎样把自己的经验、流程、角色结构接进来
- 系统怎样长出更多执行者与协作结构

### 3.6 agents 和 subagents 必须分开成组

这是卷五结构里后来被进一步钉死的重要判断：

- `agents` 组解决的是：**系统怎样长出更多执行者**
- `subagents` 组解决的是：**这些执行者怎样被主执行者继续组织成协作结构**

也就是说：

> **前者是执行者扩展问题，后者是协作结构问题。**

如果把两者混在一起写，subagent 极易被降格成 agent 的一个小功能，而丢掉卷五里最值钱的一层认知递进。

### 3.7 hooks 和 plugins 都不能只写成边角补丁

随着文章级拆分继续下探，卷五后段又形成了两个额外判断：

- `hooks` 不能只停在“平台有接缝层”这种总论上，必须至少有一篇解释各类 hooks 分别在拦什么、接什么、改什么
- `plugins` 不能只停在“系统也支持 plugins”，必须至少有一篇解释它为什么代表更完整的封装、分发和复用形态

这两个判断的意义在于：

> **卷五不是只讲“扩展对象存在”，而是要把这些对象为什么各自独立成立讲透。**

### 3.8 第三篇总地图要以 runtime 接入关系为骨，但不能牺牲对象分工辨识度

关于第 3 篇总论，当前聊天里已经确认：

- `runtime 接入地图` 权重约 **40%**
- `能力对象地图` 权重约 **60%**

因此这一篇的任务不是做纯管线图，也不是做纯名词解释，而是：

> **先把各扩展对象放回 runtime 接入关系里，再交代它们各自是什么、为什么不能混在一起讲。**

### 3.9 卷尾必须压回扩展能力总图

卷五最后不能停在“对象都讲过了”，也不能只留下几类机制的使用感。

卷尾必须收束为：

- 这些对象怎样共同构成 Claude Code 的平台层
- 为什么这套平台层让系统能够适应复杂场景
- 卷五和卷六的边界在哪里：卷五讲能力怎样长出来，卷六讲这些能力怎样被用户入口、控制层和产品结构收成一体

也就是说，卷尾最该留下的是：

> **Claude Code 的强，不只在它会执行，而在它能把能力源、工作方式、执行者和接缝层接成长久可用的平台结构。**

## 4. 建议篇幅与粒度

卷五不宜过薄。

最初聊天里对卷五的判断是：

> **卷五按大卷处理，规模预期为 15+，甚至 20+。**

而在继续细拆后，这个判断又被进一步收实为：

> **卷五当前工作版按 25 篇处理。**

具体构成为：

- 前段总论：**3 篇**
- skills：**5 篇**
- MCP：**3 篇**
- agents：**4 篇**
- subagents：**3 篇**
- hooks：**3 篇**
- plugins：**3 篇**
- 卷尾收束：**1 篇**

之所以允许卷五长到 25 篇，不是为了写厚，而是因为卷五承担的职责本来就重：

- 它既要讲扩展层为什么存在
- 又要讲扩展权为什么交给用户
- 还要讲对象怎样接入 runtime
- 还要覆盖能力源、执行者、协作结构、接缝层与扩展封装

如果压成 6～8 篇，只能得到一个概念总览；如果接受 20+ 的规模，才有可能把平台层真正讲透。

## 5. 当前确定的 25 篇结构

### 5.1 总论 3 篇

1. 为什么复杂场景会逼 Claude Code 长出扩展层
2. 为什么 Claude Code 选择把扩展权交给用户
3. skills / MCP / agents / subagents / hooks / plugins 是怎样接入 Claude Code 的

### 5.2 skills 组 5 篇

4. 为什么 skills 不是“长 prompt”那么简单
5. skills 是怎样把用户经验、流程和角色结构接进 Claude Code 的
6. SkillTool / skills runtime 是怎样接进执行链的
7. 什么样的 skill 才算好的 runtime skill
8. skill、tool、agent 三者的边界到底是什么

关于这组，当前已经额外确认：

- “最佳实践”不做成附录，而单独立成第 7 篇
- 这一篇不是技巧清单，而是 runtime skill 的质量判断篇

### 5.3 MCP 组 3 篇

9. 为什么 MCP 不是“多了一批远程工具”
10. Claude Code 是怎样通过 MCP 接入外部能力源和资源系统的
11. MCP 和 skills / hooks / plugins 分别是什么关系

这组的固定顺序是：

- **先打误解**
- **再讲接入**
- **最后讲边界**

### 5.4 agents 组 4 篇

12. 为什么 agent 不是“另一个工具”
13. Claude Code 是怎样长出更多执行者的
14. agent 的角色化、分工化和委派化是怎么成立的
15. agent、skill、tool 之间的边界和协作关系

这组的固定顺序是：

- **先去误解**
- **再立执行者扩展层**
- **再讲角色化与分工化**
- **最后做边界收口**

### 5.5 subagents 组 3 篇

16. 为什么主 agent 还要派生 subagent
17. subagent 是怎样把执行者体系展开成协作结构的
18. 主 agent / subagent 的职责边界与信息回流

这一组之所以保留 3 篇，而不是压成 2 篇，关键是第 18 篇非常重要：

> **如果没有“职责边界与信息回流”这一篇，subagent 很容易只剩概念感，而看不见它为什么是可控协作结构。**

### 5.6 hooks 组 3 篇

19. 为什么平台层不仅要有能力对象，还要有运行时接缝
20. Claude Code 的 hooks 在 runtime 里到底扮演什么角色
21. Claude Code 里的各类 hooks 分别在拦什么、接什么、改什么

其中第 21 篇的职责已经被明确钉死：

- 必须解释不同 hooks 类型各自的作用
- 不能只停在“有一层 hooks”这种总论

### 5.7 plugins 组 3 篇

22. 为什么有了 skills / MCP / hooks 之后，系统还需要 plugins
23. plugins 和其它扩展对象分别处在什么层级
24. plugins 为什么代表更完整的封装、分发和复用形态

其中第 24 篇的职责也已明确：

- plugins 不只是另一种对象
- 它代表平台层成熟度的一层表现

### 5.8 卷尾 1 篇

25. 为什么这些扩展对象最终会收成一层平台能力

这一篇负责把全卷重新压回扩展能力总图，并自然导向卷六。

## 6. 卷五收尾方式

卷五最后应收成：

> **扩展能力总图。**

这篇收束文需要完成三件事：

1. 把 `skills / MCP / agents / subagents / hooks / plugins` 放回同一张扩展层地图
2. 说明为什么这些对象共同构成的不是“功能列表”，而是 Claude Code 的平台层
3. 自然导向卷六，但不抢卷六的控制层和产品整合主问题

卷五和卷六的边界在这里必须说清：

- **卷五讲：能力怎样长出来、怎样接进来、怎样形成平台层**
- **卷六讲：这些能力怎样被命令入口、控制层接口和产品结构收成今天这个产品**

## 7. 与前后卷的边界

### 7.1 和卷三的边界

卷三已经讲过：

- 模型如何发出 `tool_use`
- 执行层如何把意图落成执行
- SkillTool / AgentTool 作为执行对象怎样进入执行层地图

卷五不重复讲执行层主链，而是继续回答：

- 这些对象为什么会存在
- 它们怎样从“执行对象”长成“平台能力”
- 系统怎样因此获得扩展性和多执行者结构

### 7.2 和卷四的边界

卷四已经讲过：

- Claude Code 怎样维持持续工作
- 当前可工作上下文如何被构造与治理
- 系统怎样通过 restore / recovery 继续接活

卷五不转去讲上下文治理，而是继续回答：

- 在持续工作已经成立的前提下，系统怎样继续长能力
- 用户怎样把外部能力源、工作方式和执行者结构接进这条持续工作的线里

### 7.3 和卷六的边界

卷六才详细展开：

- slash commands
- runtime 命令入口
- skill frontmatter / runtime interface
- command / tool / skill / agent 的边界
- Claude Code 最终为什么呈现为今天这个产品

所以卷五在对象展开时，必须避免：

- 提前把命令入口讲成正题
- 提前把控制层分发逻辑讲成卷五主线
- 提前把产品形态收束做完

卷五的重点始终是：

> **扩展层如何成立。**
