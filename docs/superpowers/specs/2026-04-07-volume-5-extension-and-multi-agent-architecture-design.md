## 1. 卷五定位

卷五定为：

> **卷五：外部扩展与多代理能力**

它承接卷三和卷四已经建立起来的两层事实：

- 卷三已经解释了 Claude Code 怎样把模型意图落成真实执行
- 卷四已经解释了 Claude Code 怎样维持持续工作，而不是一轮跑完就散

卷五继续回答的，不再是“怎么执行”，也不再是“怎么持续维持”，而是：

> **当真实工作场景变得复杂、多变、不可预置时，Claude Code 怎样把扩展权交给用户，并把系统之外的能力源、工作方式和更多执行者接进 runtime？**

因此，卷五不是扩展名词清单，也不是功能入口目录，而是：

> **Claude Code 适应复杂场景的扩展层主线卷。**

---

## 2. 卷五读完后，读者必须留下什么

卷五最重要的目标，不是让读者背下 `skills / MCP / agents / subagents / hooks / plugins` 这些名字，而是让读者脑中留下：

> **扩展能力总图。**

也就是说，读完卷五之后，读者至少要能复述：

- 为什么复杂场景下，内置能力迟早不够
- 为什么 Claude Code 选择把扩展权交给用户，而不是只靠产品内建能力硬撑
- 这些扩展对象分别在解决什么问题
- 这些对象怎样接入 runtime，而不是平行漂在系统外面
- 为什么 `skills` 和 `agents` 会成为卷五的双核心
- `MCP / hooks / plugins / subagents` 分别怎样补足能力源、接缝层和协作结构
- 为什么这些对象最终共同构成的，不是一组插件名词，而是一层真正的平台能力

源码链路当然要有，但它是为了支撑这张“扩展能力总图”，而不是把卷五写成能力对象百科。

## 3. 卷五的组织原则

### 3.1 先回答“为什么必须有扩展层”，再展开对象主线

卷五第一篇不应该一上来就解释 skill 或 MCP，也不应该先做对象百科。

更好的开法是先回答：

> **复杂场景下，为什么内置能力迟早不够，而 Claude Code 需要把扩展权交给用户？**

只有这个总问题先立住，后面的 `skills / MCP / agents / subagents / hooks / plugins` 才不会退化成名词目录。

### 3.2 先总后分

卷五整体采用：

- 先立复杂场景为什么逼出扩展层
- 再立“为什么把扩展权交给用户”
- 再给一张扩展对象总地图
- 再回到对象主线，分别展开不同扩展机制
- 最后把这些对象重新压回统一的平台层总图

也就是说，卷五不是按旧文堆栈排列，而是按“系统怎样长出适应复杂场景的能力”的认知坡度排列。

### 3.3 前段按系统问题立骨，后段按对象主线展开

卷五前段更适合按系统问题拆，但不宜过长。

当前聊天中已经确认：

> **前段总论按 3 篇处理。**

这 3 篇的职责分别是：

1. 复杂场景为什么逼出扩展层
2. 为什么 Claude Code 把扩展权交给用户
3. 扩展对象总地图：以 runtime 接入关系为骨，以对象分工解释为面

在这 3 篇之后，主线重新回到对象展开。

### 3.4 后段按对象展开，但不做平均主义配额

卷五后段确认回到按对象展开，当前顺序为：

- `skills`
- `MCP`
- `agents`
- `subagents`
- `hooks`
- `plugins`

但这里不能预设“每类都写 1 篇”或“每类都写 2 篇”。

更准确的规则是：

> **每类对象至少要有主文，但具体篇幅按材料厚度、系统地位和卷内职责弹性展开。**

当前可以提前确认：

- `skills` 不会小
- `agents` 不会小
- `MCP` 与 `subagents` 大概率也不会轻
- `hooks / plugins` 保留弹性，视 spec 细化时再决定拆分力度

### 3.5 `skills` 和 `agents` 是双核心，不是单点主角

卷五不能只围绕一种扩展对象展开。

当前 brainstorm 已确认：

> **`skills` 是“能力定制”重轴，`agents` 是“执行者扩展”重轴。**

这意味着卷五真正要解释的，是 Claude Code 最强的两层平台能力：

- 用户怎样把自己的经验、流程、角色结构接进来
- 系统怎样长出更多执行者与协作结构

其它对象则分别补足：

- `MCP`：系统外能力源与资源接入
- `subagents`：执行者体系的进一步展开
- `hooks`：运行时接缝与事件注入点
- `plugins`：更完整的平台扩展封装形态

### 3.6 第三篇总地图要以 runtime 接入关系为骨，但不能牺牲对象分工辨识度

关于第 3 篇总论，当前聊天里已经确认：

- `runtime 接入地图` 权重约 **40%**
- `能力对象地图` 权重约 **60%**

因此这一篇的任务不是做纯管线图，也不是做纯名词解释，而是：

> **先把各扩展对象放回 runtime 接入关系里，再交代它们各自是什么、为什么不能混在一起讲。**

### 3.7 卷尾必须压回扩展能力总图

卷五最后不能停在“对象都讲过了”，也不能只留下几类机制的使用感。

卷尾必须收束为：

- 这些对象怎样共同构成 Claude Code 的平台层
- 为什么这套平台层让系统能够适应复杂场景
- 卷五和卷六的边界在哪里：卷五讲能力怎样长出来，卷六讲这些能力怎样被用户入口、控制层和产品结构收成一体

也就是说，卷尾最该留下的是：

> **Claude Code 的强，不只在它会执行，而在它能把能力源、工作方式、执行者和接缝层接成长久可用的平台结构。**

## 4. 建议篇幅与粒度

卷五不宜过薄。

当前聊天中已经确认：

> **卷五按大卷处理，规模预期为 15+，甚至 20+。**

这样判断的原因不是想把卷五写厚，而是卷五承担的职责本来就重：

- 它既要讲扩展层为什么存在
- 又要讲对象分工
- 还要讲对象怎样进入 runtime
- 还要覆盖能力源、执行者、接缝层与扩展封装

如果压成 6～8 篇，极易重新滑回“概念总览”；而如果接受 15+ 的规模，卷五才有空间把平台层真正讲透。

但即便是大卷，也不应机械平均拆分。更好的做法是：

- 前段总论 3 篇固定
- 后段对象主线至少保证 6 组对象存在
- 重点对象允许多篇展开
- 次重点对象保留 1 篇或 2 篇的弹性
- 卷尾保留 1 篇总收束篇

## 5. 建议的卷五结构（第一版）

下面这版不是最终篇数锁死版，而是卷级结构骨架。

## 01｜为什么复杂场景会逼 Claude Code 长出扩展层

### 这篇要解决的问题
- 为什么真实工作不是一组内置工具就能覆盖的稳定环境
- 为什么 Claude Code 一旦进入复杂场景，就必须面对能力不足、环境异质、流程差异和资源外部化
- 为什么卷五必须从“复杂场景”而不是“扩展对象”开篇

### 这篇的卷内作用
- 卷五总起篇
- 扩展层存在理由篇

### 这篇最该留下的判断
> 复杂场景不是 Claude Code 的边缘案例，而是它必须正面适应的常态；扩展层正是在这里变成必要能力。

### 这篇不讲什么
- 不先做对象百科
- 不提前比较所有扩展机制
- 不把这篇写成产品宣传文

---

## 02｜为什么 Claude Code 选择把扩展权交给用户

### 这篇要解决的问题
- 为什么“产品团队持续内建一切能力”不是可持续路径
- 为什么真正适应复杂场景，必须允许用户把自己的能力、工作方式和环境资源接进系统
- 为什么“把扩展权交给用户”是 Claude Code 的核心选择之一

### 这篇的卷内作用
- 卷五总判断篇
- 从场景压力过渡到平台选择的关键转折篇

### 这篇最该留下的判断
> Claude Code 的强，不在于它预装了多少能力，而在于它把扩展权交给了用户，让系统能够贴着真实工作继续生长。

### 这篇不讲什么
- 不写成开放平台宣言
- 不抢后文对象分工图的角色
- 不提前深讲命令入口或产品界面

---

## 03｜skills / MCP / agents / subagents / hooks / plugins 是怎样接入 Claude Code 的

### 这篇要解决的问题
- 卷五涉及的主要扩展对象分别是什么
- 它们在 runtime 里大致处在什么位置
- 为什么这些对象不是一个“扩展大桶”里的同义词

### 这篇的卷内作用
- 扩展对象总地图篇
- 从总论进入对象主线的桥接篇

### 这篇最该留下的判断
> `skills / MCP / agents / subagents / hooks / plugins` 不是并排功能点，而是一组以不同方式接入 runtime、共同构成扩展层的平台对象。

### 这篇不讲什么
- 不把每个对象正文讲完
- 不展开命令层与产品入口
- 不把这篇写成术语词典

---

## 04+｜对象主线（按对象展开，篇幅弹性拆分）

当前确认的对象顺序为：

1. `skills`
2. `MCP`
3. `agents`
4. `subagents`
5. `hooks`
6. `plugins`

### 5.1 skills 组

#### 这组要解决的问题
- skills 为什么不是“长 prompt”这么简单
- skills 怎样把经验、流程、角色结构和工作方法接进 Claude Code
- skill 是怎样从文本指令长成运行时能力组织单元的

#### 这组的卷内作用
- 卷五第一重轴
- “能力定制”主线

#### 当前判断
- 这一组明显不会小
- 至少需要 1 篇主文
- 大概率需要再拆 skill 结构、运行时接入、边界关系等子题

### 5.2 MCP 组

#### 这组要解决的问题
- Claude Code 怎样接入系统之外的能力源和资源系统
- MCP 为什么不只是“远程工具列表”
- MCP 在卷五里和 skills、hooks、plugins 分别怎样区分

#### 这组的卷内作用
- 外部能力源接入主线
- 从本地扩展走向外部环境扩展的关键一组

#### 当前判断
- 这一组大概率不轻
- 至少需要 1 篇主文
- 是否拆出“资源接入 / 协议位置 / runtime 集成”需在细化时判断

### 5.3 agents 组

#### 这组要解决的问题
- Claude Code 怎样从单执行者走向更多执行者
- agent 在卷五里为什么不只是“另一个工具”
- agent 怎样承接角色化、分工化、委派化的系统能力

#### 这组的卷内作用
- 卷五第二重轴
- “执行者扩展”主线

#### 当前判断
- 这一组明显不会小
- 至少需要 1 篇主文
- 大概率要再拆 agent runtime、角色体系、委派结构等子题

### 5.4 subagents 组

#### 这组要解决的问题
- 主 agent 为什么还要派生 subagent
- subagent 怎样把执行者体系展开成协作结构
- subagent 在概念上和 agent 是什么关系，又不是简单同义词

#### 这组的卷内作用
- 多执行者协作结构主线
- 从 agent 扩展到 multi-agent 结构的关键一组

#### 当前判断
- 这组不应被压成 agent 的脚注
- 先后顺序上放在 agents 之后
- 大概率至少 1 篇主文，是否需要多篇展开看细化材料厚度

### 5.5 hooks 组

#### 这组要解决的问题
- Claude Code 运行时有哪些接缝点允许被注入、观察或干预
- hooks 为什么属于扩展层，而不是产品界面的边角配置
- hooks 和 plugins、skills 的关系分别是什么

#### 这组的卷内作用
- 运行时接缝层主线
- 平台扩展点之一

#### 当前判断
- hooks 不与 plugins 合并讲
- 是否 1 篇或多篇，保留弹性

### 5.6 plugins 组

#### 这组要解决的问题
- plugins 为什么更像一种完整的平台扩展封装形态
- plugins 怎样区别于 hooks、skills、MCP
- Claude Code 的平台层为什么需要这种更完整的扩展形态

#### 这组的卷内作用
- 平台扩展封装主线
- 扩展层成熟度的补足对象

#### 当前判断
- plugins 单独成组
- 是否需要多篇拆分，保留弹性

## 6. 卷五收尾方式

卷五最后应收成：

> **扩展能力总图。**

这篇收束文需要完成三件事：

1. 把 `skills / MCP / agents / subagents / hooks / plugins` 放回同一张扩展层地图
2. 说明为什么这些对象共同构成的不是“功能列表”，而是 Claude Code 的平台层
3. 自然导向卷六，但不抢卷六的控制层和产品整合主问题

卷五和卷六的边界在这里必须说清：

- **卷五讲：能力怎样长出来、怎样接进来、怎样形成平台层**
- **卷六讲：这些能力怎样被命令入口、控制层接口和产品结构收成今天这个产品**

## 7. 与前后卷的边界

### 7.1 和卷三的边界

卷三已经讲过：

- 模型如何发出 `tool_use`
- 执行层如何把意图落成执行
- SkillTool / AgentTool 作为执行对象怎样进入执行层地图

卷五不重复讲执行层主链，而是继续回答：

- 这些对象为什么会存在
- 它们怎样从“执行对象”长成“平台能力”
- 系统怎样因此获得扩展性和多执行者结构

### 7.2 和卷四的边界

卷四已经讲过：

- Claude Code 怎样维持持续工作
- 当前可工作上下文如何被构造与治理
- 系统怎样通过 restore / recovery 继续接活

卷五不转去讲上下文治理，而是继续回答：

- 在持续工作已经成立的前提下，系统怎样继续长能力
- 用户怎样把外部能力源、工作方式和执行者结构接进这条持续工作的线里

### 7.3 和卷六的边界

卷六才详细展开：

- slash commands
- runtime 命令入口
- skill frontmatter / runtime interface
- command / tool / skill / agent 的边界
- Claude Code 最终为什么呈现为今天这个产品

所以卷五在对象展开时，必须避免：

- 提前把命令入口讲成正题
- 提前把控制层分发逻辑讲成卷五主线
- 提前把产品形态收束做完

卷五的重点始终是：

> **扩展层如何成立。**

## 8. 当前最稳的结构判断

截至目前，聊天中已经确认的卷五判断如下：

- 卷五严格按总纲走：**外部扩展与多代理能力**
- 开篇总问题：**复杂场景下，内置能力迟早不够，Claude Code 为什么把扩展权交给用户**
- 组织方式：**先总后分**
- 前段总论：**3 篇**
- 第 3 篇总地图：**runtime 接入关系为骨，对象分工解释为面**
- 后段主线顺序：**skills → MCP → agents → subagents → hooks → plugins**
- `skills / agents`：**双核心**
- 总体规模：**15+，甚至 20+**
- 篇幅策略：**按对象分组，但不预先均匀配额**
- 卷尾主收口：**扩展能力总图**

这套判断已经足够支持下一步：

- 写卷五 `README-writing-cards.md`
- 细化对象组内拆篇
- 再进入具体产文顺序安排
