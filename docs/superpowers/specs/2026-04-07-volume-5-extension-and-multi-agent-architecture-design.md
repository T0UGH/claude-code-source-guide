## 1. 卷五定位

卷五定为：

> **卷五：外部扩展与多代理能力**

它承接卷三和卷四已经建立起来的两层事实：

- 卷三已经解释了 Claude Code 怎样把模型意图落成真实执行
- 卷四已经解释了 Claude Code 怎样维持持续工作，而不是一轮跑完就散

卷五继续回答的，不再是“怎么执行”，也不再是“怎么持续维持”，而是：

> **当真实工作场景变得复杂、多变、不可预置时，Claude Code 怎样把扩展权交给用户，并把系统之外的能力源、工作方式和更多执行者接进 runtime？**

因此，卷五不是扩展名词清单，也不是功能入口目录，而是：

> **Claude Code 适应复杂场景的扩展层主线卷。**

---

## 2. 卷五读完后，读者必须留下什么

卷五最重要的目标，不是让读者背下 `skills / MCP / agents / subagents / hooks / plugins` 这些名字，而是让读者脑中留下：

> **扩展能力总图。**

读完卷五之后，读者至少要能复述：

- 为什么复杂场景下，内置能力迟早不够
- 为什么 Claude Code 选择把扩展权交给用户，而不是只靠产品内建能力硬撑
- 这些扩展对象分别在解决什么问题
- 这些对象怎样接入 runtime，而不是平行漂在系统外面
- 为什么 `skills` 和 `agent 主轴` 会成为卷五的双核心
- `MCP / hooks / plugins` 分别怎样补足能力源、接缝层与封装层
- 为什么这些对象最终共同构成的，不是一组插件名词，而是一层真正的平台能力

源码链路当然要有，但它是为了支撑这张“扩展能力总图”，而不是把卷五写成能力对象百科。

## 3. 卷五的组织原则

### 3.1 先回答“为什么必须有扩展层”，再展开对象主线

卷五第一篇不应该一上来就解释 skill 或 MCP，也不应该先做对象百科。

更好的开法是先回答：

> **复杂场景下，为什么内置能力迟早不够，而 Claude Code 需要把扩展权交给用户？**

只有这个总问题先立住，后面的 `skills / MCP / agents / hooks / plugins` 才不会退化成名词目录。

### 3.2 先总后分

卷五整体采用：

- 先立复杂场景为什么逼出扩展层
- 再立“为什么把扩展权交给用户”
- 再给一张扩展对象总地图
- 再回到对象主线，分别展开不同扩展机制
- 最后把这些对象重新压回统一的平台层总图

### 3.3 前段按系统问题立骨，后段按对象主线展开

卷五前段总论固定为 3 篇：

1. 复杂场景为什么逼出扩展层
2. 为什么 Claude Code 把扩展权交给用户
3. 扩展对象总地图：以 runtime 接入关系为骨，以对象分工解释为面

在这 3 篇之后，主线重新回到对象展开。

### 3.4 后段按对象展开，但不做平均主义配额

卷五后段确认回到按对象展开，但当前正式主轴应改成：

- `skills`
- `MCP`
- `agent 主轴（含 subagent / fork worker）`
- `hooks`
- `plugins`

这里不是均匀平均拆篇，而是按对象在平台层中的系统地位、材料厚度和卷内职责分配篇幅。

### 3.5 `skills` 和 `agent 主轴` 是双核心

当前 brainstorm 已确认：

> **`skills` 是“能力定制”重轴，`agent 主轴` 是“执行者扩展”重轴。**

这意味着卷五真正要解释的，是 Claude Code 最强的两层平台能力：

- 用户怎样把自己的经验、流程、角色结构接进来
- 系统怎样长出更多执行者，并把这些执行者继续组织成 worker / fork / subagent 协作结构

### 3.6 agent 和 subagent 不应拆成两个平级对象组

这是在回看 v1 旧文后修正出来的关键判断。

从 v1 的 `10-agenttool / 11-runAgent / 12-forkSubagent / 30-skill-vs-agent` 这条线看：

- `AgentTool` 讲的是委派与多 agent runtime 入口
- `runAgent` 讲的是 agent runtime 装配线
- `forkSubagent` 讲的是在同一上下文里分叉出 worker

这条链表明：

> **subagent 更像 agent 主线内部继续展开出来的一种 worker / fork / 分支运行形态，而不是和 skills / MCP / hooks / plugins 平级的独立对象类。**

因此卷五不再采用：

- agents 组
- subagents 组

而改成一个统一的：

> **Agent 主轴**

这条主轴前半讲执行者扩展，后半讲 worker / fork / subagent 结构。

### 3.7 hooks 和 plugins 都不能只写成边角补丁

随着文章级拆分继续下探，卷五后段还形成了两个额外判断：

- `hooks` 不能只停在“平台有接缝层”这种总论上，必须至少有一篇解释各类 hooks 分别在拦什么、接什么、改什么
- `plugins` 不能只停在“系统也支持 plugins”，必须至少有一篇解释它为什么代表更完整的封装、分发和复用形态

### 3.8 第三篇总地图要以 runtime 接入关系为骨，但不能牺牲对象分工辨识度

关于第 3 篇总论，当前聊天里已经确认：

- `runtime 接入地图` 权重约 **40%**
- `能力对象地图` 权重约 **60%**

因此这一篇的任务不是做纯管线图，也不是做纯名词解释，而是：

> **先把各扩展对象放回 runtime 接入关系里，再交代它们各自是什么、为什么不能混在一起讲。**

### 3.9 卷尾必须压回扩展能力总图

卷五最后不能停在“对象都讲过了”，也不能只留下几类机制的使用感。

卷尾必须收束为：

- 这些对象怎样共同构成 Claude Code 的平台层
- 为什么这套平台层让系统能够适应复杂场景
- 卷五和卷六的边界在哪里：卷五讲能力怎样长出来，卷六讲这些能力怎样被用户入口、控制层和产品结构收成一体

## 4. 建议篇幅与粒度

卷五当前工作版仍按 **25 篇** 处理，但分组已经修正为：

- 前段总论：**3 篇**
- skills：**5 篇**
- MCP：**3 篇**
- agent 主轴：**7 篇**
- hooks：**3 篇**
- plugins：**3 篇**
- 卷尾收束：**1 篇**

之所以允许卷五长到 25 篇，不是为了写厚，而是因为卷五承担的职责本来就重：

- 它既要讲扩展层为什么存在
- 又要讲扩展权为什么交给用户
- 还要讲对象怎样接入 runtime
- 还要覆盖能力源、执行者、worker 协作结构、接缝层与扩展封装

## 5. 当前确定的 25 篇结构

### 5.1 总论 3 篇

1. 为什么复杂场景会逼 Claude Code 长出扩展层
2. 为什么 Claude Code 选择把扩展权交给用户
3. skills / MCP / agents / subagents / hooks / plugins 是怎样接入 Claude Code 的

### 5.2 skills 组 5 篇

4. 为什么 skills 不是“长 prompt”那么简单
5. skills 是怎样把用户经验、流程和角色结构接进 Claude Code 的
6. SkillTool / skills runtime 是怎样接进执行链的
7. 什么样的 skill 才算好的 runtime skill
8. skill、tool、agent 三者的边界到底是什么

### 5.3 MCP 组 3 篇

9. 为什么 MCP 不是“多了一批远程工具”
10. Claude Code 是怎样通过 MCP 接入外部能力源和资源系统的
11. MCP 和 skills / hooks / plugins 分别是什么关系

### 5.4 Agent 主轴 7 篇

12. 为什么 agent 不是“另一个工具”
13. Claude Code 是怎样长出更多执行者的
14. runAgent 为什么像一条 agent runtime 装配线
15. 为什么主 agent 还要派生 subagent
16. forkSubagent 为什么不是“再开一个 agent”
17. 主 agent / worker agent 的职责边界与信息回流
18. agent、skill、tool 之间的边界和协作关系

这里的基本判断是：

- 12-14：执行者扩展主线前半段
- 15-17：worker / fork / subagent 主线后半段
- 18：整条执行者主轴的边界收口篇

### 5.5 hooks 组 3 篇

19. 为什么平台层不仅要有能力对象，还要有运行时接缝
20. Claude Code 的 hooks 在 runtime 里到底扮演什么角色
21. Claude Code 里的各类 hooks 分别在拦什么、接什么、改什么

### 5.6 plugins 组 3 篇

22. 为什么有了 skills / MCP / hooks 之后，系统还需要 plugins
23. plugins 和其它扩展对象分别处在什么层级
24. plugins 为什么代表更完整的封装、分发和复用形态

### 5.7 卷尾 1 篇

25. 为什么这些扩展对象最终会收成一层平台能力

卷五最后应收成：

> **扩展能力总图。**

这篇收束文需要完成三件事：

1. 把 `skills / MCP / agent 主轴 / hooks / plugins` 放回同一张扩展层地图
2. 说明为什么这些对象共同构成的不是“功能列表”，而是 Claude Code 的平台层
3. 自然导向卷六，但不抢卷六的控制层和产品整合主问题

## 7. 与前后卷的边界

### 7.1 和卷三的边界

卷三已经讲过：

- 模型如何发出 `tool_use`
- 执行层如何把意图落成执行
- SkillTool / AgentTool 作为执行对象怎样进入执行层地图

卷五不重复讲执行层主链，而是继续回答：

- 这些对象为什么会存在
- 它们怎样从“执行对象”长成“平台能力”
- 系统怎样因此获得扩展性和多执行者结构

### 7.2 和卷四的边界

卷四已经讲过：

- Claude Code 怎样维持持续工作
- 当前可工作上下文如何被构造与治理
- 系统怎样通过 restore / recovery 继续接活

卷五不转去讲上下文治理，而是继续回答：

- 在持续工作已经成立的前提下，系统怎样继续长能力
- 用户怎样把外部能力源、工作方式和执行者结构接进这条持续工作的线里

### 7.3 和卷六的边界

卷六才详细展开：

- slash commands
- runtime 命令入口
- skill frontmatter / runtime interface
- command / tool / skill / agent 的边界
- Claude Code 最终为什么呈现为今天这个产品

所以卷五在对象展开时，必须避免：

- 提前把命令入口讲成正题
- 提前把控制层分发逻辑讲成卷五主线
- 提前把产品形态收束做完

截至目前，聊天中已经确认的卷五判断如下：

- 卷五严格按总纲走：**外部扩展与多代理能力**

- 前段总论：**3 篇**
- 第 3 篇总地图：**runtime 接入关系为骨，对象分工解释为面**
- 后段主线顺序：**skills → MCP → agent 主轴 → hooks → plugins**
- `skills / agent 主轴`：**双核心**
- `agent / subagent`：**不再拆成两个平级组，而合并成一条执行者主线**

- 总体规模：**25 篇工作版**
- 卷尾主收口：**扩展能力总图**
