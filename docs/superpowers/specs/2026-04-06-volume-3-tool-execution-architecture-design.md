# 卷三架构设计草案 v1

日期：2026-04-06  
状态：approved in chat  
类型：guidebookv2 卷级结构设计

## 1. 卷三定位

卷三定为：

> **卷三：工具系统怎么把模型意图落成执行**

它承接卷二已经建立起来的主循环：

- 请求怎样进入系统
- 当前工作面怎样形成
- 当前判断怎样出现
- 什么时候会从回答路径切到执行路径
- 结果怎样回流当前 turn

卷三继续回答的，不再是“一轮怎么跑”，而是：

> **模型决定调用能力之后，runtime 怎么把这个意图真正落成现实执行？**

因此，卷三不是工具目录，也不是旧工具文章的重新排版，而是：

> **Claude Code 执行能力层的主线卷。**

---

## 2. 卷三读完后，读者必须留下什么

卷三最重要的目标，不是让读者记住所有工具文件名，而是让读者脑中留下：

> **执行层稳定运行图。**

也就是说，读完卷三之后，读者至少要能复述：

- 为什么模型意图不能直接变成现实动作，而必须经过 tool runtime
- `tool_use -> orchestration -> execution -> tool_result` 这条执行主线怎样成立
- Tool 为什么是 runtime 的正式执行接口
- 本地工具家族怎样分别承担命令执行、文件读取、文件修改、材料搜索、能力发现
- SkillTool / AgentTool 为什么也属于执行对象的一种，但不等于卷三主线里的本地工具家族

源码链路当然要有，但它是为了支撑这张执行层总图，而不是替代它。

---

## 3. 卷三的组织原则

### 3.1 先回答“为什么必须有这一层”，再讲这层怎么跑

卷三第一篇不应该一上来就展示工具列表，也不应该直接进入某个具体工具。

更好的开法是先回答：

> **为什么模型意图不能直接变成现实动作，必须经过 tool runtime？**

只有这个问题先立住，后面的执行主线、Tool 抽象、本地工具家族才不会退化成工具图鉴。

### 3.2 先总后分

卷三整体采用：

- 先立执行层存在理由
- 再立执行主线总图
- 再立 Tool 抽象与 orchestration 骨架
- 再分到具体工具家族与执行对象边界
- 最后再压回执行层稳定运行图

也就是说，卷三不是按“先有什么旧文”来排列，而是按执行层的认知坡度来排列。

### 3.3 旧文只能作为素材仓，不能决定新文骨架

卷三从这一版开始，必须遵守已经写入 skill 的 skeleton-first 规则：

1. 先定这篇在新卷里的职责
2. 先定这篇不讲什么
3. 先定新的 3~5 段主骨架
4. 再回旧文抽素材
5. 最后才参考源码锚点和旧解释

因此，旧工具文章可以贡献：

- 证据
- 素材
- 例子
- 局部判断

但不能反向决定：

- 新文章的章节骨架
- 展开顺序
- 读者节奏

### 3.4 本地工具家族必须保留清晰落点

用户已经明确要求：旧文中每种 tool 的展开，在新卷里也都要有对应落点。

因此，卷三不走“压到只剩总论”的路线，而接受较厚的篇幅，只要整体主线不散。

### 3.5 权限线不进卷三主线

权限判断、路径权限、持久化等内容与执行层关系紧密，但为了守住卷三“怎么执行”而不是“怎么控制”的边界，卷三不主讲权限线。

权限系统主展开留给卷六处理。

### 3.6 SkillTool / AgentTool 要出现，但不按卷五方式展开

卷三不能让读者误以为执行层只等于 Bash / File / Search 这些本地工具。

因此，SkillTool / AgentTool 应在卷三中出现，但只作为：

> **执行对象补全篇**

它们在卷三里只回答：

- 为什么 tool runtime 不只接本地工具
- 为什么 skill / agent 也可以成为执行对象的一种
- 它们和本地工具路径的共性与差异是什么

但不在卷三里展开：

- skill 发现 / frontmatter / runtime interface 细节
- agent / subagent / team runtime 主链
- 多代理协作与扩展平台结构

这些仍留给卷五 / 卷六。

### 3.7 卷尾必须压回执行层稳定运行图

卷三如果没有单独卷尾收束，很容易重新散成“几组工具文章”。

因此卷尾必须完成：

- 回收前文各篇职责
- 把执行层压成一张稳定运行图
- 自然导向卷四 / 卷五 / 卷六

---

## 4. 建议篇幅与粒度

卷三不走轻薄路线。

用户已经明确表示：

- 不怕长
- 不希望旧工具文中的每种 tool 展开被压没

因此，卷三接受偏厚结构。

当前推荐：

> **11 篇**

原因：

- 少于 9 篇，会过度压缩工具家族差异，重新滑向总论
- 9~10 篇虽然可行，但在 Bash、文件工具、搜索工具、SkillTool / AgentTool 都要保留落点的前提下，容易重新出现合并过度
- 11 篇可以同时保住：
  - 执行层骨架
  - 本地工具家族样本
  - 执行对象边界补全
  - 卷尾稳定运行图

只要前半骨架足够稳，11 篇并不会变碎；它会更像一卷厚一点的执行能力卷。

## 5. 建议的卷三 11 篇结构

## 01｜为什么模型意图不能直接变成现实动作

### 这篇要解决的问题
- 为什么模型产出的不是现实动作本身
- 为什么 Claude Code 必须存在一个 tool runtime
- 为什么“让模型直接做事”不是系统设计答案

### 这篇的卷内作用
- 卷三总起篇
- 执行层存在理由篇

### 这篇最该留下的判断
> 模型意图不能直接落成现实动作；Claude Code 必须通过 tool runtime 把结构化意图翻译成可控、可执行、可回流的现实操作。

### 这篇不讲什么
- 不直接进入工具图鉴
- 不细讲单个工具
- 不提前展开权限系统

---

## 02｜执行主线总图：`tool_use -> orchestration -> execution -> tool_result`

### 这篇要解决的问题
- 一次能力调用的主线到底怎么跑
- `tool_use` 到 `tool_result` 之间，执行层的基本闭环是什么

### 这篇的卷内作用
- 执行主线总图篇
- 全卷主线锚点篇

### 这篇最该留下的判断
> Claude Code 的执行层不是“工具被调一下”，而是一条从 `tool_use` 到 orchestration，再到 execution，最后回到 `tool_result` 的正式执行链。

### 这篇不讲什么
- 不提前深讲单个工具家族
- 不把 orchestration 与 Tool 抽象细节写爆

---

## 03｜Tool 为什么是 runtime 的正式执行接口

### 这篇要解决的问题
- Tool 到底是什么对象
- 为什么 runtime 要把执行能力收成统一接口
- 为什么不同能力能在同一层被接住

### 这篇的卷内作用
- Tool 抽象本体篇
- 执行对象统一接口篇

### 这篇最该留下的判断
> Tool 不是零散函数集合，而是 Claude Code 把不同执行能力收进同一执行层的正式接口对象。

### 这篇不讲什么
- 不展开某个具体工具
- 不提前讲扩展机制

---

## 04｜orchestration 怎么接住一次 `tool_use`

### 这篇要解决的问题
- 一次 `tool_use` 进入 runtime 后，系统怎样识别、分发、调度
- orchestration 在执行层里的职责到底是什么

### 这篇的卷内作用
- 执行分发篇
- 从抽象接口走向具体执行的桥接篇

### 这篇最该留下的判断
> orchestration 的职责不是“替工具跑代码”，而是把一次 `tool_use` 准确接住、分发给合适执行对象，并把这次执行纳入统一运行链。

### 这篇不讲什么
- 不展开权限系统细则
- 不深讲具体工具内部行为

---

## 05｜BashTool 为什么像执行层的通用执行器

### 这篇要解决的问题
- BashTool 为什么在执行层里地位特殊
- 为什么它不是“普通 shell 包装”
- 它怎样成为最强执行样本

### 这篇的卷内作用
- 本地工具旗舰样本篇
- 命令执行能力篇

### 这篇最该留下的判断
> BashTool 不是给模型一个 shell，而是执行层里最像通用执行器的正式能力对象。

### 这篇不讲什么
- 不主讲权限制度
- 不把整篇写成命令安全篇

---

## 06｜FileReadTool 怎么把现实材料接进当前判断

### 这篇要解决的问题
- 为什么读取文件不是附属动作
- FileReadTool 怎样把现实材料接进执行链与当前判断

### 这篇的卷内作用
- 文件读取能力篇
- 从现实材料到执行链的输入桥接篇

### 这篇最该留下的判断
> FileReadTool 的价值不只是“读文件”，而是把现实材料正式接入当前判断，让执行层拿到可继续工作的真实上下文。

### 这篇不讲什么
- 不和 FileEdit / FileWrite 混成一篇
- 不讲长期上下文治理

---

## 07｜FileEdit / FileWrite 怎么把当前判断落回现实文件

### 这篇要解决的问题
- 修改文件与写入文件为什么属于另一半执行语义
- 当前判断怎样真正落回现实文件系统

### 这篇的卷内作用
- 文件修改与落盘能力篇
- 从判断到现实变更的执行落地点篇

### 这篇最该留下的判断
> FileEdit / FileWrite 的核心不是“改几个字符”，而是把当前判断正式落回现实文件，让执行层真正改变工作对象。

### 这篇不讲什么
- 不回头重讲 FileRead 的角色
- 不写成 API 参数清单

---

## 08｜GrepTool 怎么在现实材料里找东西

### 这篇要解决的问题
- GrepTool 解决的到底是什么问题
- 为什么它属于现实材料检索，而不是能力发现

### 这篇的卷内作用
- 材料搜索能力篇
- 搜索工具家族第一篇

### 这篇最该留下的判断
> GrepTool 的作用不是泛泛“搜索”，而是在现实材料里做高效定位，缩短执行层从问题到证据的距离。

### 这篇不讲什么
- 不和 ToolSearchTool 混成一篇
- 不写成能力选择篇

---

## 09｜ToolSearchTool 怎么在能力面里找该用什么工具

### 这篇要解决的问题
- ToolSearchTool 为什么不是普通搜索
- runtime 怎样把“该用什么能力”也变成可执行的一环

### 这篇的卷内作用
- 能力发现篇
- 搜索工具家族第二篇

### 这篇最该留下的判断
> ToolSearchTool 搜索的不是现实材料，而是执行能力面；它解决的是“接下来该调用什么工具”，而不是“材料里有什么”。

### 这篇不讲什么
- 不回头重讲 GrepTool
- 不提前讲 SkillTool / AgentTool

---

## 10｜为什么执行层不只接本地工具：SkillTool / AgentTool 的位置

### 这篇要解决的问题
- 为什么执行层不只等于 Bash / File / Search
- SkillTool / AgentTool 为什么也算执行对象的一种
- 它们与本地工具路径的共性与差异是什么

### 这篇的卷内作用
- 执行对象补全篇
- 卷三与卷五之间的边界说明篇

### 这篇最该留下的判断
> Claude Code 的执行层不只接本地工具，也可以把模型意图转交给 skill / agent 这样的更高阶执行对象；执行层真正统一的是“可被 runtime 接住的执行对象”。

### 这篇不讲什么
- 不展开 skill frontmatter / runtime interface 细节
- 不展开 agent / subagent / team runtime 主链
- 不展开扩展平台结构

---

## 11｜把整条执行层重新压成一张稳定运行图

### 这篇要解决的问题
- 卷三前十篇共同建立了什么
- 读者最后该怎样把执行层压成一张稳定图
- 卷四 / 卷五 / 卷六 分别从哪里继续接手

### 这篇的卷内作用
- 卷三收尾篇
- 执行层总图回收篇
- 对后续各卷的导流篇

### 这篇最该留下的判断
> 卷三真正建立的，不是若干工具说明书，而是一张执行层稳定运行图：模型意图如何进入 tool runtime，被 orchestration 接住，落成不同类型的执行对象，再把结果接回主循环。

### 这篇不讲什么
- 不重新展开每个工具家族正文
- 不抢卷四 / 卷五 / 卷六的主展开职责

---

## 6. 卷三与后续各卷的边界

### 不吃掉卷四
- 卷三讲：执行层怎么把意图落成现实动作
- 卷四讲：上下文、状态与消息体系怎样让系统长期维持

卷三可以点到结果回流与工作面，但不主讲长期上下文治理。

### 不吃掉卷五
- 卷三讲：SkillTool / AgentTool 为什么也属于执行对象的一种
- 卷五讲：skills / agents / subagents / MCP / plugins 如何成为系统扩展能力

卷三里的 SkillTool / AgentTool 只承担执行对象边界补全，不承担扩展机制主展开。

### 不吃掉卷六
- 卷三讲：执行层怎么跑
- 卷六讲：权限、命令入口、运行时接口、控制层与最终产品整合

因此，权限系统不进卷三主线。

---

## 7. 一句话总结

> 卷三不是工具图鉴，也不是旧工具文章的重新摆放，而是把“模型决定调用能力之后，runtime 怎么把这个意图真正落成现实执行”这条执行主线拆开，再在卷尾重新压成一张执行层稳定运行图。
