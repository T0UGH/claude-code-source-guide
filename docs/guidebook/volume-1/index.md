# 卷一｜Claude Code 系统全景导论

如果第一次看 Claude Code，很容易把它理解成一个会聊天的 CLI，或者一组能调工具的工程功能集合。这样理解不算错，但远远不够。真正的问题是：**Claude Code 到底是一套怎样的系统？**

如果这个问题不先讲清，后面无论去看主循环、工具、上下文，还是 skill、agent、MCP 这些扩展能力，读起来都会像在逛一堆分散源码，而不是在理解同一个 runtime。

所以卷一先不急着深挖某个局部，而是先做一件更基础的事：

> **替读者搭出 Claude Code 的整张系统地图。**

这也是这一卷的任务：

- 先说明 Claude Code 是什么系统
- 再认清它由哪些核心对象组成
- 再看一次请求大致怎么跑成一轮 agent turn
- 再理解模型意图怎么落成执行能力
- 再补上上下文、状态与持续工作的那一层
- 最后回到扩展能力地图，看这套系统为什么还能继续长出新的能力

如果把这一卷压成一句阅读路径，就是：

> **先知道它是什么系统 → 再知道它由哪些对象构成 → 再知道它怎么跑一轮 → 再知道它怎么执行 → 再知道它怎么持续 → 最后知道它怎么扩展。**

---

## 建议阅读顺序

### 01｜Claude Code 到底是什么系统
先亮总图，摆正全书入口姿势。

### 02｜Claude Code 由哪些核心对象组成
把 prompt、command、tool、agent、skill、context、session、runtime 这些对象认全。

### 03｜一次请求是怎么跑成一次 Agent Turn 的
先看动态主线，但只讲总流程，不深挖实现。

### 04｜Claude Code 怎么把模型意图落成执行能力
从全景角度理解 tool、tool_use、runtime 执行能力层。

### 05｜Claude Code 怎么维持上下文、状态与持续工作
解释为什么它不是一问一答后就重启。

### 06｜Claude Code 怎么长出更多能力
以 skill、agent、subagent、MCP、plugin 为主，顺手把后面几卷导航出去。

---

## 这一卷先不讲深什么

卷一不会先把具体工具、主循环实现、上下文压缩机制、MCP 或 plugin 的实现细节当成主菜展开。

不是因为这些不重要，而是因为：

> **卷一的任务是先立图，不是先钻洞。**

这些内容都会在后面的卷里继续拆开。卷一只负责先告诉你：

- 它们分别在系统地图上的什么位置
- 为什么后面值得专门读它们

---

## 这一卷之后接着读什么

如果按主路径往下走，卷一之后最自然的下一步是：

> **卷二：一次 Agent Turn 怎么跑起来**

因为到这时，系统地图已经先立住了，接下来最顺的就是沿着主循环那条动态主线继续往下拆。

---

## 一句话收口

> 卷一最重要的任务，不是讲透某个局部，而是先替读者搭出 Claude Code 的整张系统地图。它应该是一卷少篇幅、多 Mermaid 图、强导航、弱细节泥潭的全景导论卷。
