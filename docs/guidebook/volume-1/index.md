# 卷一｜Claude Code 系统全景导论

## 这一卷在讲什么

如果第一次看 Claude Code，很容易把它理解成一个会聊天的 CLI，或者一组能调工具的工程功能集合。这样理解不算错，但远远不够。真正的问题是：**Claude Code 到底是一套怎样的系统？**

如果这个问题不先讲清，后面无论去看主循环、工具、上下文，还是 skill、agent、MCP 这些扩展能力，读起来都会像在逛一堆分散源码，而不是在理解同一个 runtime。

所以新卷一先不急着深挖某个局部，而是先做一件更基础的事：

> **替读者搭出 Claude Code 的整张系统地图。**

也就是说，这一卷的任务不是先把某个模块讲透，而是先回答下面这些更基础的问题：

- Claude Code 到底是什么系统？
- 它由哪些核心对象组成？
- 一次请求大致是怎么跑成一轮 agent turn 的？
- 模型意图怎么被落成执行能力？
- 上下文、状态与持续工作为什么会成为这套系统的关键部分？
- skill、agent、subagent、MCP、plugin 这些能力又是怎么接进来的？

如果后面几卷是在分层细拆这套系统，那卷一做的就是：

> **先把地图亮出来，再告诉读者后面该沿着哪条路继续往下走。**

所以卷一不是细节卷，而是全景导论卷。

---

## 为什么要重写卷一

旧卷一最大的问题，不是文章质量不够，而是角色太混：

- 有些文章在讲工具层
- 有些文章在讲 agent 和 skill
- 有些文章已经明显进入主循环视角
- 还有些文章更像全书导航或早期探索记录

单篇看都成立，合在一起却会让读者不断切换视角：

- 刚在看 tool
- 下一篇跳到主循环
- 再下一篇又在讲 skill runtime
- 然后又回到命令入口或 agent 边界

这样的问题不是“文章不好”，而是：

> **卷的主问题不够稳定。**

所以这次重写卷一，核心不是局部微调，而是先做一个结构性决定：

> **卷一先不深挖局部，而先把 Claude Code 的系统全景、核心对象、动态主线、执行能力层、状态持续层、扩展能力层讲清楚。**

也正因为如此，新卷一不再需要十几二十篇碎片，而会压成少数几篇大章节。

---

## 这一卷的阅读路径

卷一不会把所有东西平均摊开来讲，而是按一条明确的认知路径往前推：

> **先知道它是什么系统 → 再知道它由哪些对象构成 → 再知道它怎么跑一轮 → 再知道它怎么执行 → 再知道它怎么持续 → 最后知道它怎么扩展。**

也就是说，这一卷虽然是 6 篇文章，但它们不是并列摆放，而是一条有顺序的台阶：

- 先把系统定位摆正
- 再把核心对象认全
- 再把动态主线先看一遍
- 再补执行能力层
- 再补上下文与持续工作层
- 最后回到扩展能力地图，并把后面几卷顺手送出去

这样安排的目的，不是排目录好看，而是让读者脑子里的地图先长完整。后面进入任何一卷时，都不会再像在看散点文件。

---

## 建议阅读顺序

### 01｜Claude Code 到底是什么系统
先亮总图，摆正全书入口姿势。

### 02｜Claude Code 由哪些核心对象组成
把 prompt、command、tool、agent、skill、context、session、runtime 这些对象认全。

### 03｜一次请求是怎么跑成一次 Agent Turn 的
先看动态主线，但只讲总流程，不深挖实现。

### 04｜Claude Code 怎么把模型意图落成执行能力
从全景角度理解 tool、tool_use、runtime 执行能力层。

### 05｜Claude Code 怎么维持上下文、状态与持续工作
解释为什么它不是一问一答后就重启。

### 06｜Claude Code 怎么长出更多能力
以 skill、agent、subagent、MCP、plugin 为主，顺手把后面几卷导航出去。

---

## 这一卷不打算先讲深什么

卷一不会把下面这些内容展开成主菜：

- 具体工具细节，例如 BashTool、FileReadTool、FileEditTool 的实现深挖
- 主循环的完整实现细节，例如 QueryEngine 的逐层调用链
- 扩展机制的具体实现细节，例如 MCP auth、plugin loader、skill frontmatter 字段级分析

不是因为这些不重要，而是因为：

> **卷一的任务是先立图，不是先钻洞。**

这些内容都会在后面的卷里成为真正的展开对象。卷一只负责先告诉你：

- 它们分别在地图上的什么位置
- 为什么后面值得继续拆它们

---

## 这一卷适合谁读

这一卷最适合下面三类读者：

### 1. 想系统理解 Claude Code，而不是只查一个点的人
如果你不是只想找某个函数、某个 feature，而是想知道这套系统整体是怎么组织起来的，那卷一应该先读。

### 2. 已经在用 Claude Code，但总觉得内部结构模糊的人
你可能知道它能调工具、能用 skill、能开 agent、能接 MCP，但这些东西在系统里到底怎么接，脑中还没有一张稳定地图。卷一就是先把这张地图补出来。

### 3. 打算继续往后面几卷深入读的人
卷一最重要的价值之一，是让后面几卷不再像“堆源码文章”，而像顺着同一张系统地图继续往下拆。

---

## 读这一卷前，不需要先懂什么

你不需要预先搞懂：

- QueryEngine 的完整调用链
- BashTool 的所有执行策略
- compact / microcompact / snip 的细节
- MCP / hooks / plugin 的实现机制
- team runtime 的完整生命周期

这些都不是卷一的任务。

你只需要接受一个前提：

> **Claude Code 不是“会聊天的 CLI”，而是一套 agent runtime。**

这个前提一旦放稳，卷一后面的结构就会顺很多。

---

## 读完这一卷，你应该得到什么

如果卷一写得对，读完以后你至少应该拿到下面这些东西：

### 1. 一张稳定的系统总图
你不一定记得每个实现细节，但至少知道 Claude Code 的核心层次是什么。

### 2. 一组核心对象的边界感
你不会再把 tool、skill、agent、prompt、context 混成一团。

### 3. 一条后续阅读路径
你知道卷二到卷六分别会继续讲什么，也知道自己接下来该顺着哪条线继续读。

### 4. 一个比较稳的系统判断
也就是：

> **Claude Code 的关键，不是单点 feature，而是它怎样把模型、runtime、执行能力、上下文和扩展能力编织成一套可持续运行的系统。**

这也是卷一真正的承诺：

> **不是先把某个局部讲深，而是先让后面的每一卷都变得更容易被理解。**

---

## 这一卷之后接着读什么

如果按主路径往下走，卷一之后最自然的下一步是：

> **卷二：一次 Agent Turn 怎么跑起来**

因为到这时，系统地图已经先立住了，接下来最顺的就是沿着主循环那条动态主线继续往下拆。

如果你中途对某个专题更感兴趣，也可以按兴趣跳转：

- 想看执行能力：去卷三
- 想看上下文与持续工作：去卷四
- 想看扩展机制与多代理能力：去卷五
- 想看命令入口与控制层：去卷六

但如果按整本书的主路线读，还是建议从卷二接着走。
