# 卷一弱去重执行表（第一轮）

> 目标不是删篇，而是把相邻文章的职责切开，让读者连读时不觉得“又在重新讲一遍”。

---

## 使用原则

这一轮弱去重，主要做四件事：

1. **保留主文**  
   哪篇是主解释篇，就让它承担定义责任。

2. **副文瘦身**  
   副文不要再重新从头起手，只保留它独有的问题。

3. **导语显式切边界**  
   最有效的方法往往不是删正文，而是改开头 120–180 字。  
   直接告诉读者：
   - 这篇不再解决什么
   - 这篇只解决什么
   - 它和上一篇 / 下一篇怎么分工

4. **增加交叉引用**  
   在篇末或导语里写清：
   - 想看定义层，去哪篇
   - 想看执行层，去哪篇
   - 想看写法判断，去哪篇

## 一、`01 / 02`

### 当前问题
两篇都在承担 tool 总图 / tool 是什么 的功能。

### 重叠点
- 都在解释 tool 不是零散命令
- 都在立 tool 作为系统对象的认知
- 都带“总览性”气质

### 建议分工

#### `01-tool-overview-and-bash-entry.md`
定位为：

> 卷一正文入口 + 为什么 tool 是最适合的第一入口

重点回答：
- 为什么卷一从 tool 起手
- BashTool 为什么是最佳切入口
- tool 层为什么是 runtime 的底座之一

#### `02-tool-system-overview.md`
定位为：

> `Tool.ts` 如何把 tool 定义成统一运行时对象

重点回答：
- Tool 类型长什么样
- buildTool / ToolUseContext / permission / render 协议怎么组成统一抽象
- 这是制度层，不是阅读入口层

### 操作建议
- `01` 减少对 `Tool.ts` 细节的展开
- `02` 开头明确写：这篇不是再讲“为什么先看 tool”，而是讲“tool 被定义成什么”

## 二、`15 / 16 / 17`

### 当前问题
skill 段起手三连，容易让人感觉连续三篇都在重新定义“skill 怎么进系统”。

### 重叠点
- 都在讲 skill 进入 runtime
- 都在解释 skill 不只是 markdown
- 都涉及“入口 / 定义 / 执行”三个词，但边界还没完全拉开

### 建议分工

#### `15-skilltool-bridge.md`
定位为：

> skill 在整套 runtime 中的桥位

回答：
- 为什么 skill 不直接被当普通 prompt 处理
- SkillTool 为什么是桥
- 它连接了哪些系统：command、tool runtime、fork / inline

#### `16-loadskillsdir.md`
定位为：

> skill 定义层：什么算 skill，skill 从哪来

回答：
- skill 来源
- frontmatter 何时开始有运行语义
- `loadSkillsDir` 如何把 skill 压进统一 command 系统

#### `17-skilltool-execution-entry.md`
定位为：

> skill 执行层：已定义好的 skill 如何被真正跑起来

回答：
- 调用合法性检查
- inline / fork 分流
- 可授权、可观察、可追踪的执行路径

### 操作建议
- `15` 少谈 load / parse 细节
- `16` 少谈 execute / invoke 路径
- `17` 开头直接承接：“上一讲已经定义好了 skill，这一讲只讲怎么执行”

## 三、`21 / 22 / 23`

> 这是第一轮最该切的一组。

### 当前问题
三篇都围着 frontmatter / runtime / 好 skill 打转，递进有，但读者容易糊。

### 建议分工

#### `21-skill-frontmatter-fields.md`
定位为：

> 字段参考页

应该尽量像手册，而不是像长正文。

保留：
- 最小字段
- 常用字段
- 进阶字段
- 每个字段一句解释 + 极短例子

删弱：
- 过多方法论铺垫
- 和 `22`、`23` 重复的“运行时意义大论述”

#### `22-frontmatter-runtime-interface.md`
定位为：

> frontmatter 为什么不是注释，而是运行时接口

重点：
- parse / 容错 / degrade
- 字段如何进入 runtime 语义
- 为什么它不是文档元数据

少做：
- “怎么写好一个 skill”的泛化建议

#### `23-good-runtime-skill.md`
定位为：

> 如何根据前两篇的理解，判断一个 skill 写得稳不稳

重点：
- 最小权限
- 最小而明确的 frontmatter
- 正文写成动作框架，不是愿望清单
- inline / fork 何时该选

少做：
- 再回头详细解释字段本体

### 操作建议
- `21` 变短
- `22` 变硬
- `23` 变成判断框架

## 四、`25 / 26`

### 当前问题
都是 bundled skill 案例，且都带“官方工程哲学”味道，容易镜像。

### 建议分工

关键词：
- 验证 = 运行 + 观察 + 留证据
- 不等于只看代码
- 不等于只跑测试

#### `26-debug.md`
主轴固定成：

> Claude Code 对“诊断”的定义

关键词：
- 当前 session
- debug log
- 动态 prompt
- 现场信息优先

### 操作建议
- `25` 多强调“验证边界”
- `26` 多强调“诊断现场”
- 避免两篇都写成“官方又给了一个很有工程味的 skill”

## 五、`28 / 29 / 30`

### 当前问题
后段三篇连续讲 skill 发现 / 感知 / 边界，容易连成一团。

### 建议分工

#### `28-skill-name-description.md`
定位为：

> skill 信息何时暴露给模型

核心问题：
- 开局是否全量注入
- `name` / `description` / `when_to_use` 以什么形式先进入上下文
- 完整 skill 正文何时才展开

#### `29-when-to-use-vs-description.md`
定位为：

> 在已暴露的简介层里，哪个字段更影响发现命中

核心问题：
- `description` 定义“这是什么”
- `when_to_use` 定义“什么时候想起它”
- 自动发现里为什么 `when_to_use` 更关键

#### `30-skill-vs-agent.md`
定位为：

> 就算 skill 已能被发现，为什么仍需要 agent 作为执行主体

核心问题：
- 发现层 vs 执行层
- 方法组织 vs 工作承接
- skill / agent 为什么不是简单轻重之分

### 操作建议
- `28` 专讲“暴露时机”
- `29` 专讲“发现命中”
- `30` 专讲“系统分工”

### `30-34 / 35-37`（旧编号口径）

如果按旧编号映射表理解：

- `30-34` 更像“skill 写法 / bundled skill / 小结”
- `35-37` 更像“发现机制 / 分工澄清”

这个切法本身是对的。

### 后续目录层建议
如果后面继续整理，建议把 skill 段进一步拆成四个小层：

1. 接入层
2. 接口层
3. 范式层
4. 发现与分工层

这样读者会舒服很多。

## 七、推荐执行顺序

建议按下面顺序推进，投入产出最高：

1. `21 / 22 / 23`
2. `15 / 16 / 17`
3. `28 / 29 / 30`
4. `25 / 26`
5. `01 / 02`

原因：
- 前三组最影响连读体感
- 后两组虽然也要切，但紧迫度略低

## 八、一句话总结

这一轮弱去重的本质不是删内容，而是把每篇的职责从：

> 都在谈同一片 skill 世界

切成：

> 每篇只守自己那一层
