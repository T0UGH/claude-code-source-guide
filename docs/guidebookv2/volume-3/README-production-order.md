# 卷三生产顺序说明

> 用途：记录卷三当前确认的生产顺序、分回合策略、串行 / 并行边界，避免后续起稿时重新发散。

---

## 一、卷三当前总前提

卷三已确定为：

> **卷三：工具系统怎么把模型意图落成执行**

主问题是：

> **模型决定调用能力之后，runtime 怎么把它落成真实执行？**

当前已经确认的执行约束：

- 按**执行主线**拆，不按旧工具文目录拆
- 采用**先总后分**结构
- 旧文只能作为素材仓，不能反向决定新文骨架
- 本地工具样本必须保留清晰落点
- 权限线不进卷三主线，留给卷六
- SkillTool / AgentTool 在卷三里作为**执行对象补全篇**出现
- 卷尾必须压回一张**执行层稳定运行图**

---

## 二、卷三 11 篇当前顺序

1. 为什么必须有 tool runtime
2. 执行主线总图：`tool_use -> orchestration -> execution -> tool_result`
3. Tool 为什么是 runtime 的正式执行接口
4. orchestration 怎么接住一次 `tool_use`
5. BashTool 为什么像执行层的通用执行器
6. FileReadTool 怎么把现实材料接进当前判断
7. FileEdit / FileWrite 怎么把当前判断落回现实文件
8. GrepTool 怎么在现实材料里找东西
9. ToolSearchTool 怎么在能力面里找该用什么工具
10. 为什么执行层不只接本地工具：SkillTool / AgentTool 的位置
11. 把整条执行层重新压成一张稳定运行图

---

## 三、推荐生产顺序：3 个回合

当前建议不要 11 篇一口气全并。

最稳的方案是：

> **3 个回合完成卷三。**

原因：

- 前半 4 篇承担卷级骨架，不适合并行主写
- 中段 5 篇属于本地工具家族，适合在统一骨架下分批并行
- 最后 2 篇承担执行对象补全与卷尾收束，不适合和中段工具样本混写

---

## 回合 1：先立执行层骨架

### 建议方式
**串行**，不并行。

### 本回合包含文章
1. 为什么必须有 tool runtime
2. 执行主线总图：`tool_use -> orchestration -> execution -> tool_result`
3. Tool 为什么是 runtime 的正式执行接口
4. orchestration 怎么接住一次 `tool_use`

### 为什么必须串行
这 4 篇不是普通样本，而是卷三前半的骨法：

- 第 1 篇决定卷三为什么存在
- 第 2 篇决定整卷执行主线图
- 第 3 篇决定 Tool 的统一抽象
- 第 4 篇决定后面具体工具样本怎样被接住

如果这 4 篇并行，最容易出现：

- 02 和 03 互抢“总图 vs 抽象”
- 03 和 04 互抢“接口 vs orchestration”
- 04 过早滑回旧工具文的实现细节

所以这组应该：

> **串行跑，边写边校准边界。**

### 本回合结束后要检查什么
- 前四篇的边界是否清楚
- 执行主线与 Tool 抽象是否互不抢活
- orchestration 是否已经成为后面工具家族的统一观察框架

---

## 回合 2：铺本地工具家族

### 建议方式
**分批并行**，优先使用 subagent。

### 本回合包含文章
5. BashTool 为什么像执行层的通用执行器
6. FileReadTool 怎么把现实材料接进当前判断
7. FileEdit / FileWrite 怎么把当前判断落回现实文件
8. GrepTool 怎么在现实材料里找东西
9. ToolSearchTool 怎么在能力面里找该用什么工具

### 推荐并行拆法
不要 5 篇一起全开。

建议拆成两波：

#### 第一波
- 05｜BashTool
- 06｜FileReadTool
- 08｜GrepTool

#### 第二波
- 07｜FileEdit / FileWrite
- 09｜ToolSearchTool

### 为什么这样拆
- 07 最好看过 06 的最终边界后再落，避免“读 / 改写”重新混成一篇
- 09 最好在 08 定完后再写，避免两篇都写成“搜索篇”

所以回合 2 更准确地说是：

> **一个工具家族回合，内部采用两波并行。**

### 哪些最适合 subagent
- 05 BashTool
- 06 FileReadTool
- 07 FileEdit / FileWrite
- 08 GrepTool
- 09 ToolSearchTool

这些篇的共性是：
- 卷级骨架已定
- 职责边界相对清楚
- 每篇主要面对自己的执行语义问题
- 适合匿名 reader review 收紧

### 本回合结束后要检查什么
- Bash / File / Search 三组能力是否各守其位
- FileRead 与 FileEdit / FileWrite 是否已经清晰分开
- GrepTool 与 ToolSearchTool 是否已经清晰分开
- 本地工具家族有没有重新长成旧工具目录的影子

---

## 回合 3：补执行对象边界 + 卷尾收束

### 建议方式
**串行主写**，必要时再做并行 review。

### 本回合包含文章
10. 为什么执行层不只接本地工具：SkillTool / AgentTool 的位置
11. 把整条执行层重新压成一张稳定运行图

### 为什么不建议直接并行正文
这两篇天然互相依赖：

- 10 要决定“执行层不止本地工具”的边界意义
- 11 要把整卷回收成稳定总图

如果它们同时写，最容易出现：

- 11 抢写 10 的边界意义
- 10 过早开始做卷尾总结

所以更稳的方式是：

1. 先串行写 10
2. 再写 11
3. 必要时对 10 / 11 做一次并行 reader review 或横向 review

### 本回合结束后要检查什么
- SkillTool / AgentTool 是否被写成“执行对象补全篇”，而不是卷五提前版
- 卷尾是否真正压成执行层稳定运行图
- 卷尾是否自然导向卷四 / 卷五 / 卷六

---

## 四、不推荐的做法

### 1. 不建议 11 篇一口气全并
这样会让：
- 前半骨架还没稳定
- 后半工具样本已经铺开
- 主会话最后收口成本过高

### 2. 不建议把回合 1 并行化
卷三前四篇承担的是卷级定义、抽象边界和执行骨架，不适合多篇同时主写。

### 3. 不建议把 10 / 11 和本地工具家族混成一个回合
这样会把：
- 本地工具语义
- 非本地执行对象边界
- 卷尾收束

三件事挤在一起，卷三后半会明显变糊。

## 五、runbook 化执行规则

这份文件不只是讨论摘要，而是卷三当前的**执行协议**。

后续如果要稳定跑卷三，默认按下面的 runbook 执行：

### 1. 启动格式
当主会话或执行会话收到类似下面的指令时：
- 跑卷三回合 1
- 继续卷三下一回合
- 按卷三 runbook 执行

默认含义是：
- 读取本文件
- 读取卷三写作卡片
- 按本文件定义的顺序、串并行边界和回合检查来执行
- 不重新发散顺序，不临时改回合拆法

### 2. 每篇的默认流水线
卷三每篇默认都走这条流水线：
1. 读取卷级 spec 与写作卡片
2. 按 skeleton-first 规则先建新骨架
3. 再从旧文抽素材
4. 需要时补源码锚点
5. 跑 `reader-friction-reviewer`
6. 本地 commit
7. 除非明确要求，否则不在单篇阶段单独 push

### 3. 每回合的交付格式
每回合结束后，执行会话应统一回报：
- 本回合完成了哪些文章
- 每篇文件路径
- 每篇一句核心判断
- 每篇提交 hash
- 每篇是否完成 reader review
- 本回合横向风险
- 是否允许进入下一回合

### 4. 回合级停止条件
即使用户要求尽量自动执行，卷三 runbook 仍然应在这些情况暂停：
- 前半骨架篇之间出现明显串线，导致后续样本篇无法稳定开写
- 某篇无法守住卷三边界，持续滑向卷四 / 卷五 / 卷六
- 并行批次之间出现明显职责冲突
- git / push / 文件缺失 / 环境异常等技术阻塞

如果没有这些情况，就继续按 runbook 执行。

### 5. push 策略
默认不要求每篇单独 push。
更稳的方式是：
- 单篇完成后本地 commit
- 每回合结束后统一检查
- 回合通过后再按需要统一 push

这样能明显降低并行期的 git 噪音。

---

## 六、一句话执行结论

> **卷三最稳的做法是 3 个回合：前 4 篇骨架串行、5 篇本地工具分两波并行、最后 2 篇补全与收束串行。按 runbook 执行时，每回合先完成本回合内部文章，再做横向检查，通过后才进入下一回合。**
