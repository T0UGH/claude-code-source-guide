# Claude Code 源码导读手册 v2

`guidebookv2/` 是这套手册新一轮重写的正式工作区。

这里不再先按旧卷目录或源码目录往前推，而是先按**读者的认知台阶**重组全书。目标不是让读者“逛源码”，而是让读者一层一层看懂：Claude Code 到底是什么系统、一次请求怎么跑、执行能力怎么落地、上下文为什么能长期维持、系统又怎样不断长出更多能力。

所以 `guidebookv2/` 的核心重组原则已经固定为：

> **按 6 个认知台阶重组。每一卷都要回答一个比上一卷更深一级的问题。**

这套顺序的目标非常明确：

- 读者不是在逛源码目录
- 读者是在一层一层理解这套系统
- 每一卷都必须有独立主问题
- 卷与卷之间必须形成自然递进，而不是材料堆放

## 全书重组总纲：6 个认知台阶

### 卷一：Claude Code 是什么系统

这一卷解决的是：

- Claude Code 不是“一个聊天壳”，而是一个 agent runtime
- 这套系统里有哪些核心对象
- 用户输入、模型输出、tools、agent、skill、上下文分别在什么位置

这一卷的任务是给读者一张全景图，但不细钻。

一句话说：

> **卷一负责让读者先知道自己在看什么系统。**

---

### 卷二：一次 Agent Turn 是怎么跑起来的

这一卷解决的是：

- 用户输入怎么进入系统
- `processUserInput.ts` 到 `query.ts` 之间发生了什么
- 模型回复、`tool_use`、继续追问，这一轮怎么闭环

这一卷就是主循环卷，解决动态主线。

一句话说：

> **卷二负责让读者看见一次 agent turn 是怎么真正跑起来的。**

---

### 卷三：工具系统怎么把模型意图落成执行

这一卷解决的是：

- Tool 抽象
- tool orchestration
- tool execution
- Bash / File / Search 这些本地能力怎么被 runtime 接住

这一卷就是执行能力卷。

一句话说：

> **卷三负责回答：模型已经决定做事之后，runtime 怎么把这个意图真正落成现实执行。**

---

### 卷四：上下文与状态怎么维持系统持续工作

这一卷解决的是：

- messages
- context construction
- system prompt / additional contexts
- collapse / compaction / projection
- 为什么它不会无限膨胀死掉

这一卷就是持续工作卷。

一句话说：

> **卷四负责解释：Claude Code 为什么不是跑一轮就散，而是能持续工作、长期维持。**

---

### 卷五：外部扩展与多代理能力

这一卷解决的是：

- MCP
- skills
- agents
- subagents
- 外部资源怎么接入
- Claude Code 怎么变成一个可扩展平台

这一卷就是扩展能力卷。

一句话说：

> **卷五负责解释：这套系统怎样长出更多能力，并接入系统之外的能力源。**

---

### 卷六：命令、工作流与产品层整合

这一卷解决的是：

- slash commands
- runtime 命令入口
- skill frontmatter / runtime interface
- command / tool / skill / agent 的边界
- Claude Code 最终为什么呈现为今天这个产品

这一卷是产品整合卷 / 控制层卷。

一句话说：

> **卷六负责把用户入口、控制层、运行时接口和最终产品形态收成一体。**

---

## 这版重组思路和旧卷结构的关键区别

这版设计和旧卷结构相比，最关键的差异不是换了几个标题，而是：

> **每一卷都必须回答一个比上一卷更深一级的问题。**

旧结构里，工具、skills、agents 等材料虽然很多，但“读者为什么按这个顺序学”并没有被钉得足够硬。

而 `guidebookv2/` 的重组目标是：

- 卷一先回答：这是什么系统
- 卷二再回答：它一轮怎么跑
- 卷三继续回答：它怎么真正做事
- 卷四再回答：它怎么长期维持
- 卷五继续回答：它怎么长能力
- 卷六最后回答：它怎么收成今天这个产品

这样读者的感觉就会变成：

- 我不是在逛源码目录
- 我是在一层一层理解这套系统

这才更像一本书，而不是一组源码笔记的重新摆放。

## 从旧结构到新结构，当前已经确认的几个判断

### 1. 现有旧卷一实际上混了三卷内容

从旧资产看，原 `volume-1/` 里长期混放了三类东西：

- 工具系统与执行能力（应归卷三）
- agent / skill / subagent 等扩展能力（应归卷五）
- command / verify / debug / 控制入口（应归卷六）

也就是说，旧卷一的问题不是单纯“太大”，而是：

> **它其实是三卷混装。**

所以 `guidebookv2/` 不是在给旧卷一减肥，而是在把它重新拆回正确的认知台阶。

### 2. 后半三卷的边界必须比前半两卷更硬

旧结构里，后半部分最容易互相串线：

- MCP / hooks / plugins 更像扩展能力，应偏卷五
- permissions 既贴执行层，又贴控制层，是卷三 / 卷六之间最容易摇摆的一组材料
- team runtime 明显更接近 agent / multi-agent 能力，应偏卷五

所以从卷三开始，`guidebookv2/` 不只是在继续写新卷，还在做后半系统的重新分层。

### 3. 真正的重组工作，不是先搬文章，而是先写死卷级问题

旧 Obsidian 设计里已经明确过：

> **真正最该先做的不是改卷名，而是先重写每卷 README / index。**

原因很简单：只有卷级 README / index 先回答清楚：

- 这卷解决什么问题
- 这卷不解决什么问题
- 这些文章为什么在这里

后面的文章迁移、导航调整和边界修剪才不会乱。

## 对后续写作的硬约束

从这一版开始，后续卷级设计、单卷重写、文章迁移，都默认遵守下面这几条：

1. **卷优先于文章。** 先回答这一卷解决什么，再决定文章怎么拆。
2. **主问题优先于材料归档。** 不是先看有哪些旧文章，再决定卷；而是先定卷的主问题，再决定旧材料怎么回收。
3. **认知递进优先于源码目录。** 读者的理解坡度，比源码文件夹结构更重要。
4. **卷与卷之间必须形成导流。** 每一卷都要自然把读者送进下一卷，而不是断开。
5. **避免提前抢后面卷的职责。** 每一卷只解决自己那一级问题，不提前深讲下一层正题。
6. **先写卷级 README / index，再大规模迁文。** 卷的问题没有写死之前，不做大规模文章归位。
7. **后半卷边界要显式声明。** 尤其是卷三 / 卷五 / 卷六，对执行层、扩展层、控制层的区分必须在卷级说明里先写清。
