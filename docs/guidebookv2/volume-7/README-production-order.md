# 卷七执行说明

> 本文件是卷七执行协议。卷七生产必须按本文件执行。

## 一、总则

1. 卷七必须按 spec / writing cards / 本执行说明三件套推进。
2. 卷七当前唯一有效结构为 8 篇。
3. 卷七主问题固定为：

> **用户入口、运行时接口、产品控制层，是怎样收成一个完整系统的？**

4. 卷七重点是控制层如何成立，不是命令索引，也不是产品宣传。

## 二、当前唯一有效结构

1. 为什么 Claude Code 最终一定会长出一层控制层
2. slash commands / prompt commands 为什么不是普通快捷方式
3. 命令入口是怎样接进 runtime 的
4. skill frontmatter / command interface 为什么是运行时接口，不只是元数据
5. 工作流控制层是怎样在 Claude Code 里成立的
6. command、tool、skill、agent 的边界为什么最终要在卷七收口
7. 为什么说 Claude Code 的产品形态，本质上是 runtime 被包装给用户的方式
8. 为什么用户入口、运行时接口和工作流控制层最终会收成今天这个产品

## 三、唯一允许的运行策略

卷七必须采用：

> **前半入口与 interface 串行，中段 control layer 串行，后半产品形态与收束串行。**

也就是：

- 回合 1：01 → 02 → 03 → 04
- 回合 2：05 → 06
- 回合 3：07 → 08

卷七不得采用：
- 8 篇全并行
- 先写产品形态再补入口
- 把 06 提前到 03 前
- 把 08 抢跑成卷尾总结

## 四、分回合执行顺序

### 回合 1：先立控制层与入口层
- 01 → 02 → 03 → 04
- 必须严格串行
- 未通过不得进入回合 2

### 回合 2：立 workflow control layer 与边界收口
- 05 → 06
- 必须严格串行
- 未通过不得进入回合 3

### 回合 3：进入产品形态与总收束
- 07 → 08
- 必须严格串行
- 08 必须最后启动

## 五、主会话职责

主会话必须负责：
1. 锁定卷七主问题不漂移
2. 锁定 8 篇顺序不漂移
3. 每回合结束做职责验收
4. 拒绝把卷七写成命令索引或产品宣传文
5. 在卷尾前确认入口、interface、workflow control layer 都已成立

## 六、接单 subagent 硬规则

1. Claude Code 系列单篇文章任务，第一步必须优先使用 `claude-code-article-orchestrator`。
2. 不得操作 git。
3. 不得私改结构、顺序、回合。
4. 不得把卷七写成全书抒情总结。

## 七、统一验收清单

每篇完成后检查：
1. 主问题是否单一
2. 是否有源码证据链
3. 主图是否承担正文职责
4. 是否守住卷七控制层边界
5. 是否为下一篇留下导流

## 八、卷七完成标准

只有同时满足下面 5 条，卷七才算完成：
1. 8 篇全部完成并通过验收
2. 卷七已明确立成控制层卷
3. 命令入口、runtime interface、workflow control layer 已形成连续主线
4. 产品形态判断已成立
5. 卷尾已完成卷七自身总收束，而不是营销式总结
