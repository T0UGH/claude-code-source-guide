# 卷五 11｜MCP 和 skills / hooks / plugins 分别是什么关系

## 导读

- **上一篇**：[卷五 10｜Claude Code 是怎样通过 MCP 接入外部能力源和资源系统的](./10-how-claude-code-uses-mcp-to-connect-external-capabilities-and-resource-systems.md)
- **下一篇**：待后续回合衔接 Agent 主轴

## 这篇要回答的问题

到了第 11 篇，MCP 组已经先做完了两件事：

- 第 09 篇先打掉“远程工具扩容”这个浅理解
- 第 10 篇把 MCP 立成了外部能力源和资源系统的接入主线

接下来最重要的工作，就不是继续给 MCP 加细节，而是收边界。

因为只要边界不收，卷五后面几组就很容易互相吃线：

- skills 容易被写成“本地版 MCP”
- hooks 容易被写成“围着 MCP 转的回调层”
- plugins 容易被写成“把 MCP 配置收进去的外壳”

这几种写法都不稳。

所以第 11 篇要回答的是：

> **MCP 和 skills / hooks / plugins，到底各处在什么层级，它们分别在解决什么问题？**

## 先给结论

### 结论一：MCP 解决的是外部能力源接入，不是工作方法封装

skills 关心的是：

- 这类事情怎么做
- 工作流如何组织
- 用户经验和角色结构怎样进入系统

MCP 关心的则是：

- 系统外面的能力怎么正式接进来
- 外部资源系统怎样进入当前工作面
- 外部能力节点如何被 runtime 识别和治理

所以两者一个在**方法层**，一个在**能力源接入层**。

### 结论二：MCP 解决的是外部能力接入，不是运行时接缝控制

hooks 关心的不是“接什么能力”，而是：

- runtime 在哪些节点开放观察、注入和干预
- 什么事件发生时允许插手
- 输入、权限决策、工具输出等环节怎样被改写或编排

所以 hooks 站在**运行时接缝层**，而 MCP 站在**外部能力面接入层**。

### 结论三：MCP 也不等于平台扩展封装单元

plugins 解决的重点不是“怎么连某个外部系统”，而是：

- 怎么把多种扩展能力封装成统一单元
- 怎样安装、启停、校验、治理和分发
- 怎么把 skills / hooks / MCP 等不同内容收成一个平台对象

所以 plugins 属于**统一封装层 / 治理层**，不是 MCP 的别名。

## 为什么第 11 篇必须专门收边界

因为 MCP 太容易被写成“中间那层万能胶”。

它一头连着外部系统，一头又会在 Claude Code 里暴露出可调用能力，看起来确实很像一个会到处碰边的对象。

但正因为它容易碰边，所以更需要明确它**只负责什么**，**不负责什么**。

如果这里不切稳，卷五后面会连续出问题：

- 把 skills 写轻
- 把 hooks 写歪
- 把 plugins 写成兜底名词

第 11 篇的任务，就是避免这种串线。

## MCP 和 skills：一个接能力源，一个组织工作方法

### skills 先回答“怎么做”

skills 最重要的价值，是把用户自己的经验、流程和角色结构稳定压进系统。

它解决的是：

- 这类任务通常按什么顺序推进
- 哪些判断应该优先做
- 哪些边界不能越
- 什么时候该 inline，什么时候该切执行位置

所以 skill 更像一种工作方法单元。

### MCP 回答“外面有什么能接进来”

MCP 则不是在组织做法，而是在扩大 Claude Code 可正式使用的外部能力边界。

它解决的是：

- 哪个外部系统可以连接
- 那个系统带来了哪些动作和资源
- 这些外部能力怎样进入当前 runtime 工作面

所以 MCP 更像一种能力源接入单元。

### 两者怎样协作，但为什么不能混

它们当然可以一起工作。

一个 skill 完全可能默认依赖某些 MCP 能力；也可能把“先调用某个外部资源系统，再继续整理结果”写成稳定流程。

但即便它们协作，也不能混成同一种对象。

因为：

- **skill 决定怎么组织工作**
- **MCP 决定外部能力怎样进入系统**

前者是方法层，后者是能力边界层。

## MCP 和 hooks：一个接能力面，一个开放运行时接缝

### hooks 关心的是运行时节点，不是外部系统本身

hooks 最核心的问题不是“这能力来自哪里”，而是：

- runtime 在哪里允许被观察
- 哪里允许注入额外逻辑
- 哪里允许做干预、改写和编排

所以 hooks 面向的是运行过程中的接缝。

### MCP 关心的是外部能力节点如何被纳入工作面

MCP 的焦点则始终在外部系统：

- 它有没有接进来
- 它暴露了什么能力和资源
- 它当前是否可用
- 它如何持续处于系统治理之下

所以两者最短的区分可以写成：

> **MCP 解决“接什么能力进来”，hooks 解决“runtime 在哪里允许插手”。**

### 为什么不能把 hooks 理解成 MCP 的附属层

因为即使完全没有 MCP，hooks 依然成立。

系统一样需要：

- 观察输入
- 插手权限判断
- 改写某些输出
- 在关键节点做运行时干预

这说明 hooks 的地位并不依附于 MCP。

反过来，即使没有 hooks，MCP 也还是外部能力接入层。

两者可能在某些具体场景里配合，但它们解决的是两类不同问题。

## MCP 和 plugins：一个是接入面，一个是封装与治理单元

### plugin 不是“更大一点的 MCP”

很多人看到 plugin 能包含 MCP server，就容易顺手把 plugin 理解成“把 MCP 打包起来的东西”。

这也不稳。

因为 plugin 解决的核心问题不是接入某个外部能力，而是：

- 如何把不同扩展内容收成统一能力包
- 如何定义来源、启停、安装、校验、设置和策略
- 如何把分散扩展对象纳入一套平台治理边界

也就是说，plugin 关注的是更高一层的封装与治理。

### MCP 可以成为 plugin 的内容之一，但不等于 plugin 本身

在平台层设计里，MCP 很适合成为 plugin 能承载的一部分内容。

但“能被收进去”不等于“本体相同”。

更准确的关系是：

- MCP 是一种外部能力接入对象
- plugin 是一种更完整的扩展封装单元

前者像接线面的正式定义，后者像平台治理下的能力包。

### 为什么这一点对卷五后半很重要

因为如果把 plugin 写成“MCP 的上位词”，后面 plugins 组三篇就会失去独立价值。

而卷五现在已经明确：

- MCP 讲能力源接入
- hooks 讲运行时接缝
- plugins 讲更成熟的封装、分发和复用形态

所以第 11 篇必须把这个层级差守住。

## 把四者放回同一张图里看

如果只用最短的话来压缩，四者可以先这样区分：

- **skills**：把用户经验、流程和角色结构组织成工作方法单元
- **MCP**：把系统外能力源和资源系统正式接进 runtime
- **hooks**：在运行时关键节点开放观察、注入和干预接缝
- **plugins**：把多种扩展内容封装成可安装、可治理、可分发的统一单元

这四层并不是互斥孤岛，它们可以互相协作。

但它们协作的前提，恰恰是各自职责先被切稳。

## 这篇不展开什么

### 1. 不重写第 09 / 10 篇

第 11 篇不是再论证一次“为什么 MCP 不是远程工具”，也不是再讲一遍它怎样接入 runtime。

它只负责收边界。

### 2. 不提前展开 hooks 组或 plugins 组的完整正文

这里只做层级切分，不抢后两组自己的锚点篇和展开篇职责。

### 3. 不把所有对象做成术语字典

卷五不是对象手册。

这里要留下的是一张稳固的分工图，而不是百科索引。

## 和前后文的边界

### 它承接第 10 篇

第 10 篇把 MCP 立成外部能力源和资源系统的接入主线。

第 11 篇接着做的，就是防止这条主线继续外溢，把后面几组的职责吃掉。

### 它导向后续各组

一旦第 11 篇把边界切稳，后面的 Agent 主轴、hooks 组、plugins 组就都能在不被 MCP 吞并的前提下继续展开。

## 一句话收口

> **MCP 在卷五里解决的是系统外能力源与资源系统的正式接入；它不负责像 skills 那样封装工作方法，不负责像 hooks 那样提供运行时接缝，也不负责像 plugins 那样承担统一封装、安装和治理，因此它必须被稳稳放在“外部能力接入层”，而不是中间什么都能解释的万能对象。**
