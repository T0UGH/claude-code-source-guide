# 卷五执行说明

> 本文件不是讨论稿，不是建议清单，也不是写作心得。  
> **本文件是卷五执行协议。**  
> 卷五生产必须按本文件执行。

---

## 一、总则

### 1. 卷五必须按本执行说明推进

卷五必须按本文件规定的：

- 分组顺序
- 回合顺序
- 串并行边界
- 主会话职责
- 接单 subagent 职责
- 验收与回退规则

执行。

**不得临时改组、改顺序、改主轴、改执行协议。**

如果结构判断发生变化，必须先修改：

1. 卷五 spec
2. 卷五写作卡片
3. 本执行说明

三者一致后，才允许继续推进卷五。

### 2. 卷五当前唯一有效结构

卷五当前唯一有效结构为：

- 总论 3 篇
- skills 5 篇
- MCP 3 篇
- Agent 主轴 7 篇
- hooks 3 篇
- plugins 3 篇
- 卷尾 1 篇

**合计 25 篇。**

对象主线顺序固定为：

> **skills → MCP → Agent 主轴 → hooks → plugins**

### 3. Agent 主轴不得拆回两个平级组

卷五中：

- `agent`
- `subagent`
- `fork worker`

**统一并入 Agent 主轴。**

主会话、执行会话、接单 subagent：

- **不得再把它们拆回 “agents 组 + subagents 组” 两个平级对象组**
- **不得按两组平行推进**
- **必须把它们视为同一条执行者主线的前半段 / 后半段**

### 4. 本文件优先于临场发挥

卷五执行期间：

- 不允许“写着看”
- 不允许“先跑一篇再说”
- 不允许“这次特例”

**临场发挥不能覆盖本文件。**

## 三、卷五唯一允许的运行策略

### 0. 回合 2-6 的派单方式是强制规则

回合 2 到回合 6：

- **必须各自由一个主执行 subagent 负责整组推进**
- **主会话不得把这五个回合重新拆回“逐篇散派”模式**
- **一个回合只能有一个主执行 subagent 对该回合结果负责**

固定对应关系为：

- 回合 2（skills）→ 1 个主执行 subagent
- 回合 3（MCP）→ 1 个主执行 subagent
- 回合 4（Agent 主轴）→ 1 个主执行 subagent
- 回合 5（hooks）→ 1 个主执行 subagent
- 回合 6（plugins）→ 1 个主执行 subagent

卷五**不得**采用：

- 全串行一篇篇慢跑
- 全并行一口气乱跑
- 组间跳跃推进

卷五**必须**采用：

> **分组串行、组内有限并行。**

解释如下：

1. **组与组之间必须串行推进**
2. **组内只有在锚点篇完成后，展开篇才允许启动**
3. **边界篇必须后置，禁止抢跑**

为避免歧义，卷五每组统一分三类文章：

### 1. 锚点篇
- 先写
- 先验收
- 未通过，不得启动本组后续文章

### 2. 展开篇
- 只能在锚点篇通过后启动
- 可有限并行
- 不得越界吃掉边界篇职责

### 3. 边界 / 收口篇
- 必须后置
- 必须在本组主要展开篇完成后再写
- 用于收边界、防串线、导向下一组

## 五、主会话硬职责

主会话必须负责：

1. 选择当前回合
2. 指定当前允许启动的文章
3. 向接单 subagent 提供：
   - 卷五 spec
   - 卷五写作卡片
   - 本执行说明
   - 当前文章编号 / 标题 / 所属回合 / 所属角色
4. 回收草稿并做验收
5. 决定是否放行下一篇或下一回合

主会话**不得**：

- 跳过验收直接放行下一回合
- 允许边界篇抢跑
- 允许 Agent 主轴被拆回两个平级组
- 允许接单 subagent 绕开规定 skill
- 把回合 2-6 重新拆回逐篇散派
- 把 git 操作下放给接单 subagent

## 六、接单 subagent 硬规则

### 1. 接收单篇文章任务后的第一条硬规则

只要是 **Claude Code 系列单篇文章任务**，接单 subagent：

> **第一步必须优先使用 `claude-code-article-orchestrator`。**

这是硬要求，不是建议。

### 2. 禁止事项

接单 subagent **不得**：

- 绕开 `claude-code-article-orchestrator` 自由改稿
- 在未读取 spec / 写作卡片 / 本执行说明前直接起稿
- 擅自改文章顺序
- 擅自改变当前文章在本组中的角色
- 擅自把边界篇写成总论篇
- 擅自吞并下一组职责

### 3. 违规则处理

出现以下任一情况：

- 未使用 `claude-code-article-orchestrator`
- 未读取卷五 spec / 写作卡片 / 本执行说明
- 越界写作
- 角色错位写作

则该次产出：

> **视为无效稿。**

无效稿必须退回重跑，不得带病推进下一篇。

### 4. Git 权限与接单后的唯一允许动作序列

接单 subagent **不得操作 git**。

这包括但不限于：

- 不得 `git add`
- 不得 `git commit`
- 不得 `git push`
- 不得自行决定提交粒度或提交时机

**git 操作只由主会话负责。**

接单后，subagent 必须按以下顺序执行：

1. 读取卷五 spec
2. 读取卷五写作卡片
3. 读取本执行说明
4. 确认当前文章编号 / 标题 / 所属回合 / 所属角色
5. **使用 `claude-code-article-orchestrator`**
6. 按 orchestrator 内部链路完成：
   - source anchor
   - explanation 优化
   - reader friction 检查
   - project 边界守护
7. 返回产物与简要验收说明
8. 等待主会话决定是否执行 git 操作

这条顺序：

> **不得改序，不得省略。**

## 七、单篇默认流水线

卷五每篇默认必须走这条流水线：

1. 读取卷级 spec
2. 读取卷级写作卡片
3. 读取本执行说明
4. 确认当前文章角色
5. 使用 `claude-code-article-orchestrator`
6. 按 skeleton-first 规则先建新骨架
7. 再从旧文抽素材
8. 需要时补源码锚点
9. 完成后交主会话验收

任何一篇文章：

- **未走完这条流水线，不得视为完成**
- **未经验收，不得自动放行下一篇**

## 八、失败处理与回退规则

### 1. 越界写作
如果某篇文章吃掉了下一组职责：
- 主会话必须退回
- 不得“先留着以后再修”
- 修正后才能继续

### 2. 锚点篇失效
如果锚点篇未立住本组口径：
- 本组后续文章全部暂停
- 必须先修锚点篇
- 锚点篇通过后才能恢复推进

### 3. 边界篇抢跑
如果边界篇在本组主体未稳定前启动：
- 该稿视为无效推进
- 必须回退到本组主体文章
- 边界篇重新后置

### 4. 未使用指定 skill
如果接单 subagent 未使用 `claude-code-article-orchestrator`：
- 该稿直接视为无效稿
- 必须重跑
- 不得做“人工补救后算通过”处理

## 九、卷五当前唯一允许的执行结论

截至目前，卷五执行唯一有效结论是：

- 总论 3 篇：**严格串行**
- 对象组之间：**串行推进**
- 组内：**锚点优先、展开后置、边界最后**
- Agent 相关内容：**统一并入 Agent 主轴**
- 回合 2-6：**必须各自由一个主执行 subagent 负责整组推进**
- 接收单篇文章任务的 subagent：**必须优先使用 `claude-code-article-orchestrator`**
- 接单 subagent：**不得操作 git；git 只由主会话负责**

本文件的目标不是解释结构，而是：

> **让卷五可以被硬性执行地跑完。**
