# 卷五生产顺序说明

> 用途：记录卷五当前确认的生产顺序、分回合策略、串行 / 并行边界，以及主会话与接单 subagent 的执行协议，目标不是讨论结构，而是**让卷五可以一轮一轮稳定跑完**。

---

## 一、卷五当前总前提

卷五已确定为：

> **卷五：外部扩展与多代理能力**

主问题是：

> **复杂场景下，内置能力迟早不够，Claude Code 为什么把扩展权交给用户，并把系统之外的能力源、工作方式和更多执行者接进 runtime。**

当前已经确认的执行约束：

- 采用**先总后分**结构
- 前段总论 3 篇必须先立住卷级口径
- 对象主线顺序固定为：**skills → MCP → Agent 主轴 → hooks → plugins**
- `skills` 与 `Agent 主轴` 是双核心
- `Agent 主轴` 内部包含：agent、subagent、fork worker
- `hooks / plugins` 都不能写成边角补丁
- 卷尾必须压回扩展能力总图
- 旧文只能作为素材仓，不能反向决定新文骨架

卷五当前工作版按 **25 篇** 处理：

- 总论 3
- skills 5
- MCP 3
- Agent 主轴 7
- hooks 3
- plugins 3
- 卷尾 1

## 三、推荐生产策略：分组串行、组内有限并行

卷五默认不采用“全串行”，也不采用“最大化并行”，而采用：

> **分组串行、组内有限并行。**

原因：

- 总论 3 篇决定全卷口径，必须严格串行
- 对象组之间边界重，必须按组推进，不能乱跳
- 组内又存在明显的“锚点篇 / 展开篇 / 边界篇”差异，允许在锚点已稳定后做局部并行
- 如果不这样做，卷五极易一边太慢，一边又容易写散

### 3.1 三类文章角色

为方便稳定派单，卷五每组默认区分三类文章：

1. **锚点篇**
   - 先出
   - 决定本组主问题和核心口径

2. **展开篇**
   - 可在锚点篇稳定后局部并行
   - 负责把本组往下拆开讲透

3. **边界 / 收口篇**
   - 靠后
   - 负责防串线、做分工收口、把本组送向下一组

## 回合 1：总论 3 篇

### 执行方式
**严格串行**。

### 顺序
1 → 2 → 3

### 为什么必须串行
这三篇决定：
- 卷五是不是“扩展层主线卷”
- 卷五的开门问题到底是什么
- 后面对象组是不是有统一总图可依附

### 本回合结束后检查
- 卷五是否已经从“复杂场景”与“扩展权交给用户”开篇，而不是从对象目录开篇
- 第 3 篇是否已经建立足够稳的对象总地图
- 后续各对象组是否已经可以在同一张图下往下写

## 回合 2：skills 组

- **锚点篇**：05
- **前置澄清篇**：04
- **展开篇**：06 / 07
- **边界收口篇**：08

### 推荐运行顺序
1. 先出 **05**
2. 再出 **04**
3. 再并行 **06 / 07**
4. 最后出 **08**

- 08 必须最后收边界，否则 skills 很容易吃掉 agent 主轴

### 本组结束后检查
- skills 是否已经被立成“能力定制重轴”
- 06 / 07 是否没有彼此重叠
- 08 是否把 skill / tool / agent 的边界切稳了

## 五、不推荐的做法

### 1. 不要把 Agent 主轴拆回 agents 组 + subagents 组平级推进
这会直接破坏 v1 旧文已经显露出来的主线：
- AgentTool / runAgent / forkSubagent 本来就是一条连续链
- subagent 更像 agent 主线的后半段，而不是独立对象类

### 2. 不要把各组锚点篇和边界篇一起并行首稿
锚点篇先立口径，边界篇再收职责；两者一起跑，最容易互抢角色。

### 3. 不要把 hooks 组写成“配置项索引”
必须显式解释 hooks 类型差异，否则这一组没有真正成立。

### 4. 不要把 plugins 写成“生态宣传”
这一组必须守住封装层与平台成熟度，不然会虚。

## 六、playbook 化执行规则

这份文件不只是讨论摘要，而是卷五当前的**执行协议**。

### 1. 启动格式
当主会话或执行会话收到类似下面的指令时：
- 跑卷五回合 2
- 继续卷五下一回合
- 按卷五 playbook 执行

默认含义是：
- 读取本文件
- 读取卷五 spec
- 读取卷五写作卡片
- 按本文件定义的顺序、串并行边界和回合检查来执行
- 不临时改对象分组，不重排主轴

### 2. 主会话职责
主会话默认负责：
1. 选择当前回合
2. 指定本回合要跑的文章
3. 给接单 subagent 提供：
   - 卷五 spec
   - 卷五写作卡片
   - 本 playbook
   - 当前目标文章编号 / 标题 / 组内角色（锚点篇 / 展开篇 / 边界篇）
4. 回收草稿并做轻验收
5. 决定是否放行下一篇或下一回合

### 3. 接单 subagent 的硬规则
接收 Claude Code 系列**单篇文章任务**的 subagent，必须遵守：

> **必须优先使用 `claude-code-article-orchestrator`。**

这不是建议，是硬要求。

也就是说：
- 不能自由发挥式改稿
- 不能绕开 orchestrator 直接按个人习惯重写
- 不能把单篇文章任务退化成随手编辑

subagent 接单后默认执行协议为：
1. 先读取卷五 spec、卷五写作卡片、本 playbook
2. 确认当前文章在本组中的角色（锚点 / 展开 / 边界）
3. **优先使用 `claude-code-article-orchestrator`**
4. 按 orchestrator 内部链路完成：
   - source anchor
   - explanation 优化
   - reader friction 检查
   - project 边界守护
5. 返回产物与简要验收说明

### 4. 单篇默认流水线
卷五每篇默认都走这条流水线：
1. 读取卷级 spec 与写作卡片
2. 读取本 playbook，确认当前文章角色
3. 使用 `claude-code-article-orchestrator`
4. 按 skeleton-first 规则先建新骨架
5. 再从旧文抽素材
6. 需要时补源码锚点
7. 完成后由主会话轻验收
8. 除非明确要求，否则不在单篇阶段单独 push

### 5. 每回合的交付格式
每回合结束后，执行会话应统一回报：
- 本回合完成了哪些文章
- 每篇文件路径
- 哪些篇已经稳定可作为后续锚点
- 哪些边界仍需主会话确认
- 是否可以进入下一回合

## 七、当前最稳的运行结论

截至目前，卷五 playbook 的核心结论是：

- 总论 3 篇：**严格串行**
- 对象组之间：**串行推进**
- 组内：**锚点优先、展开后置、边界最后**
- Agent 相关内容：**统一并入 Agent 主轴**，不再拆成两个平级对象组
- 接收单篇文章任务的 subagent：**必须优先使用 `claude-code-article-orchestrator`**

这份 playbook 的目标不是解释结构，而是：

> **让卷五可以被一轮一轮稳定跑完。**
