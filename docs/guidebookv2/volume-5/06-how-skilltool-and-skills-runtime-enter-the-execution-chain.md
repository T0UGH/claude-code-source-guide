# 卷五 06｜SkillTool / skills runtime 是怎样接进执行链的

- **下一篇**：[卷五 07｜什么样的 skill 才算好的 runtime skill](./07-what-makes-a-good-runtime-skill.md)

## 这篇要回答的问题

第 05 篇已经把 skills 的组内锚点立住了：

> skills 最重要的价值，不是让 prompt 可复用，而是让用户自己的工作方法可以稳定进入系统。

但接下来还必须回答一个更硬的问题：

> **这种“工作方法进入系统”不是比喻，那它到底怎样接进执行链？**

如果这里讲不清，skill 就还是容易被读成一种外挂文档：

- 平时放在目录里
- 需要时抄进提示词
- 顶多算一套模板库

而第 06 篇要立住的是相反的判断：

> **skill 不是外挂文档，而是会被 runtime 正式识别、加载和推进的执行组织单元。**

## 先给结论

### 结论一：skill 并不是直接以 markdown 身份进入执行链，而是先被编译成系统可识别的对象

这是理解第 06 篇的第一步。

用户表面上看到的是：

- 一个目录
- 一份 `SKILL.md`
- 顶部一些 frontmatter
- 正文一套工作流说明

但 Claude Code 真正接手它的时候，并不是把这份 markdown 原样当作文档挂在那里，而是会先把它变成 runtime 能认识的结构。

这一步的意义在于：

- skill 不再只是文本材料
- skill 变成了可发现、可加载、可调用的对象

### 结论二：SkillTool 的作用不是“读一篇文档给模型看”，而是把 skill 接进统一执行链

SkillTool 最重要的地方，不是它会展开一段文字，而是它把 skill 带进了和其他执行对象同一条正式路径：

- 被 assistant 选中
- 被 runtime 接住
- 被分发到对应执行模式
- 再把结果回流到当前 turn

这说明 skill 在 Claude Code 里已经不只是内容模块，而是执行对象谱系的一员。

### 结论三：skills runtime 真正接进来的，是“方法组织层”而不是“动作原语层”

Bash、Read、Edit 这类对象，更像直接动作原语。

skill 则不同。

它通常不会亲自完成底层动作，而是：

- 组织当前任务该怎么做
- 决定用哪类执行节奏推进
- 在需要时走 inline 或 fork 路径
- 把动作原语编进一个较高阶的工作流

所以 skill 进入执行链之后，补上的不是又一类底层工具，而是：

> **方法组织层。**

## skill 为什么不是“目录里有个文件”就算接进系统

从系统角度看，一个文件存在于磁盘上，和一个对象进入 runtime，是两回事。

前者只说明材料存在。

后者则要求至少成立几件事：

- 系统知道它在哪
- 系统知道它叫什么
- 系统知道它适合什么时候被发现
- 系统知道它带不带权限和执行边界
- 系统知道它被调用后该走哪条路径

skill 这条线真正做掉的，正是这几步。

所以说它接进执行链，意思不是“模型最后看到了它”，而是：

> **runtime 已经替它建立了从发现、装配到执行的正式入口。**

## skill 接进执行链，至少要过三层

### 第一层：定义层——把用户写的 skill 读成系统对象

这一层解决的是：

- skill 从哪里被发现
- markdown 怎样被解析
- frontmatter 和正文怎样被分开
- 哪些字段会进入运行时语义

这一层最关键的变化是：skill 不再是“给人阅读的一页说明”，而是开始变成“给系统读取的一份定义”。

也正因为有这一层，后面才会有：

- 条件激活
- 权限边界
- 上下文模式
- 模型和 effort 覆盖

这些 runtime 语义。

### 第二层：入口层——把 skill 作为可调用能力挂进当前能力面

skill 只有被定义出来还不够。

它还必须能被当前 turn 的能力面看见。

也就是说，到了真正执行的时候，Claude Code 要能够把它和其他可调用对象一起交给 assistant：

- 当前有哪些工具
- 当前有哪些 commands / skills
- 当前有哪些 agents

这一层的意义在于：skill 不再只是“配置已存在”，而是“当前回合可被选中”。

### 第三层：执行层——被 SkillTool 正式接住并继续推进

真正进入主链的是这一层。

assistant 一旦发出对 skill 的调用，SkillTool 就不再只是展示说明，而是正式开始做几件事：

- 匹配对应 skill
- 组装执行上下文
- 判断走 inline 还是 fork
- 把必要的权限、模型、上下文约束带进执行
- 最后把结果重新回流给当前线程

到这里，skill 才算真的接进执行链。

## SkillTool 为什么是关键桥位

卷三已经解释过，SkillTool / AgentTool 都属于执行对象谱系里的高阶成员。

放到卷五里，SkillTool 的意义则更具体：

> **它是把方法组织层接入执行链的桥。**

### 1. 它让 skill 不是停在定义层

如果没有 SkillTool，skill 最多只是被发现、被列出、被看见。

有了 SkillTool 之后，skill 才真正从“定义好的方法模块”变成“当前回合可以正式调用的运行单元”。

### 2. 它让 skill 进入统一的 runtime 语言

SkillTool 最值钱的一点，是它没有让 skill 变成独立漂浮的一套旁路机制。

相反，它把 skill 接回了统一执行链：

- 由 assistant 发起
- 由 runtime 接住
- 由执行层推进
- 再回到当前 turn

这意味着 skill 和其他执行对象虽然职责不同，但语言是统一的。

### 3. 它让 skill 既能 inline，也能 fork

这里正好说明 skill 为什么不只是长 prompt。

如果它只是提示词长度增强，那最自然的做法就是：

- 直接展开到当前上下文
- 说完就结束

但 Claude Code 给它保留了两条正式路径：

- **inline**：把方法组织直接塞回当前线程继续推进
- **fork**：把这一段工作交给独立 worker 路径执行，再把结果回流

这说明 skill 在系统里被理解成“组织执行的一层”，而不是“文本补丁”。

## inline 和 fork，分别说明了什么

### inline 说明：skill 可以作为当前线程的方法层直接参与执行

这条路径最适合：

- 当前问题仍然需要紧贴主线程推进
- 只是要补一套做事方法
- 不值得单独切出子执行者

这时候 skill 更像把一套工作流压进当前上下文。

重点不是“多贴一段 prompt”，而是：

- 当前线程接下来该怎么做
- 权限和约束怎样随之生效
- 这个方法怎样进入本轮执行节奏

### fork 说明：有些 skill 已经足够像独立工作包

而另一条路径说明得更彻底。

有些 skill 本身就带着很明显的独立处理特征：

- 输入相对完整
- 输出相对清晰
- 过程较长
- 适合交给 worker 去跑

这时候 skill 不再只是“给当前线程一点方法建议”，而更像：

> **把一个方法化工作包交给独立执行分支。**

这也正是第 05 篇为什么会提到“角色结构”的原因。

## 从卷五结构看，第 06 篇到底立什么

第 06 篇不需要把源码每个函数都展开，但必须把这条主线讲清：

1. 用户写的是 skill 文件
2. 系统读到的不是普通文档，而是定义对象
3. 当前 turn 看到的不是一页说明，而是一项可调用能力
4. 执行链接住它的，不是复制粘贴逻辑，而是 SkillTool
5. SkillTool 决定它如何进入当前执行：inline 或 fork

只要这五层关系立住，读者就能真正明白：

> **skill 进入 Claude Code，不是“被读到”，而是“被装进执行链”。**

## 和前后文的边界

### 它承接第 05 篇

第 05 篇已经说明 skills 接进来的是用户工作方法。

第 06 篇继续回答：这种方法不是抽象说法，它会通过定义层、入口层和 SkillTool 桥位，正式进入 Claude Code 的执行链。

### 它导向第 07 / 08 篇

一旦执行链位置立住，后面两个问题才会自然出现：

- 什么样的 skill 才真能稳定承担这条 runtime 位置
- 它和 tool / agent 的边界究竟在哪里

## 一句话收口

> **SkillTool / skills runtime 真正做成的，不是让一份 SKILL.md 被模型“看到”，而是把它从用户写下的工作方法，逐层装配成当前回合可发现、可调用、可分流、可回流的执行组织单元；它接进执行链的不是文档，而是方法层。**
