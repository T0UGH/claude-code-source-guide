# 卷五 17｜主 agent / worker agent 的职责边界与信息回流

## 这篇要回答的问题

Agent 主轴后半段到了这里，真正要收的不是“还可以再派多少 worker”，而是：

> **主 agent 和 worker agent 的职责边界到底怎么划，worker 做完后的结果又怎样重新回流主线？**

因为一个协作结构是否成熟，不取决于“能不能分叉”，而取决于：

- 分出去之后会不会失控
- 主线和支线是否各守自己的边界
- 结果是否能重新进入主线判断

所以第 17 篇要立住的核心判断是：

> **worker 之所以还是协作结构而不是失控分叉，关键就在职责边界和信息回流被系统收得住。**

## 先给结论

### 结论一：主 agent 负责主线判断与整合，worker agent 负责受托子任务推进

主 agent 的核心职责不是把所有局部都亲自做完，而是：

- 保持主任务目标
- 判断下一步推进方向
- 决定哪些子问题要拆出去
- 在结果回来后继续整合与推进

worker agent 则更像：

- 接手一段边界清楚的子任务
- 在受控范围内独立推进
- 把结果而不是整条主线带回

这两者一旦不分清，协作结构很快就会塌回单线程混乱。

### 结论二：worker 的价值不在“替主线思考一切”，而在“替主线处理一段可回收的局部工作”

如果 worker 也开始承担主线判断、全局目标漂移和总体整合，那么它就不再是 worker，而像另一条失控主线。

Claude Code 更稳的做法，是让 worker 的职责保持在：

- 局部处理
- 明确产物
- 结果回交

这正是“受控协作”成立的前提。

### 结论三：信息回流不是附加步骤，而是整个 worker 结构成立的收口机制

如果 worker 做完后只是各自散开，主线就无法真正吸收分叉收益。

所以回流不是锦上添花，而是 worker 结构本身的一部分：

> **分出去的工作，最终必须重新进入主 agent 的可判断主线。**

## 主 agent 和 worker agent 的职责边界怎么切

### 1. 主 agent：保主线

主 agent 最重要的任务，是守住主线。

这意味着它负责：

- 当前总体目标是否仍成立
- 哪些任务值得继续推进
- 哪些工作该拆出去
- 回收结果后如何继续做下一步判断

所以主 agent 的价值不只是“最强的那个执行者”，而是主线责任承担者。

### 2. worker agent：保子任务

worker agent 更适合承担的是：

- 一段边界明确的局部工作
- 一次相对独立的分析、修改或搜索
- 一份可被主线吸收的中间结果

它不是整个系统的总协调者，也不是另一条平级主线。

所以 worker 最重要的不是自主性有多大，而是边界有多清楚。

### 3. 两者关系：不是谁更高级，而是谁负责哪一层工作

如果把主 agent 和 worker agent 写成“上下级能力比较”，很容易跑偏。

更稳的写法是：

- 主 agent：负责主线推进与整合
- worker agent：负责局部工作展开与结果回交

它们的区别首先是职责分工，不是功能强弱。

## 为什么职责边界收不住，协作结构就会失控

### 第一，因为主线判断一旦外溢，主 agent 就会失去主线

如果 worker 也开始替主线改目标、改推进方向、自己决定总体结论，那主线实际上已经被拆散了。

这会导致：

- 主 agent 失去整合位置
- 多条执行线开始彼此争主语
- 协作结构变成并列漂移

所以主线判断必须收在主 agent 手里。

### 第二，因为子任务边界一旦模糊，worker 就会变成自由扩张执行者

worker 的价值，建立在“这是一段被切出来的局部工作”上。

如果边界不清，它就很容易：

- 越做越宽
- 把别的子任务也顺手吞进去
- 最后带回一大坨未经整理的过程

这样主线反而更难整合。

所以 worker 不是越自由越好，而是越能守住边界越有价值。

## 信息回流为什么是后半段真正的收口点

### 1. 因为分叉的目的不是分裂，而是为了让主线更稳地继续推进

subagent / worker 的目标，不是让系统长出若干独立小世界。

它们存在的意义，是为了让主线：

- 不必被局部细节拖住
- 又能吸收这些局部工作的成果

这意味着分叉一定要以回流为终点。

### 2. 因为主 agent 只有在结果回流后，才能继续承担整合责任

主 agent 不可能对没回来的结果负责。

它必须拿到：

- 一个相对清楚的子任务结果
- 可吸收的中间结论
- 足以继续判断下一步的工作产物

只有这样，主线才能真正恢复为一条连续推进的线。

### 3. 因为“回流”本身就是把 worker 放回可控协作结构的动作

worker 如果只负责做，不负责回交，那它更像外包出去的独立执行体。

但 Claude Code 这里要成立的，不是外包关系，而是受控协作结构。

所以“结果回流”本身就是控制机制的一部分。

第 17 篇进一步完成的，就是证明：

如果第 17 篇写稳，读者脑子里会留下很清楚的一张图：

- 前半段：更多执行者怎样成立

这正是执行者主线前半段 / 后半段的完整闭环。
