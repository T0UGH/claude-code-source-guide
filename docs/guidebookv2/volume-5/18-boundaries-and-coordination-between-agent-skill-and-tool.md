# 卷五 18｜agent、skill、tool 之间的边界和协作关系

## 导读

- **所属卷**：卷五：外部扩展与多代理能力
- **卷内位置**：18 / 25
- **上一篇**：[卷五 17｜主 agent / worker agent 的职责边界与信息回流](./17-boundaries-and-information-flow-between-main-agent-and-worker-agent.md)
- **下一篇**：待后续回合衔接 hooks 组

## 这篇要回答的问题

到了第 18 篇，Agent 主轴已经走完了：

- agent 不是工具
- Claude Code 怎样长出更多执行者
- runAgent 怎样把执行者装成可运行体
- 为什么主 agent 还要派生 subagent
- forkSubagent 为什么是 worker 分叉
- 主 agent / worker agent 的边界与信息回流怎样收稳

接下来必须做的，不是再补一个术语总结，而是把整条主线重新压回卷五对象地图：

> **agent、skill、tool 各自到底处在哪个层级，它们怎样协作，又为什么不能互相替代？**

这篇如果写空，很容易变成“再说一遍概念区别”。

所以第 18 篇必须真正完成边界收口：

> **tool 是执行能力，skill 是能力组织，agent 是执行者结构；三者相关，但绝不是同一类对象。**

## 先给结论

### 结论一：tool 解决“做什么动作”，skill 解决“怎么组织这些动作”，agent 解决“谁来承担这段工作”

这是第 18 篇最该留下来的骨架。

- **tool**：动作能力
- **skill**：方法组织
- **agent**：执行者结构

卷五写到这里，读者不该只记住三句术语，而应该能用这张图重新理解整条主线。

### 结论二：skill 可以组织 tool，agent 可以使用 skill 和 tool，但层级不会因此合并

很多混乱都来自“它们会协作，所以应该差不多”。

但协作不等于同类。

更稳的理解是：

- skill 可以规定工具怎么搭配、何时切换执行位置
- agent 可以在承担工作时使用 skill 和 tool
- tool 则负责把具体动作落掉

三者刚好是分层协作，而不是互为别名。

### 结论三：Agent 主轴真正收住的，不只是 agent 自己，而是执行者结构在整张对象图里的位置

第 18 篇不能只讲 agent 本身，还要把它放回卷五全图里：

- 它不像 tool 那样站在动作层
- 它不像 skill 那样站在方法层
- 它站在“执行责任由谁承担、怎样分叉协作”的结构层

这才是整个 Agent 主轴真正完成的收口。

## 为什么第 18 篇不能写成空泛术语文

因为卷五前面已经有过一次类似收边界动作：第 08 篇处理的是 skill / tool / agent。

如果第 18 篇只是把那篇换个顺序重说一遍，就会失去价值。

第 18 篇的独立意义在于：

- 第 08 篇站在 skills 组末尾收边界
- 第 18 篇站在 Agent 主轴末尾，带着 13-17 的新信息再收一次

也就是说，这里必须把已经成立的执行者主线重新压回那张图，而不是做抽象术语复读。

## 从 Agent 主轴这边回看，tool 到底是什么

### tool：动作原语，负责把事情真正做下去

tool 仍然是 Claude Code 最直接的执行能力层。

它负责的是：

- 读取
- 修改
- 搜索
- 执行
- 查询

也就是把某个具体动作正式落地。

它能很强，但它的核心仍然是动作调用，而不是方法组织或执行责任承担。

### 为什么 Agent 主轴里不能把 tool 写轻

因为 agent 再重要，也不是凭空做事。

执行者结构之所以成立，仍然要靠 tools 把具体动作落地。

所以第 18 篇不能把 tool 写成小零件，而要明确：

> **tool 是一切落地动作的执行层。**

只是它不负责决定整段工作该怎么组织，也不负责决定谁来承担这段工作。

## 从 Agent 主轴这边回看，skill 到底是什么

### skill：方法模块，负责把能力组织成稳定工作流

第 05-08 篇已经说明过，skill 最重要的价值是把用户经验、流程和角色结构组织进系统。

到了第 18 篇，从 Agent 主轴这边再看，skill 的位置会更清楚：

- 它不是执行者本体
- 它不是动作原语
- 它更像执行者在工作时可采用的方法模块

也就是说，skill 既可以被主 agent 使用，也可能指导 worker 怎么推进某类任务；但它自己并不承担执行者身份。

### 为什么 skill 容易和 agent 混，但仍然必须切开

因为 skill 也会碰到：

- 何时该 fork
- 是否适合交给 worker
- 哪些角色该切换

这会让它看上去很像在“管执行者”。

但更准确的说法仍然是：

- **skill 在组织方法**
- **agent 在承担执行**

skill 可以建议切换执行位置，却不等于那个执行位置本身。

## 从 Agent 主轴这边回看，agent 到底是什么

### agent：执行者结构，负责承接和分配执行责任

第 13-17 篇已经把 agent 主线讲完整了。

所以第 18 篇这里最重要的，不是再说 agent 很强，而是把它压成一句清楚判断：

> **agent 解决的是执行责任由谁承担，以及执行者之间怎样继续分叉协作。**

这也是为什么它天然会连到：

- runAgent 装配
- subagent 派生
- worker 分叉
- 信息回流

这些都不是 tool 或 skill 的中心问题。

### agent 怎样与 skill / tool 协作

把三者压成一条最短协作线，大致可以这样看：

1. **tool** 提供具体动作能力
2. **skill** 组织这些能力的使用方式
3. **agent** 承担这段工作的执行责任，并决定是否继续分叉协作

这样看，整条 Agent 主轴就能和前面的 skills 组自然咬合起来。

## 三者为什么相关，但绝不能互相替代

### 1. tool 不能替代 agent

即使一组 tool 很强，它们也主要回答“能做哪些动作”。

它们不会天然回答：

- 谁来承担整段工作
- 何时需要分工
- 结果由谁整合

这些是 agent 的问题。

### 2. skill 不能替代 agent

即使一个 skill 写得很完整，它也主要回答“这类任务怎么做更稳”。

它不会天然成为执行者本体。

它可以组织执行、建议分叉、规定边界，但真正承担执行责任的仍然是 agent。

### 3. agent 也不能替代 skill 和 tool

agent 再重要，也不能脱离 skill 和 tool 单独成立为完整工作系统。

没有 tool，它就缺少动作落地点。

没有 skill，它就缺少稳定的方法组织层。

所以 agent 不是凌驾于二者之上的“万能对象”，而是和它们分层协作。

## 从卷五总图看，第 18 篇到底完成了什么

### 完成一：把 Agent 主轴重新压回对象分工图

到这里，卷五后半的对象地图已经越来越清楚：

- skill：方法层
- MCP：外部能力接入层
- agent：执行者结构层
- hooks：运行时接缝层
- plugins：统一封装层

第 18 篇的作用，就是把 Agent 主轴从局部机制重新压回这张图里。

### 完成二：防止 Agent 主轴继续外溢到 hooks / plugins / 卷六边界

如果第 18 篇不收边界，Agent 主轴很容易继续膨胀成：

- 什么都能组织
- 什么都能调度
- 什么都能解释

这会直接吃掉后面的 hooks、plugins，甚至卷六的 command / interface 问题。

所以第 18 篇也承担一个重要任务：

> **把 Agent 主轴稳稳收在“执行者结构”这一层。**

### 2. 不重讲卷三工具主链

### 3. 不提前抢卷六的 command / tool / skill / agent 总边界

卷五这里收的是对象分工，不是产品控制层接口。

### 它承接第 17 篇

第 17 篇把主 agent / worker agent 的职责边界与信息回流收稳。

第 18 篇接着做的，就是把整条执行者主线重新压回卷五对象图，和 skill / tool 切开。

## 一句话收口

> **tool 是执行能力，skill 是能力组织，agent 是执行者结构：tool 把动作落地，skill 把动作编成稳定工作流，agent 则承担整段工作的执行责任并决定是否继续分叉协作；三者可以紧密配合，但正因为各自职责不同，才不会在卷五后半重新糊成一个“大扩展桶”。**
