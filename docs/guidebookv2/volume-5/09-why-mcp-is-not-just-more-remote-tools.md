# 卷五 09｜为什么 MCP 不是“多了一批远程工具”

## 导读

- **卷内位置**：09 / 25
- **上一篇**：[卷五 08｜skill、tool、agent 三者的边界到底是什么](./08-boundaries-between-skill-tool-and-agent.md)
- **下一篇**：[卷五 10｜Claude Code 是怎样通过 MCP 接入外部能力源和资源系统的](./10-how-claude-code-uses-mcp-to-connect-external-capabilities-and-resource-systems.md)

## 这篇要回答的问题

一进入 MCP 组，最常见的第一反应通常是：

> **MCP 不就是让 Claude Code 多接几把远程工具吗？**

这个理解之所以常见，是因为从表面看，它确实会带来一些很像 tool 的东西：

- 可以调用的新能力
- 来自外部系统的动作入口
- 不在本地内建工具池里的额外选项

所以把 MCP 理解成“远程工具扩容”，并不是完全没道理。

但如果停在这里，后面第 10 篇和第 11 篇就都会失焦。

因为一旦把 MCP 写成“多了一批远程工具”，你很容易继续滑向几层错误理解：

- MCP 只是 tool 的远端版本
- MCP 的价值主要是数量扩容
- MCP 跟 resources、认证、连接状态关系不大
- MCP 和 plugins / hooks 只是不同接法而已

所以第 09 篇的任务不是把 MCP 讲完，而是先拆掉这个过浅入口。

## 先给结论

### 结论一：MCP 看上去会带来工具，但它真正增加的是外部能力边界

读者第一眼看到的，往往是“多了几个能调用的名字”。

但这些名字后面，真正被接进来的不是一个个孤立动作，而是一整块系统外的能力边界。

也就是说，MCP 的价值不是单纯多几把扳手，而是让 Claude Code 能把原本在系统外面的能力面，稳定接进自己的工作范围。

### 结论二：MCP 不只是动作扩容，它还涉及资源、连接与身份状态

如果 MCP 只是远程工具，那么它最关心的就该只是“能不能调用”。

但真实情况不是这样。

MCP 一旦进入 Claude Code 运行时，系统还必须面对：

- 这个能力节点当前有没有连上
- 有没有完成认证
- 它提供的是不是还包括资源读取面
- 这些能力能否被持续地发现、管理和治理

这些问题都说明：

> **MCP 接进来的不是几次远程调用，而是一类正式的外部能力节点。**

### 结论三：把 MCP 只看成远程工具，会把卷五的对象边界全部写轻

因为后面你会越来越分不清：

- skill 为什么不是 MCP
- hooks 为什么不是“对 MCP 再加一些回调”
- plugin 为什么不是“把几个 MCP 打个包”

所以第 09 篇必须先守住入口：

> **MCP 的意义，不只是远程工具扩容，而是让 Claude Code 拥有稳定接入系统外能力源的通道。**

## 为什么“远程工具扩容”这个理解不够

### 第一，因为“多几把工具”强调的是数量，MCP 处理的是边界

“多了一批远程工具”这个说法，最容易把注意力放在数量变化上：

- 工具池更大了
- 可调用项更多了
- 能做的事变多了

但 MCP 更关键的变化，不是数量，而是边界。

Claude Code 原本主要工作的，是自己已经内建、已经接入、已经可控的能力面。

MCP 的出现则意味着：

- 系统之外还存在别的能力源
- 这些能力源可以不是临时脚本，而是正式接入节点
- runtime 要开始处理“外部能力怎样进入系统”这件事

所以 MCP 真正改变的，不是工具清单长度，而是系统边界的可扩展性。

### 第二，因为“远程工具”会让人忽略资源系统

很多外部系统接进来后，并不只是提供动作，还提供资源面。

也就是说，Claude Code 面对的不一定只是：

- 去执行一个外部动作

还可能是：

- 去读取一个外部对象
- 去浏览一片外部材料
- 去把某个远端资源纳入当前工作上下文

一旦把这层看见，MCP 的性质就明显不只是 tool catalogue。

它更像在给 Claude Code 打开一扇正式的外部资源入口。

### 第三，因为“远程工具”会让人忽略连接和认证状态

本地工具通常天然在系统控制之内。

MCP 面对的则是一个更麻烦的世界：

- 服务可能断开
- 身份可能失效
- scope 可能不足
- 节点可能存在但暂时不可用

这些都不是“工具多了一些”能概括的。

它们说明 Claude Code 处理 MCP 时，实际上是在处理：

> **外部能力节点的可进入性。**

这已经明显是运行时层面的正式问题，而不是工具列表的小修小补。

## 为什么把 MCP 写成“能力源接入层”更稳

### 1. 因为它接的是外部系统，不是单个动作

一个 tool 更像一个动作原语。

但 MCP 常常对应的是某个外部系统整体的接入关系：

- 有连接
- 有身份
- 有资源面
- 有动作面
- 有可持续管理的能力边界

所以更稳的理解方式不是“工具新增”，而是：

> **外部系统能力以一种正式接口进入了 Claude Code runtime。**

### 2. 因为它接进来之后还要被治理

如果它只是“外面有个 API，拿来调一下”，那还谈不上卷五里的对象地位。

MCP 真正值得单列，是因为这些外部能力接进来之后，还要继续接受系统治理：

- 发现与暴露
- 认证与连接维护
- 权限约束
- 错误处理与状态回退

所以它不是一次性接线，而是一种持续接入关系。

### 3. 因为它让 Claude Code 开始组织“系统外面还接着什么”

skills 让系统更像用户自己的工作方式。
agent 主轴让系统长出更多执行者。

MCP 则让系统开始认真处理另一件事：

> **系统外面还连着哪些能力源和资源系统，它们怎样正式进入当前工作。**

这就是为什么它在卷五里必须占一组，而不能只当 tool 附录。
