## 这篇要回答的问题

卷五到了 skills 组，真正要先立住的，不是“skill 有哪些字段”，也不是“怎么写一个 SKILL.md”。

skills 组的锚点问题其实更靠前：

> **Claude Code 为什么要有 skill 这层对象？它到底把什么东西接进了系统？**

如果把 skill 理解成“更长一点的 prompt”，那读者最后留下来的理解会很浅：

- 好像只是把常用提示词存起来
- 好像只是把一些方法写成模板
- 好像只是为了省一点重复输入

这些都不完全错，但都不够。

因为 Claude Code 里 skill 真正接进来的，不只是提示词文本，而是用户自己的工作方式：

- 用户长期积累下来的判断经验
- 一件事通常按什么步骤推进
- 哪些时候该停下来核对
- 哪些时候该切到另一种角色或执行模式

所以第 05 篇真正要立住的一句话是：

> **skills 最重要的价值，不是让 prompt 可复用，而是让用户自己的工作方法可以稳定进入系统。**

## 先给结论

### 结论一：skills 接进系统的不是“内容”，而是“做事的方法”

很多系统也支持模板、片段、预设指令。

但 Claude Code 的 skill 更进一步的地方在于：它试图承接的不是一句现成答案，而是一段可重复执行的方法结构。

这意味着 skill 真正保存下来的东西，通常不是：

- 某一句写得很漂亮的话
- 某一段润色过的提示词

而是：

- 这类问题该怎么判断
- 先做什么，再做什么
- 产物应该长什么样
- 缺信息时如何处理
- 哪一步该继续 inline，哪一步该交给独立 worker

换句话说，skill 不是把“说法”存下来，而是把“做法”存下来。

### 结论二：skills 把用户经验从临场发挥，变成了 runtime 可反复调用的能力单元

没有 skills 的时候，很多高质量协作其实也能发生。

但那种高质量往往很依赖：

- 这次会话刚好上下文很全
- 用户刚好又提醒了一遍偏好
- 模型刚好记得这套推进方式

这类能力并不稳定。

skills 的价值，就在于把这些原本散落在会话里的经验，压缩成一个之后还能被系统再次调起的单元。

也就是说，它把：

- 经验
- 流程
- 角色分工
- 权限边界
- 触发条件

从一次性的配合，变成可持续复用的 runtime 资产。

### 结论三：skills 是 Claude Code 的“能力定制重轴”

卷五必须把 skills 写重，不只是因为它常用，而是因为它决定了一件更大的事：

> **Claude Code 到底能不能学会“按这个用户的方式做事”。**

MCP 解决的是外部能力源接入，agent 主轴解决的是更多执行者怎样长出来；而 skills 解决的，是用户自己的工作方法怎样真正进系统。

所以 skills 在卷五里不是边角机制，而是第一条重轴。

## 为什么说 skill 接进来的首先是“用户经验”

### 第一，因为很多高价值协作本来就不是工具缺失，而是方法缺失

真实工作里，经常不是没有工具，而是不知道怎么组织它们。

比如同样是写一篇文章，难点往往不在：

- 能不能读文件
- 能不能搜索资料
- 能不能改文稿

而在：

- 先立问题还是先找材料
- 旧文该当素材还是当提纲
- 哪些话题这篇不能抢
- 产物到底是综述、锚点篇，还是边界收口篇

这类东西并不是某个单独工具能替你回答的。

它们更像用户在长期工作里形成的经验判断。

skills 恰好承接的，就是这层经验。

### 第二，因为 skill 最适合承载的是“反复出现的判断模式”

用户工作里真正值得沉淀的，通常不是一次性的细节，而是会重复出现的判断模式。

例如：

- 遇到 bug 先做系统化排查，再动手修
- 遇到大改动先做 brainstorming，再进入实现
- 处理技术文章先做 skeleton-first，再抽旧文素材
- 收到 code review 不先顺手改，而先验证评论是否成立

这类模式的共同点是：

- 不等于一个答案
- 不等于一个工具
- 不等于一个角色
- 但会反复决定整段工作怎么走

skill 正好就是把这类模式压成一个可调用对象的地方。

## 为什么它接进来的又不只是“经验”，还包括“流程”

如果只有经验判断，没有流程结构，skill 还是容易飘。

Claude Code 里的好 skill 之所以稳定，是因为它通常还会把动作顺序一并写死。

也就是说，它不只告诉模型“应该重视什么”，还告诉模型：

1. 先做什么
2. 再做什么
3. 遇到例外怎么处理
4. 输出需要包含什么
5. 什么情况下才算完成

这就是为什么很多好 skill 读起来不像散文，反而更像 SOP。

不是因为写作者不讲文气，而是因为 runtime 真正吃得动的，本来就是流程结构。

所以从系统角度看，skill 真正承载的不是“理念”，而是：

> **被写清楚的工作流。**

## skill 为什么还能把“角色结构”接进来

这是很多人第一次看 skill 时最容易低估的一点。

### 1. 有些 skill 只是当前线程里的方法提示

最轻的一类 skill，只是给当前 agent 一套稳定做法：

- 如何先读再写
- 如何组织输出
- 如何压缩结论
- 如何验证结果

这类 skill 主要接进来的是方法。

### 2. 但有些 skill 已经隐含了角色分工

还有一些 skill，本身就不只是“给当前线程一段提示”，而是在声明：

- 这件事适合独立处理
- 这一步应该交给 worker 去跑
- 当前主线程只负责收结果和继续判断

一旦 skill 走到 fork 或绑定特定 agent，它承接的就不只是工作方法，而是角色结构。

也就是说，skill 不只是告诉系统“怎么做”，还可能告诉系统：

> **该由谁、以什么执行位置来做。**

这就是为什么第 05 篇要先把“角色结构”这个词提出来。

因为它正好给后面的 agent 主轴留了坡度：

- skill 可以组织方法
- skill 也可能触发角色切换
- 但它本身仍然不是执行者本体

## 从卷五角度看，skills 真正改变了什么

### 改变一：把个体用户的方法论接进了平台

如果没有 skill，Claude Code 再强，也更多只能依赖：

- 官方内建能力
- 通用 prompt 经验
- 当前会话里的临时协调

有了 skill 之后，系统就多了一层很关键的可塑性：

- 这个用户怎样写作
- 这个团队怎样审稿
- 这个项目怎样调试
- 这类任务该怎么验收

都可以逐步进入系统。

这意味着 Claude Code 的扩展，不只是“能力更多”，而是“更像这个用户自己”。

### 改变二：把工作方法从会话记忆，抬升成运行时对象

会话里的经验会散，聊天里的方法会忘。

而 skill 的价值在于，它把这些东西从：

- 临时提醒
- 反复口头纠偏
- 偶然成功的协作节奏

抬升成一个以后还能被发现、加载、执行、追踪的对象。

这就是为什么卷五会把 skills 写成能力定制重轴。

因为它立住的不是“能不能多写几个 prompt”，而是：

> **用户自己的工作方式，是否能被系统长期吸收。**
