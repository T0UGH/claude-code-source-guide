# 卷五 03｜skills / MCP / agents / subagents / hooks / plugins 是怎样接入 Claude Code 的

## 导读

- **所属卷**：卷五：外部扩展与多代理能力
- **卷内位置**：03 / 25
- **上一篇**：[卷五 02｜为什么 Claude Code 选择把扩展权交给用户](./02-why-claude-code-chooses-to-hand-extension-power-to-users.md)
- **下一篇**：[卷五 04｜为什么 skills 不是“长 prompt”那么简单](./04-why-skills-are-more-than-long-prompts.md)

## 这篇要回答的问题

第 01 篇和第 02 篇已经把卷五的总前提立住了：

- 复杂场景会逼 Claude Code 长出扩展层
- Claude Code 选择把扩展权交给用户

接下来就不能只停在“可以扩展”这种抽象判断上了。

读者真正会继续追问的是：

> **那这些扩展权到底是通过什么东西接进 Claude Code 的？**

更具体一点说，就是：

- skills 是怎么进来的
- MCP 是怎么进来的
- agents / subagents 是怎么进来的
- hooks 是怎么进来的
- plugins 又处在什么位置

第 03 篇的任务，就是先给出一张统一总地图。

这篇不是要把所有对象正文讲完，而是先回答两个问题：

1. **这些对象都在接入 runtime，但接入方式并不相同**
2. **正因为接入方式不同，它们才不能被混成一个“扩展大桶”**

## 先给结论

### 结论一：这些对象不是并排功能点，而是一组以不同方式接入 runtime 的平台对象

如果只从产品表面看，skills、MCP、agents、hooks、plugins 很容易像是几个功能栏目。

但卷五更关心的是它们在 runtime 里的结构位置。

更稳的看法是：

- 有的对象在接入**工作方式**
- 有的对象在接入**外部能力源**
- 有的对象在接入**更多执行者**
- 有的对象在接入**运行时接缝**
- 有的对象在接入**更完整的扩展封装**

所以它们不是平铺功能点，而是平台对象。

### 结论二：这几类对象都在扩展 Claude Code，但扩展的“层”并不一样

这是第 03 篇最重要的防混点。

如果只说“都是扩展”，那你很快就会分不清：

- skill 和 plugin 有什么不同
- MCP 和 tool 有什么不同
- agent 和 skill 有什么不同
- hook 和 plugin 又为什么不是一回事

更稳的判断应该是：

> **它们都在扩展 Claude Code，但扩展的是不同层。**

### 结论三：卷五后半的所有对象组，都应该挂回一张“runtime 接入关系图”上去理解

第 03 篇不是术语词典，也不是功能百科。

它真正的作用是给后文立一张骨架图，让后面的 skills 组、MCP 组、Agent 主轴、hooks 组和 plugins 组，都能挂回这张图上。

如果没有这张图，后面 22 篇很容易写成平铺介绍。

## 先画一张最简总图

可以先把卷五对象压缩成下面这张关系图：

- **skills**：把用户的方法、流程、角色结构接进 runtime
- **MCP**：把系统外部的能力源与资源系统接进 runtime
- **agents / subagents**：把更多执行者与协作结构接进 runtime
- **hooks**：把观察、注入、限制、改写等运行时接缝接进 runtime
- **plugins**：把前面这些扩展内容进一步收成更完整的封装、分发与复用单元

这张图里，最重要的不是对象数量，而是接入方向。

## skills 在接什么

### 它接入的是工作方式

skills 这条线，不是单纯补能力源，也不是再增加一个执行者。

它更像在做一件很特别的事：

> **把用户自己的方法、流程和角色结构，稳定编进 Claude Code 的工作过程中。**

所以 skills 更靠近：

- 做事方法
- 流程组织
- 角色化表达
- 可复用的任务套路

它接入的不是“系统外部资源”，而是“用户自己的工作现实”。

### 它为什么不是外部能力接入

skill 当然也可能间接调用工具，甚至调用外部能力。

但那不是它在总图里的第一职责。

更稳的说法是：

- MCP 更关心把外面的能力接进来
- skill 更关心把里面的方法组织起来

所以 skill 这条线，首先是工作方式接入层。

## MCP 在接什么

### 它接入的是系统外部能力源与资源系统

MCP 这条线要解决的问题更直接：

> **Claude Code 怎样稳定连接系统之外的能力与资源。**

这包括但不限于：

- 外部服务
- 远程能力
- 企业系统
- 文档与资源接口
- 系统外部的结构化资源来源

MCP 让 Claude Code 不只依赖自己内置的工具池，而是可以正式连接更广的能力来源。

### 它为什么不是工作方式组织层

MCP 很重要，但它不回答“某类任务应该怎样做”。

它更偏向回答：

- 能接到哪些外部能力
- 这些能力怎样被 runtime 看见和调用
- 系统之外的资源怎样进入当前工作链

所以 MCP 和 skills 的区别，不在“哪个更强”，而在“接入的对象根本不同”。

## agents / subagents 在接什么

### 它接入的是更多执行者与协作结构

前面两条线已经有了：

- 方法怎样进入系统
- 外部能力怎样进入系统

但复杂任务还会继续逼出第三个问题：

> **一个执行者不够时，Claude Code 怎样再长出更多执行者？**

这就是 agents / subagents 这条主轴的意义。

它接入的不是单个动作能力，也不是单个方法模板，而是：

- 新的工作承担者
- 新的执行分支
- 主 agent 与 worker agent 的协作结构
- 任务分发、回流与边界治理

### 为什么这里要把 subagent 放回 agent 主轴内部

第 03 篇这里要先埋下一个卷内硬判断：

> **subagent 不是和 skills / MCP / hooks / plugins 平级的另一类外部对象，而是 agent 主线内部继续展开出来的 worker / fork 形态。**

也就是说：

- agent 主轴前半段讲“更多执行者怎样长出来”
- 后半段讲“这些执行者怎样继续分叉成 worker / subagent 并保持可控协作”

所以总图里最好把它写成一条统一的执行者扩展主线，而不是拆成两个平级栏目。

## hooks 在接什么

### 它接入的是运行时接缝

到这里，Claude Code 已经能接：

- 方法
- 外部能力源
- 更多执行者

但平台化仍然还差一层。

因为系统不只需要“多一些对象”，还需要：

- 在关键节点可观察
- 在关键节点可拦截
- 在关键节点可注入
- 在关键节点可改写

这正是 hooks 的位置。

它接入的不是某个能力对象本身，而是：

> **runtime 在不同阶段留出来的结构化接缝。**

### 它为什么不能和 skills / plugins 混写

hook 有时候也像“可以加一段逻辑”。

但它和 skill、plugin 仍然不同。

- skill 更像组织做法
- plugin 更像完整封装单元
- hook 更像卡在运行时链条上的接缝点

所以 hook 的价值不在“也能扩展”，而在“扩展发生在什么时刻、什么位置、以什么干预语义进入”。

## plugins 在接什么

### 它接入的是更完整的扩展封装与交付单元

plugins 之所以要放在卷五后段，是因为它更像把前面这些扩展能力再往上收一层。

它不只是再提供一个能力点，而是在回答：

- 多类扩展内容能不能收成一个单元
- 这个单元能不能被统一装配和治理
- 能不能被安装、来源管理、更新与复用

所以 plugin 更靠近：

> **扩展内容怎样被正式打包成交付对象。**

### 它为什么不是所有扩展对象的总称

很多系统里“plugin”容易变成一个兜底词。

但卷五这里必须防住这种写法。

更稳的分层应该是：

- skills / MCP / agents / hooks：分别在不同层面把东西接入 runtime
- plugins：把这些扩展内容进一步组织成更完整的封装与分发形态

也就是说，plugin 不是总称，而是更高一层的封装对象。

## 把这几类对象压回一张分层图

如果把第 03 篇的总图再压缩一次，可以写成：

### 第一层：工作方式接入
- skills

### 第二层：外部能力源接入
- MCP

### 第三层：执行者扩展与协作结构接入
- agents / subagents

### 第四层：运行时接缝接入
- hooks

### 第五层：统一封装、分发与复用接入
- plugins

这不是严格的软件分层图，而是卷五写作上最重要的一张对象职责图。

它的作用是帮助读者记住：

> **扩展对象不是并排罗列，而是在不同方向上把 Claude Code 从“会执行的系统”继续推进成“可持续长能力的平台”。**

## 为什么这些对象不能混成一个“扩展大桶”

### 第一，因为一混写，边界就会立刻消失

如果你把这些对象都理解成“扩展 Claude Code 的东西”，虽然不算错，但几乎没有解释力。

因为你马上会分不清：

- 工作方式和能力源的区别
- 执行者和工具的区别
- 接缝和封装的区别

卷五后面之所以还要按组展开，正是为了把这些边界重新切稳。

### 第二，因为它们解决的问题不同

它们虽然都在扩展，但并不是在解决同一个问题：

- skill 解决“怎么做”怎样进入系统
- MCP 解决“能连什么外部能力”怎样进入系统
- agent 解决“谁来承担这段工作”怎样进入系统
- hook 解决“运行时哪里允许观察与干预”
- plugin 解决“这些扩展怎样成为更完整的交付单元”

这说明它们天然不能混讲。

### 第三，因为卷五真正要建立的是平台能力总图，而不是对象名词堆

第 03 篇必须防住一个最大跑偏：不要把后文写成对象百科。

更强的写法应该是让读者带着一张图继续往后读：

- 看到 skill，知道它在接工作方式
- 看到 MCP，知道它在接外部能力源
- 看到 agent，知道它在接执行者结构
- 看到 hook，知道它在接运行时接缝
- 看到 plugin，知道它在接统一封装层

这样后面的 04-25 才会越写越清楚，而不是越写越糊。

## 这篇不展开什么

### 1. 不把每类对象的正文细节提前讲完

这篇只立总地图，不吃掉后文分组职责。

### 2. 不转去讲命令入口和产品界面

卷五先讲 runtime 接入关系，命令层与产品整合留给卷六。

### 3. 不写术语词典

这里的重点不是定义最全，而是建立最稳的卷内骨架。

## 和前后文的边界

### 它承接第 02 篇

第 02 篇先说明：扩展权为什么交给用户。

第 03 篇接着回答：这些扩展权是通过哪些对象、以什么层次接入 runtime 的。

### 它导向第 04 篇以及后续各组

从第 04 篇开始，卷五就不再讲总图，而是回到对象主线，按：

> **skills → MCP → Agent 主轴 → hooks → plugins**

逐组展开。

## 一句话收口

> **skills、MCP、agents / subagents、hooks、plugins 不是并排功能点，而是一组以不同方式接入 Claude Code runtime 的平台对象：skills 接工作方式，MCP 接外部能力源，agents 接更多执行者与协作结构，hooks 接运行时接缝，plugins 接更完整的封装与交付形态；正因为它们接入的层不同，卷五后半才必须分组讲清。**
