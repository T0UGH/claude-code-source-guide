# 卷五 08｜skill、tool、agent 三者的边界到底是什么

## 导读

- **所属卷**：卷五：外部扩展与多代理能力
- **卷内位置**：08 / 25
- **上一篇**：[卷五 07｜什么样的 skill 才算好的 runtime skill](./07-what-makes-a-good-runtime-skill.md)
- **下一篇**：[卷五 09｜为什么 MCP 不是“多了一批远程工具”那么简单](./09-why-mcp-is-not-just-more-remote-tools.md)

## 这篇要回答的问题

skills 组到这里必须收一个边界。

因为前面几篇讲完后，读者脑子里很容易同时出现三种东西：

- tool 也能做事
- skill 也能推动事情往前走
- agent 好像也能完成完整任务

如果这里不把边界切稳，后面进入 MCP 和 Agent 主轴时就会互相吃线。

所以第 08 篇真正要回答的是：

> **skill、tool、agent 分别处在哪个层级，它们为什么不能互相替换？**

## 先给结论

### 结论一：tool 是执行能力，skill 是能力组织，agent 是执行者结构

这是第 08 篇最该留下来的一句话。

三者当然彼此相关，但它们回答的问题并不一样：

- **tool** 回答的是：系统现在能直接调用什么动作能力
- **skill** 回答的是：这些能力该按什么方法和流程被组织起来
- **agent** 回答的是：这段工作该由哪个执行者承担，以及执行者怎样被装配和派生

一旦这三层分开，很多混乱就会立刻消失。

### 结论二：skill 虽然会调 tool，也可能触发 agent，但它本身既不是动作原语，也不是执行者本体

这正是 skill 最容易被误读的地方。

因为它站在中间：

- 往下会碰到 tools
- 往上又可能碰到 fork / worker / agent 路径

但“站在中间”不等于“兼任两边”。

skill 更准确的角色是：

> **把工具能力和执行者结构编排成某种稳定工作方法的那一层。**

### 结论三：边界切清之后，卷五后半的对象关系才不会糊成一个“大扩展桶”

如果不切边界，你就会得到这种错误理解：

- tool 是小 skill
- skill 是轻量 agent
- agent 是高级 tool

这三种说法听起来都像有点道理，但其实都在把层级压扁。

而卷五要做的，正好是反过来：

> **把不同对象放回它们各自真正的 runtime 位置。**

## 先看 tool：它解决的是“能做什么”

tool 是 Claude Code 最直接的一层执行能力。

它的核心特点是：

- 有明确动作
- 有清楚输入输出
- 被 runtime 正式调用
- 结果会直接回到当前 turn

例如：

- Bash 跑命令
- Read 读文件
- Edit 改文本
- Search / Grep 找材料

这些对象共同回答的是：

> **系统现在到底能直接做哪些动作。**

所以 tool 的基本视角是动作原语。

它当然也能组合，但它自己并不负责给整段工作定义方法论。

## 再看 skill：它解决的是“这些能力该怎么被组织起来”

skill 之所以夹在中间，是因为它不像 tool 那样只回答单个动作，也不像 agent 那样负责成为独立执行者。

它更像一层能力组织：

- 哪些步骤先做
- 哪些工具该怎样搭配
- 什么边界必须守住
- 输出应该长什么样
- 哪些情况下该切成独立工作包

所以 skill 解决的不是“有没有这个能力”，而是：

> **怎样把已有能力组织成一个稳定工作流。**

这也是为什么前面会反复说：

- skill 不是长 prompt
- skill 是方法组织层
- 好 skill 关键在职责和边界

因为它本来就不该被理解成动作对象。

## 最后看 agent：它解决的是“谁来做，以及怎样继续长出更多执行者”

agent 的层级比 tool 和 skill 都更靠执行者结构。

它关注的不是某个动作本身，也不是一套方法模板本身，而是：

- 当前是谁在承担这段工作
- 这个执行者怎样被装配起来
- 是否需要派生 worker / subagent
- 结果怎样回流给主执行者

所以 agent 回答的问题更像：

> **这段工作由哪个执行者承担，以及执行者之间怎样协作。**

这也是为什么卷五后面会把 agent / subagent / fork worker 合成一条执行者主线。

因为这里讨论的已经不是方法组织，而是执行者扩展。

## 为什么说 skill 不能被压成“高级 tool”

这种误解很常见，因为 skill 看起来也能让事情往前走。

但它和 tool 至少有三层差别：

### 1. tool 直接做动作，skill 组织动作

tool 的价值，在于动作本身可调用。

skill 的价值，在于动作之间的组织关系可复用。

### 2. tool 的完成通常局部明确，skill 的完成更像一段工作流闭环

tool 常常是：

- 读到了
- 改完了
- 跑完了

skill 则更常是：

- 这轮分析做完了没有
- 这套写作流程走稳了没有
- 这次排查是不是按正确顺序推进

### 3. tool 不负责定义方法论，skill 天生就带方法论

当然也有复杂工具，但它们复杂的是动作能力，不是工作方法。

skill 则恰好相反：它主要承载的正是方法。

## 为什么说 skill 也不能被压成“轻量 agent”

这同样很容易误判，尤其当 skill 走 fork 路径时更像如此。

但这里仍然要切开。

### 1. agent 是执行者，skill 是执行者使用的方法模块

即使某个 skill 会触发 worker 或指定 agent，它也仍然更像：

- 给执行者一套工作方法
- 或把某段工作包装成可被执行者接手的工作包

它本身并不等于那个执行者。

### 2. agent 关心的是执行者装配与协作，skill 关心的是工作流组织

agent 主线里真正重的问题是：

- 更多执行者怎样长出来
- runAgent 怎样把执行者装起来
- worker / fork 怎样分叉与回流

这些都不是 skill 的中心问题。

skill 可能碰到这些结构，但不以这些结构为主语。

### 3. skill 可以触发角色切换，但不能替代角色本体

这是第 05 篇提“角色结构”时最该保住的边界。

skill 当然可以声明：

- 这件事适合独立处理
- 这一段适合交给 worker

但这和“skill 自己就是执行者”仍然是两回事。

## 三者怎样协作，才是比较稳的理解

如果把三者压成一条协作线，我会这样写：

1. **tool 提供动作能力**
2. **skill 组织这些能力的使用方式**
3. **agent 承担并分配整段工作的执行责任**

这样理解之后，你会发现很多东西立刻顺了：

- 为什么 skill 能调用 tool，却不等于 tool
- 为什么 skill 能碰到 fork，却不等于 agent
- 为什么 agent 会使用 skill 和 tool，但也不是它们的别名

这三者最自然的关系，不是替代，而是分层协作。

## 从卷五顺序看，第 08 篇为什么必须最后写

这也是 skills 组为什么要把第 08 篇后置的原因。

如果第 08 篇太早写，它很容易变成空泛术语文。

因为边界判断必须建立在前面几件事已经成立之后：

- 先知道 skill 接进来的是用户工作方法（05）
- 再知道 skill 怎样正式进入执行链（06）
- 再知道什么样的 skill 才真适合 runtime（07）

到这一步，skill 的地位已经比较稳了，第 08 篇才能真正把它和 tool / agent 切开，而不是只做概念对比。

同时，第 08 篇也必须在进入 MCP 和 Agent 主轴之前完成，因为它要给后两组稳住坡度：

- skill 不要去吃 MCP 的“外部能力源”职责
- skill 不要去吃 agent 的“执行者结构”职责

tool 的位置在卷三已经解释过，这里只取边界结论。

### 2. 不抢 agent 主轴正文

这里只把 agent 先定义成执行者结构，不展开 runAgent、forkSubagent 和信息回流。

## 一句话收口

> **tool 是执行能力，skill 是能力组织，agent 是执行者结构；skill 可以调用 tool，也可能触发 agent 路径，但它本身既不是动作原语，也不是执行者本体，而是把能力与执行责任编排成稳定工作方法的中间层。**
