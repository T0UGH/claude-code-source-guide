# 卷五 02｜为什么 Claude Code 选择把扩展权交给用户

## 导读

- **所属卷**：卷五：外部扩展与多代理能力
- **卷内位置**：02 / 25
- **上一篇**：[卷五 01｜为什么复杂场景会逼 Claude Code 长出扩展层](./01-why-complex-scenarios-force-claude-code-to-grow-an-extension-layer.md)
- **下一篇**：[卷五 03｜skills / MCP / agents / subagents / hooks / plugins 是怎样接入 Claude Code 的](./03-how-skills-mcp-agents-subagents-hooks-and-plugins-enter-claude-code.md)

## 这篇要回答的问题

第 01 篇已经先立住了卷五的总前提：

> **复杂场景不是边缘案例，而是 Claude Code 必须长期面对的常态。**

一旦这个前提成立，系统就必须继续扩展。

但扩展还有两条完全不同的路：

- 一条是产品团队继续不断内建更多能力
- 另一条是产品团队把扩展权交给用户，让用户把自己的能力、方法、资源和结构接进来

Claude Code 选择的显然更靠近后者。

所以第 02 篇要回答的是：

> **为什么 Claude Code 不把自己做成一个尽量预装一切的封闭产品，而是选择把扩展权交给用户？**

## 先给结论

### 结论一：因为复杂场景的变化太贴近现场，只有用户最先知道“下一步到底缺什么”

产品团队当然可以预装很多常见能力。

但在真实工作里，最先感受到能力缺口的人，通常不是平台开发者，而是正在现场做事的用户。

用户会最先知道：

- 这个项目真正依赖哪套流程
- 这个团队到底怎么分工
- 这个公司有哪些内部系统必须接入
- 这个任务在本地到底需要怎样的审查、验证、审批或回流方式

这些信息高度现场化，也变化极快。

所以如果系统还坚持把扩展权只握在产品团队手里，就会出现一个根本矛盾：

> **最知道缺什么的人，反而最没有权力把缺的东西接进来。**

Claude Code 选择把扩展权交给用户，本质上是在消除这个矛盾。

### 结论二：因为 Claude Code 的强，不在于预装了多少能力，而在于它允许用户把自己的工作方式正式编进 runtime

一个系统当然可以靠内建能力变得好用。

但如果它想在复杂场景里持续变强，仅靠预装远远不够。

Claude Code 更关键的一步，是把增长方式改了：

- 不只靠产品团队补功能
- 也允许用户把自己的做事方法接进来
- 允许用户把外部能力源接进来
- 允许用户把新的执行者结构接进来
- 允许用户把运行时干预点和封装方式接进来

也就是说，Claude Code 真正交出去的，不只是“添加功能”的权限，而是：

> **把用户自己的工作现实接进系统的权力。**

### 结论三：把扩展权交给用户，不是放弃产品设计，而是把产品做成平台

很多人看到“交给用户扩展”，会下意识以为：

- 产品团队偷懒了
- 系统不再负责一致性了
- 平台只是把问题甩给用户

但更稳的理解恰好相反。

如果没有稳定的 runtime、边界、治理、接入层级和对象分工，扩展权根本交不出去。

所以“交给用户”不是不设计，而是另一种更重的设计：

> **先把平台边界、接入对象和运行机制设计好，再让用户在这个结构里继续长能力。**

## 为什么“继续靠产品团队内建”会遇到天花板

### 第一，因为产品团队只能覆盖共性，覆盖不了所有局部现实

中心化内建最擅长解决的是共性问题。

比如：

- 通用执行链
- 通用工具
- 通用权限控制
- 通用上下文治理
- 通用接入机制

这些都应该由产品层来做。

但复杂场景里的大量关键差异，恰恰不是共性，而是局部现实：

- 某个公司的内部审批流
- 某个团队的文档结构和知识入口
- 某类项目特有的测试、评审和上线要求
- 某个作者自己的写作流程
- 某种场景下必须遵守的特殊权限边界

这些东西如果都等产品团队一个个内建，系统就会变得越来越重，也越来越慢。

### 第二，因为变化速度快过产品发布速度

复杂场景不是静态集合，而是不断长新的需求。

如果每出现一种工作方式、资源来源、协作结构、治理要求，都要先等产品团队：

- 理解问题
- 评估优先级
- 设计方案
- 发布版本

那么系统的适应速度就一定慢于现场变化速度。

而用户自己就在现场。

把扩展权交给用户，本质上是在把适应速度拉回现场。

### 第三，因为产品团队最该内建的，不是所有内容，而是“可扩展结构”

这其实是平台设计里最关键的一次角色转换。

产品团队最值钱的工作，不应该是替每个用户写完所有局部能力，而应该是把这些问题变成：

- 哪些对象可以被扩展
- 扩展对象怎样进入 runtime
- 对象之间怎样分层
- 权限、状态、治理怎样守住边界

换句话说：

> **真正该被内建的，是扩展的结构，而不是所有扩展的内容。**

## “把扩展权交给用户”具体意味着什么

### 1. 意味着用户可以把自己的方法接进系统

这是 skills 那条主线要重点展开的部分。

用户最有价值的东西，往往不是再给系统补一条知识，而是：

- 某类任务该怎么做
- 某个流程该怎么走
- 某种角色应该怎样说话、判断和交付

这些内容如果只能停留在聊天里，就很难稳定复用。

把扩展权交给用户，意味着这些方法可以被沉淀成系统能够重新调入的对象。

### 2. 意味着用户可以把系统外部能力源接进来

这是 MCP 那条主线要展开的部分。

Claude Code 如果只活在自身工具池里，很快就会被封闭边界困住。

把扩展权交给用户，也意味着用户可以接入：

- 外部服务
- 远程资源
- 企业内部系统
- 文档、知识和资源接口

这样系统才不会只会调用自己认识的那点东西。

### 3. 意味着用户可以让系统长出新的执行结构

这是 Agent 主轴要展开的部分。

复杂任务经常不是一个执行者能优雅扛到底的。

把扩展权交给用户，也意味着系统允许：

- 定义新的执行者
- 把某段任务交给新的承担者
- 派生 worker / subagent
- 把协作结构也纳入 runtime

这已经不是“加功能”那么简单，而是在扩展执行者体系。

### 4. 意味着用户可以在运行时接缝上施加自己的规则和干预

这是 hooks 那条主线要展开的部分。

真实工作不只需要能力，也需要：

- 观察点
- 审核点
- 注入点
- 改写点

把扩展权交给用户，意味着 runtime 的关键接缝不再是完全封死的。

### 5. 意味着用户还能把一整组扩展能力收成更完整的交付单元

这是 plugins 那条主线要展开的部分。

扩展权如果只停在“本地随便改改”，平台成熟度还是不够。

更进一步的能力是：

- 把多类扩展对象收成一个单元
- 让它可以安装、治理、分发和复用

这才是平台层真正成熟的表现。

## 为什么这不是“把问题甩给用户”

### 第一，因为交出去的是扩展权，不是基础责任

Claude Code 并没有把所有事情都交给用户。

它仍然自己负责：

- 执行链
- 权限与安全边界
- 上下文治理
- 基础工具层
- runtime 接入结构
- 对象分层与治理能力

这些基础责任如果没立住，用户扩展只会变成混乱堆叠。

所以正确理解应该是：

- 产品负责打底
- 用户负责把现场现实接进来
- runtime 负责把这些对象收成可运行结构

### 第二，因为平台真正交付的是“表达差异的空间”

复杂场景之所以复杂，就在于不同用户、不同项目、不同组织之间确实存在差异。

如果系统完全不允许这种差异进入，那它就只能要求现实去配合产品。

Claude Code 反过来做了一件更难的事：

> **给现实差异留下正式空间。**

这不是甩锅，而是承认现实本来就不统一。

### 第三，因为把扩展权交给用户，反而更有机会守住一致性

这听起来像反直觉，但其实很重要。

如果没有正式扩展结构，用户一样会扩展，只不过方式会变成：

- 临时复制 prompt
- 手工拼命令
- 到处塞脚本
- 靠记忆维持流程
- 靠聊天上下文临时传递角色规则

这种扩展当然更混乱，也更难治理。

把扩展权正式交出去，反而能把这些零散做法收回到可见、可控、可复用的对象结构里。

## 从卷五总线看，第 02 篇到底立住了什么

### 立住一：扩展层不是“系统补缺口”，而是“系统承认用户现场”

卷五不是要写一个“平台多开放”的宏大口号。

第 02 篇更具体的作用，是把第 01 篇的复杂场景压力，继续推进成一个平台判断：

> **既然复杂场景无法由中心化内建完全覆盖，系统就必须承认用户的现场知识、资源与方法，并给它们正式接入的权力。**

### 立住二：后面的对象组都不是平白出现的

到了这里，后面五类对象的存在理由就都更清楚了：

- skills：因为用户方法必须能进入系统
- MCP：因为系统外部能力源必须能进入系统
- agents / subagents：因为更多执行者必须能进入系统
- hooks：因为运行时干预点必须能进入系统
- plugins：因为扩展封装与交付也必须能进入系统

这会让第 03 篇的总地图不只是对象清单，而是有同一条前提支撑的对象地图。

## 这篇不展开什么

### 1. 不把各类扩展对象的 runtime 接入关系讲完

那是第 03 篇的任务。

### 2. 不把命令入口、界面入口讲成重点

卷五先讲扩展层成立，入口层和产品整合留给卷六。

### 3. 不写成开放平台宣言

这里不是讲价值观口号，而是讲结构性选择。

## 和前后文的边界

### 它承接第 01 篇

第 01 篇先说明扩展层为什么必然出现。

第 02 篇继续问：既然要扩展，为什么扩展权最终要落到用户这一侧。

### 它导向第 03 篇

当“复杂场景压力”与“扩展权交给用户”都立住后，下一步自然就是：

> **这些扩展权到底是通过哪些对象、以什么方式接入 Claude Code runtime 的？**

这就是第 03 篇。

## 一句话收口

> **Claude Code 选择把扩展权交给用户，不是因为产品不想继续内建，而是因为复杂场景的变化太贴近现场：最先知道缺什么的人正是用户；平台真正该内建的不是所有局部能力，而是让用户把自己的方法、资源、执行结构和干预规则正式编进 runtime 的接入结构。**
