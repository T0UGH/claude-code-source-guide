# 卷五 10｜Claude Code 是怎样通过 MCP 接入外部能力源和资源系统的

## 导读

- **所属卷**：卷五：外部扩展与多代理能力
- **卷内位置**：10 / 25
- **上一篇**：[卷五 09｜为什么 MCP 不是“多了一批远程工具”](./09-why-mcp-is-not-just-more-remote-tools.md)
- **下一篇**：[卷五 11｜MCP 和 skills / hooks / plugins 分别是什么关系](./11-how-mcp-relates-to-skills-hooks-and-plugins.md)

## 这篇要回答的问题

到了 MCP 组，真正要先立住的，不是“Claude Code 支持多少 MCP server”，也不是“怎么配一个远程连接”。

第 10 篇要回答的是更核心的问题：

> **Claude Code 到底是怎样通过 MCP，把系统外面的能力源和资源系统接进 runtime 的？**

如果这里没有讲清，MCP 很容易被读成两种过浅的东西：

- 一批放在远端的 tool
- 一套把外部 API 包装进来的接线协议

这两种说法都碰到了一部分表面，但都还没碰到卷五真正关心的主线。

卷五关心的不是“能不能连”，而是：

> **外部世界里的能力，怎样从系统外的散乱存在，变成 Claude Code runtime 里可被稳定编排、调用和治理的一部分。**

所以第 10 篇是 MCP 组的锚点篇。它必须把 MCP 立成一条主线：

> **外部能力源接入主线。**

## 先给结论

### 结论一：MCP 接进来的不是几把新工具，而是一整层系统外能力源

把 MCP 理解成“远程工具接入”，不算全错，但太窄。

因为 Claude Code 通过 MCP 接住的，往往不是单个动作，而是一个更完整的外部能力面：

- 某个外部服务能提供的动作集合
- 某个资源系统里可读、可引用、可浏览的对象集合
- 某个需要认证、连接、状态维护的系统边界

也就是说，MCP 接进来的不是一把孤立扳手，而更像一整面可持续进入 runtime 的外部能力墙。

### 结论二：MCP 的关键，不在“连上去”，而在“把外部能力稳定编入 runtime”

如果只是偶尔请求一个远程 API，Claude Code 并不一定非要 MCP。

真正让 MCP 在卷五里有分量的地方，是它把外部能力接入这件事，从零散调用，抬升成了 runtime 正式处理的对象关系。

一旦这样看，重点就不再只是：

- 地址怎么配
- 协议怎么握手
- 返回值怎么长

而会转成：

- 这个外部能力节点怎样被系统识别
- 它当前有没有连通、有没有认证、能不能被安全调用
- 它带来的 tools 和 resources 怎样进入当前工作面
- 它们怎样继续接受权限、上下文和运行时治理

### 结论三：MCP 让 Claude Code 能把“系统外面有什么”纳入自己的工作组织

skills 让用户的方法进入系统，agent 主轴让系统长出更多执行者；而 MCP 解决的是另一类问题：

> **系统外面的能力和资源，怎样被正式纳入 Claude Code 的工作组织。**

这正是它在卷五里的位置。

## 先把角色说清：这篇讲什么，不讲什么

这篇是 **机制锚点篇**，但它不是协议教程。

它要完成三件事：

1. 讲清 MCP 在 runtime 里的接入位置
2. 讲清能力源、资源系统与 runtime 的关系
3. 讲清为什么它属于卷五主线，而不是工具补充

这篇**不**展开：

- 不细讲所有协议实现
- 不展开认证状态机细节
- 不细拆每一种资源类型
- 不抢 hooks / plugins 的边界职责

## 为什么说 MCP 接的是“能力源”和“资源系统”

### 第一层：能力源，不只是动作列表

很多外部系统一旦接进 Claude Code，带进来的首先当然是动作能力。

比如：

- 搜索某个平台的数据
- 调用某个服务的特定接口
- 创建、修改、查询某类远端对象

但如果只把这些看成“动作函数”，MCP 会被写得过薄。

因为 Claude Code 面对的常常不是一个函数，而是一个外部系统的能力面。

这个能力面有自己的：

- 命名空间
- 连接状态
- 身份边界
- 可见对象
- 可被发现和枚举的操作集合

所以更稳的说法不是“多了一些远端 tool”，而是：

> **系统多接进来了一块外部能力源。**

### 第二层：资源系统，不只是执行动作

MCP 另一层经常被低估的价值，是它不只带动作，还带资源。

也就是说，Claude Code 接入的未必只是“你能对外部系统做什么”，还包括“你能从外部系统里稳定拿到什么”。

这类东西可能包括：

- 结构化对象
- 文档、记录、配置、条目
- 某类可浏览、可读取、可引用的数据面

一旦卷五把这一层看见，MCP 的系统地位就会明显上升。

因为 Claude Code 不再只是偶尔向外打一枪，而是在把一个外部资源系统纳入自己可工作的材料世界。

### 第三层：连接世界，不只是调用世界

本地工具更多解决的是“当前机器上能做什么”。

MCP 则更像是在解决：

> **Claude Code 怎样和系统边界之外的世界建立正式连接。**

这层“连接世界”的意义很大，因为它意味着 runtime 要开始处理一些本地工具不那么强调的问题：

- 这个能力节点现在是否可达
- 它是否处于已认证状态
- 它的资源面和动作面是否已经可见
- 当状态变化时，系统如何持续管理它

这已经不是“再加几个函数”能概括的了。

## MCP 在 runtime 里到底站在哪

### 1. 它不在用户方法层

用户的方法层主要由 skills 负责。

skills 更像告诉 Claude Code：

- 这类事该怎么做
- 先后顺序是什么
- 哪些边界要守住
- 哪些角色该切换

MCP 不处理这些。

它不规定工作方法，它提供外部能力面。

### 2. 它也不在执行者层

执行者层是 agent 主轴的职责。

agent / subagent / worker 解决的是：

- 谁来执行
- 怎样拆分执行
- 如何在多个执行者之间协作

MCP 不是执行者。

它更像执行者能接入、能使用、能读取的一块外部能力源。

### 3. 它站在 runtime 的外部能力接入层

如果把卷五对象放回同一张 runtime 地图里，MCP 最适合站的位置是：

- 不像 skill 那样组织工作方法
- 不像 agent 那样形成执行者结构
- 而是在 runtime 外侧，把系统外面的能力面和资源面正式接进来

所以它更接近：

> **外部能力接入层。**

## Claude Code 是怎样把这层外部能力接进来的

这里不做协议分步讲解，只按卷五需要的结构来讲。

### 第一段：先把外部能力节点识别成正式对象

外部系统如果只是“那里有个 API”，对 runtime 来说仍然太散。

Claude Code 真正需要做的第一步，是把这个外部能力节点识别成系统可处理的对象：

- 它是哪一个 server / integration
- 它当前是什么连接状态
- 它暴露了哪些可用能力
- 它是否已经通过认证

一旦做到这一步，系统对它的处理就不再像对一段裸 URL 或一串临时请求，而像对一个持续存在的能力节点。

这一步其实很关键。

因为卷五真正关心的是“外部能力怎样进入系统秩序”，而不是“外部能力怎样偶尔被碰一下”。

### 第二段：再把它暴露出来的能力面编进当前工作面

当外部能力节点被系统识别后，Claude Code 才能继续把它带来的东西编进当前工作面。

这里最容易看见的是 tools，但不止于 tools。

更完整地说，它会把外部系统暴露出来的：

- 可执行动作
- 可读取资源
- 与当前任务相关的外部对象

一起拉进 Claude Code 的可工作范围。

于是模型和 runtime 面对的，不再只是内建工具池，而是一个已经纳入外部能力面的工作环境。

### 第三段：让这些外部能力继续接受 runtime 治理

这一步才是 MCP 真正值钱的地方。

如果外部能力进来之后，就完全游离于系统治理之外，那它依然只是挂在边上的连接器。

但 Claude Code 并不是这么处理的。

外部能力一旦被接进来，还要继续接受运行时管理，例如：

- 可见性与发现
- 认证状态与连接状态
- 权限判断
- 调用时的错误翻译与回退
- 在会话工作流中的持续可用性

所以 MCP 干的不是“拉一根线进来”，而是：

> **把系统外能力拉进来后，再继续纳入 Claude Code 自己的运行秩序。**

## 为什么这条线必须叫“外部能力源接入主线”

### 因为它解释的是 Claude Code 怎样长出系统外能力面

卷五不是在数扩展点，而是在解释系统怎样长能力。

skills 长的是用户方法层。
agent 主轴长的是执行者层。
hooks 长的是运行时接缝层。
plugins 长的是统一封装层。

而 MCP 长出来的，是：

> **系统外能力边界。**

也可以换一种更适合本组的说法：

> **外部能力源接入主线。**

### 因为它把“能力在哪里”也纳入了 Claude Code 的工作图

没有 MCP 的时候，Claude Code 的工作重心更集中在系统内部：

- 内建 tools
- 当前上下文
- 已接入的方法结构
- 已存在的执行者

有了 MCP，系统开始认真回答另一类问题：

- 外部还连着哪些能力源
- 哪些资源系统可以直接纳入当前工作
- 哪些外部边界已经成为 runtime 可支配的一部分

换句话说，MCP 让 Claude Code 不只组织“怎么做”，还开始组织“外面还接着什么”。

## 这条主线为什么不该写成接入教程

因为教程回答的是“你怎么配”。

而卷五这里要回答的是：

- 为什么这层对象在系统里有独立地位
- 它解决的到底是哪一类扩展问题
- 为什么它不能被写成 tool 的附录

如果第 10 篇写成接入教程，读者最后只会记住：

- MCP 要配 server
- MCP 能连资源
- MCP 会带来 tools

但记不住更重要的模型：

> **Claude Code 通过 MCP，把系统外部能力源和资源系统正式编入 runtime。**

这才是本篇该留下来的东西。

## 和前后文的边界

### 它承接第 09 篇

第 09 篇负责先打掉浅理解：MCP 不是“多了一批远程工具”。

第 10 篇则把正面模型立住：MCP 到底怎样承担外部能力源接入主线。

### 它导向第 11 篇

等这条主线立住后，第 11 篇才适合收边界：

- 它不等于 skills 的方法封装

## 一句话收口

> **Claude Code 通过 MCP 真正接进来的，不是几把摆在远端的新工具，而是一层可被 runtime 正式识别、连接、治理并持续编排的外部能力源与资源系统；MCP 的关键不在“连上去”，而在它让系统外部能力能够稳定编入 Claude Code 的工作主线。**
