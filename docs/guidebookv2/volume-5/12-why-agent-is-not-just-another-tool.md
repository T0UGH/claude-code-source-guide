## 这篇要回答的问题

一进入 Agent 主轴，最常见的误解通常是：

> **agent 不就是比 tool 更聪明一点、能一次干更多活的工具吗？**

这个理解之所以容易出现，是因为 agent 看起来确实也会推动事情往前走，甚至看上去比普通 tool 更完整。

但如果停在这里，整个 Agent 主轴都会被写歪。

因为一旦把 agent 压成“高级工具”，后面这些问题就全会糊掉：

- 为什么还需要 runAgent 这种装配主干
- 为什么还会出现 subagent / worker / fork
- 为什么执行者之间会有职责边界和信息回流

所以第 12 篇的任务，不是把 agent 全讲完，而是先打掉这个入口误解。

## 先给结论

### 结论一：tool 解决的是动作执行，agent 解决的是执行者承担

tool 更像动作原语。

它回答的是：

- 现在能做什么动作
- 这个动作怎么输入输出
- 做完后把结果怎么回到当前 turn

agent 则不是动作原语。

它更像在回答：

- 这段工作该由谁承担
- 这个承担者带着什么上下文和能力面去做
- 是否还需要继续分叉出别的 worker

所以两者根本不在同一层。

### 结论二：agent 不是“大一点的 tool”，而是 Claude Code 开始长出更多执行者时出现的结构性对象

一个 tool 再复杂，也仍然主要是在完成动作。

agent 不同。

它的出现意味着 Claude Code 开始把“执行责任”也正式纳入 runtime：

- 可以有一个执行者来接一段工作
- 这个执行者不是动作集合，而是工作承担者
- 它还能继续派生出后续执行结构

所以 agent 的意义不在强弱，而在结构层级。

### 结论三：如果把 agent 看成工具，subagent 就会被误写成“工具套工具”

这正是卷五必须避免的。

Agent 主轴后半段要写的是：

- 主 agent 为什么还要派生 subagent
- forkSubagent 为什么是 worker 分叉
- 结果怎样回流

这些都建立在一个前提上：

> **agent 是执行者，不是工具。**

## 为什么“另一个工具”这个理解不够

### 第一，因为 tool 的完成单位通常是动作，agent 的完成单位更像一段工作

一个 tool 往往有比较清楚的动作边界：

- 读文件
- 改文本
- 执行命令
- 查询数据

它可能很复杂，但它的核心仍然是动作调用。

agent 则不同。

它接住的常常不是单个动作，而是一段要继续推进的工作：

- 先判断，再规划，再执行
- 必要时再调 tool
- 必要时再拆分给后续 worker
- 最终交回一份可继续承接的结果

这已经不再是动作原语的视角了。

### 第二，因为 tool 一般不承担执行者装配问题，agent 则天生连着装配问题

tool 被 runtime 调起来后，通常直接进入动作执行。

agent 这条线却天然会碰到更多结构问题：

- 这个执行者是谁
- 它继承什么系统提示和上下文
- 它能用哪些工具和能力面
- 权限怎么叠加
- 它完成后怎样退出、怎样回流

这些都说明，agent 不是“会做更多动作的函数”，而是要被装成一个可运行执行体。

### 第三，因为 tool 很少天然导向更多执行者，agent 则天然导向协作结构

Claude Code 一旦有 agent，后面很自然就会继续碰到：

- 什么时候需要派生 subagent
- 为什么 fork 不是随便复制
- 多个执行者之间怎样保持边界

这说明 agent 一开始就不是孤立对象，而是一条执行者主线的起点。

把它理解成另一个工具，会直接挡住这条主线。

## agent 到底更像什么

### 1. 更像工作承担者，而不是动作能力包

工具包的核心是“会做什么”。

工作承担者的核心是“接住什么工作，并把它继续推进”。

agent 更像后者。

它当然会使用工具，但它自己不是工具集合的别名。

### 2. 更像执行结构里的节点，而不是工具列表里的成员

tool 常常站在工具池里，等待被调用。

agent 则更像执行结构中的一个节点：

- 可以接活
- 可以继续组织工作
- 可以把某些工作再往后分出去

也就是说，它属于执行结构，而不是动作目录。

### 3. 更像运行时对象，而不是提示词包装

很多人误判 agent，本质上是把它想成：

- 一段特殊 prompt
- 一个带说明的 tool
- 一个会自动多步执行的函数壳

但如果把它放回 Claude Code runtime 去看，更稳的说法是：

> **agent 是一种正式的执行者对象。**

它要被装配、被约束、被治理，也可能再派生新的执行者。

## 从卷五主线看，agent 到底补上了什么

### 补上的不是动作能力，而是执行责任分层

Claude Code 前面已经有：

- tool：动作能力
- skill：能力组织
- MCP：外部能力接入

到了 agent 这一层，它补上的不是新动作，而是：

> **执行责任可以被进一步分层、转交和展开。**

这是另一种扩展能力。

### 补上的不是“更强的函数”，而是“更多承担者”

如果系统只有一个执行者，那么再多工具、再多方法，也仍然可能只有一条主执行线。

agent 的出现，让 Claude Code 开始从“一个执行者调用很多能力”，走向“多个执行者协作承担工作”。

这才是卷五真正关心的增长。
