## 这篇要回答的问题

到了第 15 篇，Agent 主轴已经完成了前半段：

- 12 打掉“agent 是另一个工具”的误解
- 13 立住“更多执行者”这条主线
- 14 把 runAgent 立成装配主干

接下来，主线自然要进入后半段：

> **既然已经有 agent 了，为什么主 agent 还要继续派生 subagent / worker？**

这个问题如果答不好，subagent 就会显得像可有可无的附属功能。

但卷五这里必须立住的判断是：

> **subagent 不是附属能力，而是主执行者面对复杂任务时进一步展开协作结构的必要结果。**

## 先给结论

### 结论一：有了 agent，不等于单个执行者就足够了

agent 的出现，确实已经让 Claude Code 从“动作调用”走向“执行者承担”。

但复杂任务继续加深时，问题会从“有没有执行者”变成：

- 一个执行者要不要同时扛所有局部
- 主线判断和支线处理要不要隔离
- 某些工作是不是更适合在独立边界里推进

这时候，只有一个 agent 仍然可能不够。

### 结论二：subagent 的意义，不是多开一个名义上的 agent，而是把主执行线继续展开成协作结构

主 agent 一旦要派生 subagent，真正发生的变化不是“数量 +1”，而是：

- 某段工作被明确拆出去
- 它在独立边界内继续推进
- 主线程可以继续维持自己的判断与整合职责
- 最终结果还能回流到主线

所以 subagent 是执行者主线的后半段，而不是另一组对象。

### 结论三：subagent 回答的是“主执行者怎样继续工作而不是怎样独自硬扛”

Claude Code 真正想解决的，不只是“能不能做复杂任务”，而是：

> **复杂任务里，主执行者怎样在不失控的前提下继续展开分工。**

这正是 subagent 出现的系统理由。

## 为什么“已有 agent”仍然不够

### 第一，因为主线执行者同时承担判断、组织和局部处理时，很容易互相打断

复杂任务里，主 agent 往往同时要做几件事：

- 维持总体目标
- 判断下一步方向
- 整理中间结论
- 亲自去处理局部工作

如果所有事都压在同一条执行线上，系统很容易出现：

- 主线判断被局部细节拖住
- 中间状态被频繁覆盖
- 某些支线问题没有稳定隔离空间

所以 subagent 的第一价值，就是把一部分工作从主线中拆出去。

### 第二，因为有些任务天然需要隔离执行面

并不是所有工作都适合在同一执行面里完成。

有些任务更适合：

- 在更窄的上下文里单独处理
- 用更清楚的职责边界独立推进
- 完成后只把结果带回，而不是把整个过程混进主线

这时派生 subagent 的意义，不是显得更高级，而是为了让执行结构更稳。

### 第三，因为主 agent 需要保住“协调者”角色，而不是退化成所有子任务的亲自操作者

如果主 agent 什么都亲自做，它虽然仍叫主 agent，但结构上已经很像单执行者硬扛。

Claude Code 继续长出 subagent，本质上是在保住主 agent 的另一种职责：

- 主线判断
- 工作分配
- 结果整合
- 继续推进

这说明主 agent 真正的价值，不只是做事本身，还包括组织后续执行者。

## subagent 为什么必须放回 Agent 主轴，而不能单列一组

### 1. 因为它不是另一类平级对象，而是 agent 主线继续分化出来的协作形态

卷五这里必须特别防止一种写法：

- 前面讲 agent
- 后面再平行讲 subagent

这样会让人误以为 subagent 是另一种独立对象类型。

但更稳的理解是：

> **subagent 是 agent 主线继续往后长出来的 worker / 分叉形态。**

它属于同一条执行者主线，只是位置更靠后。

### 2. 因为它回答的是“执行者怎样继续分工”，不是“系统又多一个对象”

如果把 subagent 写成对象目录里的新成员，重点就会跑偏到：

- 它也是一种 agent
- 它和主 agent 不同
- 它有自己的能力

这些都不算错，但都不是卷五此处最重要的重心。

卷五更重要的问题是：

- 主执行者为什么会继续分工
- 分工怎样进入 runtime 结构
- 怎样从单执行者推进到协作执行结构

subagent 正好回答这组问题。

## 主 agent 派生 subagent，到底改变了什么

### 改变一：把“继续做”变成“继续分工地做”

没有 subagent 时，主线的推进更多依赖一个执行者自己切换焦点。

有了 subagent 后，系统开始可以：

- 让主线继续做主线判断
- 让支线在独立执行面里推进
- 最后再把结果接回来

这说明 Claude Code 不只是继续做事，而是开始继续分工地做事。

### 改变二：把复杂任务从单线推进，改成受控协作推进

这里最重要的是“受控”两个字。

subagent 不是失控复制，也不是自由发散。

它的价值恰恰在于：

- 是主 agent 派生出来的
- 是为了特定子任务服务的
- 最终还要重新回到主线

所以它代表的是受控协作，而不是多开窗口。

### 改变三：给后续的 fork worker 逻辑留下了清楚入口

第 15 篇还不展开 forkSubagent 的特殊性，但要先把坡度立住：

- 既然主 agent 会派生 subagent
- 那后面就要继续问：这种派生到底是不是“再开一个 agent”？
- 它继承了什么，受了什么约束，为什么更像 worker 分叉？

这就是第 16 篇要接住的问题。
