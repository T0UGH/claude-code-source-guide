## 这篇要回答的问题

进入 Agent 主轴之后，最该先立住的，不是哪一种 agent 最强，也不是 subagent 有多少花样。

锚点问题更靠前：

> **Claude Code 为什么会从一个单执行者系统，长成一套能够持续派生更多执行者的体系？**

如果这里没有先立住，后面的 runAgent、forkSubagent、worker、信息回流都会被写成零散技巧。

但卷五要回答的不是技巧，而是扩展层怎样成立。

所以第 13 篇必须先把 Agent 主轴立成卷五第二条重轴：

> **执行者扩展重轴。**

## 先给结论

### 结论一：agent 的出现，意味着系统开始把“做这段工作的人”也外化成运行时对象

在只有 tools 的时候，Claude Code 更像一个单执行者拿着不同动作原语做事。

到了 agent 这一层，系统开始处理另一类问题：

- 这段工作该由谁来接
- 这个接活者带什么上下文、权限和能力面
- 是否还要再派生出新的执行者去分段处理

也就是说，Claude Code 不只是长了更多能力，还开始长出：

> **更多执行者。**

### 结论二：subagent 不是另一组平级对象，而是这条执行者主线继续往后长出来的后半段

卷五这里最容易犯的错，就是把 agent 和 subagent 再拆成两组：

- 前面讲 agent
- 后面再平行讲 subagent

这不稳。

更稳的理解是：

- **agent** 先立“系统开始有更多执行者”
- **subagent / worker / fork** 再解释“这些执行者怎样继续分叉和协作”

所以它们不是两组平级对象，而是同一条执行者主线的前半段 / 后半段。

### 结论三：Claude Code 真正变强的，不只是会调用能力，而是会组织更多承担工作的执行者

skills 让系统学会用户的方法，MCP 让系统接住系统外能力。

Agent 主轴解决的则是另一层平台能力：

> **系统怎样把执行责任进一步分层、分工、分叉，并继续组织回统一工作流。**

这就是它为什么会成为卷五双核心之一。

## 先把角色说清：这篇讲什么，不讲什么

这篇是 Agent 主轴的 **锚点篇**。

它要完成三件事：

1. 立住“更多执行者”这条主线
2. 说明 agent 与 subagent 是同一条主线的前后段
3. 给 runAgent 与 worker / fork 留出清楚坡度

这篇**不**展开：

- 不细讲 runAgent 的 runtime 装配细节
- 不细拆 worker / fork 的继承和约束
- 不提前讲 hooks / plugins
- 不写成多 agent 案例合集

## 为什么 Claude Code 会从单执行者走向更多执行者

### 第一，因为复杂任务不是“能力不够”这么简单，常常还是“单执行线不够”

很多复杂任务失败，不是因为系统手里没有能力，而是因为所有事都挤在同一条主线里：

- 要同时看材料、规划步骤、改文件、核对边界
- 要在多个局部之间来回切换
- 要处理主任务，又要暂存旁支问题

这时候，即便工具很多，也不代表执行结构已经够了。

Claude Code 继续长 agent，本质上是在回答：

> **当一条主执行线开始不够时，系统怎么继续长出新的承担者。**

### 第二，因为“会做事”不等于“会分工做事”

tools 解决的是动作能力。
skills 解决的是工作方法。

但复杂任务再往上走，就会碰到一个更结构性的问题：

- 哪些事由当前主执行者继续扛
- 哪些事应该交给独立执行者隔离处理
- 哪些结果完成后再回到主线整合

这一层不是工具组合能自动完成的。

它需要系统把“执行者”也做成正式对象。

### 第三，因为持续工作一旦成立，下一步自然就是持续分工

卷四已经把“持续工作”立住了：Claude Code 不是一轮答完就散，而是能维持工作状态、继续推进。

在这个前提上，卷五里的 agent 主轴就显得顺理成章。

因为系统一旦能持续工作，下一步自然会问：

- 能不能把不同子任务拆出去
- 能不能让不同执行者各守自己的上下文
- 能不能让分工不是临时口头约定，而是 runtime 里的正式结构

所以 agent 主轴不是平地冒出来的，而是持续工作能力往前推进后的必然结果。

## “更多执行者”到底意味着什么

### 1. 不只是多开几个名字，而是多出一层执行责任结构

如果只是多一个标签、多一段 prompt，那还谈不上执行者扩展。

Claude Code 这里更关键的变化是：

- 某段工作可以正式交给另一个承担者
- 这个承担者有自己的运行边界
- 它不是主线程里临时附带的一句话
- 它完成后还能把结果带回主线

所以“更多执行者”讲的不是数量，而是执行责任被结构化了。

### 2. 不只是并行技巧，而是运行时对象谱系继续往上长

卷三已经说明过：执行层里不只有本地 tool，还有 SkillTool / AgentTool 这种高阶执行对象。

卷五在此基础上继续往前推：

- 既然 agent 已经是正式执行对象
- 那么系统就不只是“调用动作”
- 而是在调用“接手一段工作”的结构能力

也就是说，Claude Code 的执行层已经不只长动作原语，还长执行者谱系。

### 3. 不只是任务拆分，而是多执行者协作的开始

subagent / worker 的意义，不在“拆得更细”本身，而在：

- 主执行者不再什么都自己扛
- 某些工作可以在受控边界里独立推进
- 最终结果还能重新收回主线

所以 Agent 主轴真正打开的，是多执行者协作结构，而不是一般意义上的并行执行小技巧。

## 为什么说这是卷五的“执行者扩展重轴”

### 因为它回答的是“谁来做”这一层的扩展

卷五几组各自解决不同扩展问题：

- skills：方法怎样进入系统
- MCP：系统外能力怎样进入系统
- Agent 主轴：更多执行者怎样进入系统
- hooks：运行时接缝怎样进入系统
- plugins：扩展怎样被统一封装和治理

所以 Agent 主轴不是“又一种能力对象”，而是专门解决：

> **执行责任怎样被进一步外化和扩展。**

### 因为它把 Claude Code 从“能调能力”推进到“能组织承担者”

只会调用能力的系统，仍然可能是单线程的。

能够组织承担者的系统，才开始真正有平台味道。

Claude Code 在 agent 主轴上最关键的增长，不是“多了个 agent 功能”，而是：

> **它开始能把一段段工作，正式交给不同执行者去承担。**
