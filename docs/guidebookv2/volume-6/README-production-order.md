# 卷六执行说明

> 本文件不是讨论稿，也不是灵感备忘。  
> **本文件是卷六执行协议。**  
> 卷六生产必须按本文件执行。

---

## 一、总则

### 1. 卷六必须按本执行说明推进

卷六必须按本文件规定的：

- 文章顺序
- 回合顺序
- 串并行边界
- 主会话职责
- 接单 subagent 职责
- 验收与回退规则

执行。

**不得临时改卷六主问题，不得把卷六重新写回 team 功能目录。**

如果卷六结构判断发生变化，必须先修改：

1. 卷六 spec
2. 卷六写作卡片
3. 本执行说明

三者一致后，才允许继续推进卷六。

### 2. 卷六当前唯一有效结构

卷六当前唯一有效结构为：

1. 为什么说 Claude Code 的多 agent 能力本质上是一层协作 runtime
2. Claude Code 的 team / teammate runtime 到底在系统里处在什么位置
3. team 是怎么被创建、注册和清理的
4. InProcessTeammateTask：真正的 teammate runtime 是怎么在同进程里跑起来的
5. mailbox + idle / shutdown 协议：teammate 之间是怎么通信和收尾的
6. LocalAgentTask、RemoteAgentTask 和 teammate runtime 的边界是什么
7. 为什么说 Claude Code 的 team 系统本质上是一个 swarm

**合计 7 篇。**

对象主线顺序固定为：

> **总起判断 → 系统位置 → 生命周期 → 运行体 → 协作协议 → 承载体边界 → swarm 收束**

### 3. 卷六的唯一主问题不得漂移

卷六唯一主问题是：

> **为什么说 Claude Code 的多 agent 能力本质上是一层协作 runtime。**

因此卷六写作期间：

- **不得把卷六降成 team feature 清单**
- **不得把卷六写成 API 说明书**
- **不得把卷六偷换成命令入口卷**
- **不得提前吃掉卷七的产品层整合职责**

### 4. 卷六的唯一收口不得改变

卷六最后必须收束为：

> **Claude Code 的 team 系统，不是几个 teammate feature 的堆叠，而是一套带 leader、mailbox 和 task runtime 的 swarm。**

如果第 7 篇没有完成这层判断，卷六就不算收住。

### 5. 本文件优先于临场发挥

卷六执行期间：

- 不允许“先跑一篇看看”
- 不允许“这里临时改成并行大铺”
- 不允许“这篇顺手把卷七内容也吃掉”

**临场发挥不能覆盖本文件。**

---

## 二、卷六当前执行前提

当前已经确认的卷级前提如下：

- 卷六正式定位：**多 agent 协作运行时**
- 总组织方式：**先总后分**
- 旧卷六主线保留，但前面必须补一篇更强的总起
- 卷六重点是：**协作层如何成立**
- 卷六不负责：slash commands、runtime 命令入口、skill frontmatter / runtime interface、产品层整合
- 上述内容统一后移到：**卷七：命令、工作流与产品层整合**

因此卷六的生产原则不是“把旧卷六文章重写一遍”，而是：

> **先立协作 runtime 判断，再把旧卷六那条 team / teammate / mailbox / swarm 主线重新组织成一条更硬的运行时主线。**

---

## 三、卷六唯一允许的运行策略

卷六**不得**采用：

- 7 篇一起全并行
- 按旧文档文件名逐篇散跑
- 从 lifecycle / mailbox 任一机制篇直接起跑
- 边写边临时改卷级主问题

卷六**必须**采用：

> **前半骨架严格串行，后半局部并行，卷尾强制串行收束。**

解释如下：

1. **前 4 篇承担卷六骨法，必须逐层钉住**
2. **第 5、6 篇可在前 4 篇稳定后局部并行**
3. **第 7 篇必须最后写，且必须在前文全部通过后启动**

为避免歧义，卷六统一分为三类文章：

### 1. 锚点篇
- 定卷级判断
- 定系统位置
- 定后文观察框架
- 未通过，不得启动后续关键篇

### 2. 展开篇
- 在锚点篇通过后启动
- 负责真正拆对象、运行体、协议、边界
- 允许有限并行，但不得串线

### 3. 收束篇
- 必须最后启动
- 必须负责把前文全部压回 swarm 判断
- 不得提前启动

## 四、分回合强制执行顺序

## 回合 1：先立卷六骨架

### 本回合包含文章
1. 为什么说 Claude Code 的多 agent 能力本质上是一层协作 runtime
2. Claude Code 的 team / teammate runtime 到底在系统里处在什么位置
3. team 是怎么被创建、注册和清理的
4. InProcessTeammateTask：真正的 teammate runtime 是怎么在同进程里跑起来的

### 强制顺序
**1 → 2 → 3 → 4**

### 执行规则
- **必须严格串行**
- **第 1 篇未通过，不得启动第 2 篇**
- **第 2 篇未通过，不得启动第 3 篇**
- **第 3 篇未通过，不得启动第 4 篇**

### 为什么必须这样跑
因为这 4 篇决定卷六是不是一卷真正成立的“协作运行时卷”：

- 第 1 篇负责把“多 agent ≠ 多开几个 agent”立成卷级判断
- 第 2 篇负责给出 team / teammate runtime 的系统位置
- 第 3 篇负责把 team 从概念推进成正式对象
- 第 4 篇负责把 teammate runtime 从对象层推进成真正运行体

如果这 4 篇并行，最容易出现：

- 01 和 02 互抢“总判断 / 系统位置”
- 02 和 03 互抢“team 为什么是正式对象”
- 03 和 04 互抢“生命周期 / 运行体落地”
- 04 过早把 mailbox / shutdown 细节吃掉

所以回合 1 必须按：

> **总判断 → 位置 → 生命周期 → 运行体**

逐层推进。

### 回合 1 验收目标
主会话必须确认：

- 卷六已经明确从“协作 runtime”立题，而不是从 team 功能立题
- team / teammate runtime 的系统位置已经稳定
- lifecycle 与 runtime body 没有互抢职责
- 第 4 篇已经把 teammate runtime 立成正式运行体，而不是 wrapper 描写

未满足以上条件：

> **不得进入回合 2。**

---

## 回合 2：拆协作协议与承载体边界

### 本回合包含文章
5. mailbox + idle / shutdown 协议：teammate 之间是怎么通信和收尾的
6. LocalAgentTask、RemoteAgentTask 和 teammate runtime 的边界是什么

### 强制顺序
**先验收 4，再启动 5 / 6。**

### 执行方式
**有限并行。**

允许：
- 05 与 06 分别由独立 subagent 起稿
- 但必须共享统一前提：前 4 篇已通过、卷六主问题已稳定

不允许：
- 04 未通过就启动 05 / 06
- 05 抢写承载体边界总论
- 06 抢写 mailbox / shutdown 协议总论

### 为什么这两篇可以局部并行
因为到这一阶段：

- team 已经被立成正式对象
- teammate runtime 已经被立成正式运行体
- 后面要拆的是两种不同维度：
  - 05 拆**协作协议**
  - 06 拆**承载体边界**

它们已经不是同一层问题。

### 回合 2 验收目标
主会话必须确认：

- 05 已经把 mailbox / idle / shutdown 立成协作闭环协议，而不是边角机制
- 06 已经把 Local / Remote / Teammate 切成三种不同承载体，而不是实现差异表
- 05 没有提前把 swarm 收束做完
- 06 没有提前把卷七的 command / interface 边界带进来

未满足以上条件：

> **不得进入回合 3。**

---

## 回合 3：卷尾强制收束

### 本回合包含文章
7. 为什么说 Claude Code 的 team 系统本质上是一个 swarm

### 强制顺序
**必须最后启动。**

### 执行规则
- **第 1-6 篇未通过，不得启动第 7 篇**
- **第 7 篇必须串行主写**
- 如需 review，可后置并行 review，但正文主写不得与第 5 / 6 篇并行抢跑

### 为什么必须最后跑
因为第 7 篇不是普通总结文，而是卷六总判断落点。

它必须把前文这些东西重新压成一张图：

- team 正式对象
- teammate runtime
- mailbox / idle / shutdown 协议
- Local / Remote / Teammate 的分层承载体
- leader / teammate 协作结构

如果没有前 1-6 篇稳定结果，第 7 篇就会退化成：

- 空泛的“多 agent 很强”总结
- 或提前混入卷七产品整合问题

### 回合 3 验收目标
主会话必须确认：

- 第 7 篇已经把卷六整体压回 swarm runtime 判断
- 没有重新逐篇复述前文
- 已自然导流卷七，但没有抢卷七正文

未满足以上条件：

> **卷六不得宣告完成。**

## 五、每篇文章的职责钉死

## 01｜总起判断篇
### 唯一职责
把卷六立成“协作 runtime 卷”。

### 禁止事项
- 不得提前讲完 lifecycle
- 不得提前讲完 mailbox
- 不得提前进入命令入口 / 产品层整合

## 02｜系统位置篇
### 唯一职责
解释 team / teammate runtime 在整套系统中的位置。

### 禁止事项
- 不得抢 01 的总判断
- 不得抢 03 的生命周期
- 不得把自己写成结构总览百科

## 03｜生命周期篇
### 唯一职责
解释 team 怎样被创建、注册、清理，为什么它因此成为正式对象。

### 禁止事项
- 不得把 teammate runtime 落地细节讲完
- 不得把 mailbox 协议提前吃掉

## 04｜运行体篇
### 唯一职责
解释 InProcessTeammateTask 怎样让 teammate runtime 真正在同进程里跑起来。

### 禁止事项
- 不得把协议篇改写进来
- 不得把边界篇改写进来

## 05｜协议篇
### 唯一职责
解释 mailbox / idle / shutdown 为什么是协作闭环成立的关键协议。

### 禁止事项
- 不得把 06 的承载体边界一并写掉
- 不得提前把 07 的 swarm 判断写完

## 06｜边界篇
### 唯一职责
解释 LocalAgentTask、RemoteAgentTask 与 teammate runtime 的角色边界。

### 禁止事项
- 不得改写成任务类型速查表
- 不得把卷七 command / interface 边界提前带进来

## 07｜收束篇
### 唯一职责
把卷六全部对象、运行体、协议重新压成 swarm runtime 判断。

### 禁止事项
- 不得重新逐篇讲解前文
- 不得转成卷七产品总论

## 六、主会话职责

主会话必须负责：

1. 锁定卷六主问题不漂移
2. 锁定 7 篇顺序不漂移
3. 每回合结束做职责验收
4. 拒绝任何提前吃卷七的写法
5. 拒绝任何把卷六降回 feature list 的写法
6. 在第 7 篇前，先确认前文对象、运行体、协议、边界都已成立

主会话**不得**：

- 把卷六重新拆成“team 组 / teammate 组 / protocol 组”三个平级大组乱跑
- 在没有更新 spec / 卡片 / playbook 的情况下临时改结构
- 用“差不多能看”替代正式验收

---

## 七、接单 subagent 职责

接单 subagent 必须遵守：

1. 只写当前派单文章，不擅自扩写别篇职责
2. 先服从卷六主问题，再服从单篇题目
3. 不得把旧卷六旧文结构原样搬运成新版正文
4. 不得自行引入卷七主线
5. 写完后必须说明：
   - 本篇主问题是否完成
   - 是否有越界风险
   - 对下一篇的导流句是否已留出

如果发现当前派单文章职责本身含糊，subagent 不得私改结构，而应回主会话请求重新校边界。

---

## 八、统一验收清单

每篇文章完成后，默认按以下清单验收：

### 1. 主问题是否单一
- 这篇到底只回答一个问题了吗？
- 有没有把相邻篇的工作抢进来？

### 2. 卷内位置是否清楚
- 这篇是总起、定位、生命周期、运行体、协议、边界，还是收束？
- 有没有角色漂移？

### 3. 读者残留是否成立
- 读者读完后，脑中会留下这一篇该留下的判断吗？
- 还是只记得几个机制名词？

### 4. 是否守住卷六边界
- 有没有提前带入 slash commands / command layer / 产品入口？
- 有没有把卷七问题偷吃进来？

### 5. 是否自然导向下一篇
- 文末有没有留下下一篇自然接手的位置？
- 还是这一篇自己封死了？

---

## 九、卷六完成标准

只有同时满足下面 5 条，卷六才算完成：

1. 7 篇全部完成并通过验收
2. 卷六已明确立成“协作 runtime 卷”
3. team / teammate / mailbox / 边界四条线已经被统一压回一张图
4. 卷尾已经稳定收成 swarm runtime 判断
5. 卷七入口已经自然露出，但没有被提前展开

如果只完成了机制讲解，而没有完成上述第 2、3、4 条：

> **卷六一律视为未完成。**

---

## 十、默认启动口令

当后续主会话收到类似指令时：

- 跑卷六
- 跑卷六 playbook
- 按卷六执行说明推进
- 开始卷六生产

默认含义一律解释为：

> **按本文件规定的回合顺序推进卷六，不得跳回合，不得改主问题，不得提前吃卷七。**
