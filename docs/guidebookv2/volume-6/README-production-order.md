# 卷六执行说明

> 本文件不是讨论稿，也不是灵感备忘。  
> **本文件是卷六执行协议。**  
> 卷六生产必须按本文件执行。

---

## 一、总则

### 1. 卷六必须按本执行说明推进

卷六必须按本文件规定的：

- 文章顺序
- 回合顺序
- 串并行边界
- 主会话职责
- 接单 subagent 职责
- 验收与回退规则

执行。

**不得临时改卷六主问题，不得把卷六重新写回 team 功能目录。**

如果卷六结构判断发生变化，必须先修改：

1. 卷六 spec
2. 卷六写作卡片
3. 本执行说明

三者一致后，才允许继续推进卷六。

### 2. 卷六当前唯一有效结构

卷六当前唯一有效结构为：

1. 为什么说 Claude Code 的多 agent 能力本质上是一层协作 runtime
2. Claude Code 的 team / teammate runtime 到底在系统里处在什么位置
3. team 是怎么被创建、注册和清理的
4. InProcessTeammateTask：真正的 teammate runtime 是怎么在同进程里跑起来的
5. mailbox + idle / shutdown 协议：teammate 之间是怎么通信和收尾的
6. LocalAgentTask、RemoteAgentTask 和 teammate runtime 的边界是什么
7. 为什么说 Claude Code 的 team 系统本质上是一个 swarm

**合计 7 篇。**

对象主线顺序固定为：

> **总起判断 → 系统位置 → 生命周期 → 运行体 → 协作协议 → 承载体边界 → swarm 收束**

### 3. 卷六的唯一主问题不得漂移

卷六唯一主问题是：

> **为什么说 Claude Code 的多 agent 能力本质上是一层协作 runtime。**

因此卷六写作期间：

- **不得把卷六降成 team feature 清单**
- **不得把卷六写成 API 说明书**
- **不得把卷六偷换成命令入口卷**
- **不得提前吃掉卷七的产品层整合职责**

### 4. 卷六的唯一收口不得改变

卷六最后必须收束为：

> **Claude Code 的 team 系统，不是几个 teammate feature 的堆叠，而是一套带 leader、mailbox 和 task runtime 的 swarm。**

如果第 7 篇没有完成这层判断，卷六就不算收住。

### 5. 本文件优先于临场发挥

卷六执行期间：

- 不允许“先跑一篇看看”
- 不允许“这里临时改成并行大铺”
- 不允许“这篇顺手把卷七内容也吃掉”

**临场发挥不能覆盖本文件。**

## 二、卷六当前执行前提

当前已经确认的卷级前提如下：

- 卷六正式定位：**多 agent 协作运行时**
- 总组织方式：**先总后分**
- 旧卷六主线保留，但前面必须补一篇更强的总起
- 卷六重点是：**协作层如何成立**
- 卷六不负责：slash commands、runtime 命令入口、skill frontmatter / runtime interface、产品层整合
- 上述内容统一后移到：**卷七：命令、工作流与产品层整合**

因此卷六的生产原则不是“把旧卷六文章重写一遍”，而是：

> **先立协作 runtime 判断，再把旧卷六那条 team / teammate / mailbox / swarm 主线重新组织成一条更硬的运行时主线。**

## 三、卷六唯一允许的运行策略

### 0. 卷六必须存在一个总控 subagent

从卷六开始执行起：

- **必须存在一个卷六总控 subagent**
- 最外层主会话**不直接充当卷六执行主会话**
- 回合推进、回合验收、是否放行下一回合，**必须**由卷六总控 subagent 负责
- 单篇执行 subagent 的派发、回收、初步验收，也**必须**由卷六总控 subagent 统一调度

这意味着卷六执行默认分三层：

1. **最外层主会话**：只负责批准、git、最终验收、结构变更
2. **卷六总控 subagent**：负责回合调度、派单、回收、放行建议
3. **接单执行 subagent**：只负责当前文章，不负责改卷级结构

### 0.1 卷六总控 subagent 不得操作 git

卷六总控 subagent：

- **不得** `git add`
- **不得** `git commit`
- **不得** `git push`
- **不得**绕过最外层主会话直接宣布卷六完成

所有 git 操作统一规定为：

> **只由最外层主会话负责。**

### 0.2 回合 1-3 的派单方式是强制规则

卷六执行期间：

- **不得**回到“想到哪篇就派哪篇”的散派模式
- **不得**让执行 subagent 自行决定下一篇该写什么
- **不得**在未更新 spec / 卡片 / playbook 时临时改顺序

卷六**必须**采用：

> **前半骨架严格串行，后半局部并行，卷尾强制串行收束。**

解释如下：

1. **前 4 篇承担卷六骨法，必须逐层钉住**
2. **第 5、6 篇只允许在前 4 篇通过后局部并行**
3. **第 7 篇必须最后写，且只能在前 1-6 篇全部通过后启动**

卷六**不得**采用：

- 7 篇一起全并行
- 按旧文档文件名逐篇散跑
- 从 lifecycle / mailbox 任一机制篇直接起跑
- 边写边临时改卷级主问题
- 第 5、6 篇未稳就提前抢跑第 7 篇

为避免歧义，卷六统一分为七种文章角色：

### 1. 总起锚点篇
- 负责立卷级主判断
- 未通过，不得进入卷六主线正文

### 2. 系统定位篇
- 负责给出 team / teammate runtime 的系统位置
- 未通过，不得进入对象与运行体展开

### 3. 对象成立篇
- 负责把 team 推进成正式对象
- 未通过，不得进入 runtime body 展开

### 4. 运行体落地篇
- 负责把 teammate runtime 立成正式运行体
- 未通过，不得进入协议 / 边界并行回合

### 5. 协议展开篇
- 负责拆协作协议
- 只能在运行体落地篇通过后启动

### 6. 边界收口篇
- 负责切清 Local / Remote / Teammate 边界
- 只能在运行体落地篇通过后启动

### 7. 卷尾总收束篇
- 负责把前文全部压回 swarm runtime 判断
- 必须最后启动

## 四、分回合强制执行顺序

## 回合 0：建立卷六总控层

### 强制顺序
**先建总控，再进正文回合。**

### 执行规则
- **未建立卷六总控 subagent，不得启动回合 1-3 的任何正文派单**
- 最外层主会话必须先向总控 subagent 提供：
  - 卷六 spec
  - 卷六写作卡片
  - 本执行说明
  - 当前卷六状态
- 卷六总控 subagent 必须先确认：
  - 7 篇结构有效
  - 当前允许启动的回合
  - 当前禁止启动的文章

### 验收目标
最外层主会话必须确认：

- 卷六总控 subagent 已建立
- 总控层知道自己不能动 git
- 总控层知道自己不能私改结构
- 后续执行 subagent 派单将统一从总控层流出

未满足以上条件：

> **不得进入回合 1。**


## 回合 1：先立卷六骨架

### 本回合包含文章
1. 为什么说 Claude Code 的多 agent 能力本质上是一层协作 runtime
2. Claude Code 的 team / teammate runtime 到底在系统里处在什么位置
3. team 是怎么被创建、注册和清理的
4. InProcessTeammateTask：真正的 teammate runtime 是怎么在同进程里跑起来的

### 强制顺序
**1 → 2 → 3 → 4**

### 执行规则
- **必须严格串行**
- **第 1 篇未通过，不得启动第 2 篇**
- **第 2 篇未通过，不得启动第 3 篇**
- **第 3 篇未通过，不得启动第 4 篇**

### 为什么必须这样跑
因为这 4 篇决定卷六是不是一卷真正成立的“协作运行时卷”：

- 第 1 篇负责把“多 agent ≠ 多开几个 agent”立成卷级判断
- 第 2 篇负责给出 team / teammate runtime 的系统位置
- 第 3 篇负责把 team 从概念推进成正式对象
- 第 4 篇负责把 teammate runtime 从对象层推进成真正运行体

如果这 4 篇并行，最容易出现：

- 01 和 02 互抢“总判断 / 系统位置”
- 02 和 03 互抢“team 为什么是正式对象”
- 03 和 04 互抢“生命周期 / 运行体落地”
- 04 过早把 mailbox / shutdown 细节吃掉

所以回合 1 必须按：

> **总判断 → 位置 → 生命周期 → 运行体**

逐层推进。

### 回合 1 验收目标
主会话必须确认：

- 卷六已经明确从“协作 runtime”立题，而不是从 team 功能立题
- team / teammate runtime 的系统位置已经稳定
- lifecycle 与 runtime body 没有互抢职责
- 第 4 篇已经把 teammate runtime 立成正式运行体，而不是 wrapper 描写

未满足以上条件：

> **不得进入回合 2。**

---

## 回合 2：拆协作协议与承载体边界

### 本回合包含文章
5. mailbox + idle / shutdown 协议：teammate 之间是怎么通信和收尾的
6. LocalAgentTask、RemoteAgentTask 和 teammate runtime 的边界是什么

### 强制顺序
**先验收 4，再启动 5 / 6。**

### 执行方式
**有限并行。**

允许：
- 05 与 06 分别由独立 subagent 起稿
- 但必须共享统一前提：前 4 篇已通过、卷六主问题已稳定

不允许：
- 04 未通过就启动 05 / 06
- 05 抢写承载体边界总论
- 06 抢写 mailbox / shutdown 协议总论

### 为什么这两篇可以局部并行
因为到这一阶段：

- team 已经被立成正式对象
- teammate runtime 已经被立成正式运行体
- 后面要拆的是两种不同维度：
  - 05 拆**协作协议**
  - 06 拆**承载体边界**

它们已经不是同一层问题。

### 回合 2 验收目标
主会话必须确认：

- 05 已经把 mailbox / idle / shutdown 立成协作闭环协议，而不是边角机制
- 06 已经把 Local / Remote / Teammate 切成三种不同承载体，而不是实现差异表
- 05 没有提前把 swarm 收束做完
- 06 没有提前把卷七的 command / interface 边界带进来

未满足以上条件：

> **不得进入回合 3。**

---

## 回合 3：卷尾强制收束

### 本回合包含文章
7. 为什么说 Claude Code 的 team 系统本质上是一个 swarm

### 强制顺序
**必须最后启动。**

### 执行规则
- **第 1-6 篇未通过，不得启动第 7 篇**
- **第 7 篇必须串行主写**
- 如需 review，可后置并行 review，但正文主写不得与第 5 / 6 篇并行抢跑

### 为什么必须最后跑
因为第 7 篇不是普通总结文，而是卷六总判断落点。

它必须把前文这些东西重新压成一张图：

- team 正式对象
- teammate runtime
- mailbox / idle / shutdown 协议
- Local / Remote / Teammate 的分层承载体
- leader / teammate 协作结构

如果没有前 1-6 篇稳定结果，第 7 篇就会退化成：

- 空泛的“多 agent 很强”总结
- 或提前混入卷七产品整合问题

### 回合 3 验收目标
主会话必须确认：

- 第 7 篇已经把卷六整体压回 swarm runtime 判断
- 没有重新逐篇复述前文
- 已自然导流卷七，但没有抢卷七正文

未满足以上条件：

> **卷六不得宣告完成。**

## 五、每篇文章的职责钉死

## 01｜总起锚点篇
### 唯一职责
把卷六立成“协作 runtime 卷”。

### 禁止事项
- 不得提前讲完 lifecycle
- 不得提前讲完 mailbox
- 不得提前进入命令入口 / 产品层整合

## 02｜系统定位篇
### 唯一职责
解释 team / teammate runtime 在整套系统中的位置。

### 禁止事项
- 不得抢 01 的总判断
- 不得抢 03 的生命周期
- 不得把自己写成结构总览百科

## 03｜对象成立篇
### 唯一职责
解释 team 怎样被创建、注册、清理，为什么它因此成为正式对象。

### 禁止事项
- 不得把 teammate runtime 落地细节讲完
- 不得把 mailbox 协议提前吃掉

## 04｜运行体落地篇
### 唯一职责
解释 InProcessTeammateTask 怎样让 teammate runtime 真正在同进程里跑起来。

### 禁止事项
- 不得把协议篇改写进来
- 不得把边界篇改写进来

## 05｜协议展开篇
### 唯一职责
解释 mailbox / idle / shutdown 为什么是协作闭环成立的关键协议。

### 禁止事项
- 不得把 06 的承载体边界一并写掉
- 不得提前把 07 的 swarm 判断写完

## 06｜边界收口篇
### 唯一职责
解释 LocalAgentTask、RemoteAgentTask 与 teammate runtime 的角色边界。

### 禁止事项
- 不得改写成任务类型速查表
- 不得把卷七 command / interface 边界提前带进来

## 07｜卷尾总收束篇
### 唯一职责
把卷六全部对象、运行体、协议重新压成 swarm runtime 判断。

### 禁止事项
- 不得重新逐篇讲解前文
- 不得转成卷七产品总论

## 六、卷六总控 subagent 与最外层主会话职责

### 0. 卷六必须由总控 subagent 接管推进

从回合 1 开始：

- 最外层主会话**不直接逐篇散派**
- 卷六回合推进、回合验收、是否进入下一回合，**必须**交由卷六总控 subagent 负责
- 回合 1-3 的正文执行 subagent，**必须**由卷六总控 subagent 统一调度

### 0.1 卷六总控 subagent 不得操作 git

卷六总控 subagent：

- **不得操作 git**
- **不得** `git add` / `git commit` / `git push`
- **只负责调度、回收、验收建议、放行建议**

所有 git 操作：

> **只由最外层主会话负责。**

### 1. 最外层主会话必须负责

1. 建立卷六总控 subagent
2. 向总控层提供 spec / 写作卡片 / 执行说明
3. 批准当前回合是否启动
4. 对回合级结果做最终验收
5. 负责全部 git 操作
6. 在结构变更时，先改 spec / 卡片 / playbook，再继续执行

最外层主会话**不得**：

- 跳过总控层直接逐篇散派
- 在没有更新 spec / 卡片 / playbook 的情况下临时改结构
- 把 git 操作下放给总控 subagent 或执行 subagent
- 用“差不多能看”替代正式验收

### 2. 卷六总控 subagent 必须负责

1. 识别当前所处回合
2. 锁定当前允许启动的文章
3. 禁止任何抢跑文章启动
4. 为接单执行 subagent 提供统一上下文
5. 回收草稿并做第一轮职责验收
6. 向最外层主会话提交是否放行下一回合的建议

卷六总控 subagent **不得**：

- 私改 7 篇结构
- 绕过最外层主会话直接宣布回合通过
- 让执行 subagent 自行决定下一篇顺序
- 把第 5 / 6 篇之前的文章并行化
- 让第 7 篇提前启动

## 七、接单执行 subagent 硬规则

### 1. 接收单篇文章任务后的第一条硬规则

只要是 **Claude Code 系列单篇文章任务**，接单执行 subagent：

> **第一步必须优先使用 `claude-code-article-orchestrator`。**

这是硬要求，不是建议。

### 2. 接单执行 subagent 必须遵守

1. 只写当前派单文章，不擅自扩写别篇职责
2. 先服从卷六主问题，再服从单篇题目
3. 不得把旧卷六旧文结构原样搬运成新版正文
4. 不得自行引入卷七主线
5. 不得自行改顺序、改回合、改角色
6. 写完后必须说明：
   - 本篇主问题是否完成
   - 是否有越界风险
   - 对下一篇的导流句是否已留出

### 3. 接单执行 subagent 禁止事项

- **不得**绕过 `claude-code-article-orchestrator` 直接按旧习惯裸写
- **不得**把 lifecycle、runtime、protocol、boundary 写成一篇混合文
- **不得**提前引入卷七的命令入口、产品控制层、最终产品整合问题
- **不得**自行宣布“卷六已完成”
- **不得**操作 git

如果发现当前派单文章职责本身含糊，执行 subagent 不得私改结构，而应回总控 subagent 请求重新校边界。

## 八、统一验收清单

每篇文章完成后，默认按以下清单验收：

### 1. 主问题是否单一
- 这篇到底只回答一个问题了吗？
- 有没有把相邻篇的工作抢进来？

### 2. 卷内位置是否清楚
- 这篇是总起、定位、生命周期、运行体、协议、边界，还是收束？
- 有没有角色漂移？

### 3. 读者残留是否成立
- 读者读完后，脑中会留下这一篇该留下的判断吗？
- 还是只记得几个机制名词？

### 4. 是否守住卷六边界
- 有没有提前带入 slash commands / command layer / 产品入口？
- 有没有把卷七问题偷吃进来？

### 5. 是否自然导向下一篇
- 文末有没有留下下一篇自然接手的位置？
- 还是这一篇自己封死了？

## 九、卷六完成标准

只有同时满足下面 5 条，卷六才算完成：

1. 7 篇全部完成并通过验收
2. 卷六已明确立成“协作 runtime 卷”
3. team / teammate / mailbox / 边界四条线已经被统一压回一张图
4. 卷尾已经稳定收成 swarm runtime 判断
5. 卷七入口已经自然露出，但没有被提前展开

如果只完成了机制讲解，而没有完成上述第 2、3、4 条：

> **卷六一律视为未完成。**

## 十、默认启动口令

当后续主会话收到类似指令时：

## 十一、卷六执行禁令总表
