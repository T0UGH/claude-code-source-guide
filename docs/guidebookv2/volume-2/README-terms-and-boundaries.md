# 卷二术语与边界小词表

> 用途：给卷二当前并发写作使用。目标不是做术语百科，而是先把主线语言和篇际边界钉硬，减少 01 / 03 / 05 / 07 之间的串线。

---

## 一、卷二总问题

**用户输入怎么变成一次完整的 agent turn？**

卷二默认按**时间顺序**展开，不按组件百科展开。读者最终应该留下的，不是一堆局部机制，而是一张**主循环稳定运行图**。

---

## 二、核心术语（当前统一口径）

### 1. request（请求）
用户当前这一次进入系统的输入单元。

在卷二里，request 不是“裸文本”同义词，而是**将要被接入当前运行时的一次请求**。

### 2. 主循环
Claude Code 把一条请求接入运行时、形成当前判断、必要时触发能力、把结果接回当前 turn，并最终决定继续还是收口的那条动态主线。

### 3. current query / 当前工作面
当前这一轮真正交给系统处理的工作面。

它不是用户一句话本身，而是带着：
- 当前 messages
- system prompt / user context / system context
- 当前输入整理结果

一起形成的本轮工作面。

### 4. 当前判断 / 当前决策
系统在当前工作面上形成的“这一步下一步该怎么走”的判断结果。

在卷二里，这个词默认优先用：
- **当前判断**：泛指这一轮形成的判断
- **当前决策**：强调它已经足够明确，可以把主循环推进到下一段

不要把它写成：
- 模型心里怎么想
- 模型抽象意图
- 模型好像想做事

### 5. `tool_use`
assistant 输出里出现的、runtime 可以接管的结构化动作表达。

卷二里默认把它视为：
- **当前决策已经成形**
- 主循环需要从回答路径切到执行路径的明确信号

### 6. 结果回流
能力执行结束后，结果重新回到当前 turn，成为下一轮判断输入。

它不是“工具执行完成”这么简单，而是主循环重新拿回现实结果、继续推进这一轮。

### 7. continue / 继续
当前工作还没有收口，这一轮必须继续推进。

### 8. 收口
当前工作已经推进到可以在本轮停下的状态。

卷二里“收口”优先于“结束”“停止”这种更泛的词。因为它强调的是：
- 不是界面停了
- 不是模型停了
- 而是当前工作闭合了

### 9. turn 边界
这一轮 agent turn 到底在什么条件下算完成。

卷二默认把它理解为：
- 先看这一轮有没有继续动作（例如 `tool_use`）
- 再看运行时是否允许真正收口

## 三、四篇文章的边界分工

### 01｜一次请求怎么进入 Claude Code 的主循环
**职责：总图。**

只负责把卷二的动态主线先立起来：
- 请求如何进入运行时
- 当前判断如何形成
- 为什么会触发能力
- 结果如何回流
- 这一轮怎样继续或收口

**不要展开：**
- `submitMessage(...)` 与 `query(...)` 的精细层次
- `tool_use` 如何被识别
- continue / 收口 的细判定

它可以点到这些对象，但不拥有这些解释权。

### 03｜请求是怎么进入 QueryEngine 的
**职责：正式入口。**

只负责讲清：
- 请求如何被正式接入 QueryEngine
- 为什么真正入口是 `submitMessage(...)`
- 为什么 QueryEngine 是 turn 级运行入口，而不是文本入口

**不要重新承担：**
- 卷二总图职责（那是 01）
- 回答 vs 行动 的分流职责（那是 05）
- continue / 收口 的边界职责（那是 07）

### 05｜系统怎么决定这一轮要不要调用能力
**职责：行动决策。**

只负责讲清：
- 当前工作面上，assistant 产出的是回答，还是行动决策
- `tool_use` 为什么意味着主循环要切到执行路径
- 为什么 `tool_use` 代表 runtime 可以接管的结构化决策

**不要展开：**
- 结果回来之后为什么继续、为什么停
- stop hooks / recovery / budget 的收口逻辑

那些属于 07。

### 07｜一轮 Agent Turn 什么时候继续，什么时候收口
**职责：收口判定。**

只负责讲清：
- 为什么“没有 `tool_use`”还不等于立刻结束
- 运行时如何做第二层 continue / 收口 判定
- turn 边界为什么是运行时边界，而不只是模型停点

**不要重新解释：**
- `tool_use` 为什么出现、如何把回答切到行动路径

那是 05 的职责。

## 四、当前最容易串线的地方

### 01 ↔ 03
- 01 可以点到 QueryEngine，但不要细讲 `submitMessage(...)` / `query(...)`
- 03 可以承接 01 的总图，但不要重新把总图讲一遍

### 05 ↔ 07
- 05 只讲：为什么切到执行路径
- 07 只讲：什么时候真正允许收口

一句硬分界：

> **05 负责“为什么继续去做事”，07 负责“什么时候这轮真的能停”。**

---

## 五、卷二写作时的默认表达偏好

优先用：
- 更准确地说
- 这里真正关键的是
- 这篇要解决的是
- 这一层的作用是
- 到这里，问题会变成

尽量少用：
- 我这次
- 我看完
- 下一步自然就是
- 这篇重点看
- 我觉得这句话就是主结论

---

## 六、一句话总收束

> 卷二不是在解释几个组件，而是在解释：**一条 request 怎样被接成 current turn，形成当前判断，必要时切到执行路径，再经过结果回流与收口判定，最终成为一次完整的 agent turn。**
