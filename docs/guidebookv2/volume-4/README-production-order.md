# 卷四生产顺序说明

> 用途：记录卷四当前确认的生产顺序、分回合策略、串行 / 并行边界，避免后续起稿时重新发散。

---

## 一、卷四当前总前提

卷四已确定为：

> **卷四：上下文与状态怎么维持系统持续工作**

主问题是：

> **Claude Code 为什么不是一轮跑完就散，而是能把工作继续维持下去？**

当前已经确认的执行约束：

- 读者残留目标是**持续工作总图优先**，治理链是支撑，不是终点本身
- 采用**先总后分**结构
- 旧文只能作为素材仓，不能反向决定新文骨架
- 卷四不是上下文机制目录，也不是 compact 功能列表
- 前半必须先立住：持续工作问题、对象边界、当前工作面构造
- 中段必须先立住：为什么不能原样无限送模、治理链总图
- 后半才拆：projection / collapse、compact / compaction、restore / recovery
- 卷尾必须回到“为什么这套系统还能继续活着”，而不是停在机制说明

---

## 二、卷四 8 篇当前顺序

1. 为什么 Claude Code 不是“一轮跑完就重来”的系统
2. messages / context / system prompt / transcript / session 为什么不是一回事
3. 当前可工作的上下文是怎么被拼起来的
4. 为什么系统不能把全部历史原样一直送模
5. collapse / compaction / projection / restore 的总体关系图
6. projection / collapse：系统治理的不是 transcript 本身，而是当前可工作的视图
7. compact / compaction：主动减负机制本体
8. restore / session recovery：系统怎么把这条线重新接活

---

## 三、推荐生产顺序：3 个回合

卷四不建议像卷二那样一开始就做大规模并发实验。

最稳的方案是：

> **3 个回合完成卷四。**

原因：

- 卷四前半不是工具家族，而是高耦合的连续认知坡度
- 前 5 篇承担卷级定义、对象边界、工作面构造、治理起点、治理链总图
- 这些骨架没稳之前，后半机制篇很容易互相串线
- 真正相对适合并行的，只有后半已经被校准过边界的机制篇

---

## 回合 1：先立持续工作骨架

### 建议方式
**串行**，不并行。

### 本回合包含文章
1. 为什么 Claude Code 不是“一轮跑完就重来”的系统
2. messages / context / system prompt / transcript / session 为什么不是一回事
3. 当前可工作的上下文是怎么被拼起来的

### 为什么必须串行
这 3 篇决定卷四到底成不成立：

- 第 1 篇决定卷四是不是“持续工作卷”
- 第 2 篇决定卷四对象边界会不会全糊
- 第 3 篇决定“当前可工作上下文”能不能被正式立住

如果这 3 篇并行，最容易出现：

- 01 提前偷讲对象边界与治理机制
- 02 把对象边界写成术语百科
- 03 重新滑回 prompt / context 拼装流水账
- 02 和 03 抢“当前工作面”这个核心概念

所以这组应该：

> **串行跑，边写边校准边界。**

### 本回合结束后要检查什么
- 卷四是否已经明确从“持续工作问题”开篇，而不是从机制名词开篇
- 对象边界是否已经清楚：messages / context / system prompt / transcript / session 没有重新混桶
- “当前可工作上下文”是否已经成为后文可复用的正式观察框架
- 第 1、2、3 篇之间是否没有互抢职责

---

## 回合 2：立治理必要性与治理链总图

### 建议方式
**串行**，不并行。

### 本回合包含文章
4. 为什么系统不能把全部历史原样一直送模
5. collapse / compaction / projection / restore 的总体关系图

### 为什么这两篇也建议串行
这两篇是卷四后半的总开关：

- 第 4 篇决定“为什么必须治理”
- 第 5 篇决定“治理链到底怎么分工”

如果这两篇没稳，后面的 06 / 07 / 08 很容易发生：

- 06 把自己写成 compact 预备篇
- 07 把自己写成所有机制总论
- 08 只剩 session 恢复说明，而失去卷尾收束作用

所以这组更稳的方式是：

> **先写 04，再写 05，把后半三篇的边界一次性校准好。**

### 本回合结束后要检查什么
- 第 4 篇有没有把“治理必要性”立成卷四后半起点，而不是 token 技巧文
- 第 5 篇有没有把 collapse / compaction / projection / restore 压成一条链，而不是几块并列功能
- 第 4 与第 5 篇之间是否没有互抢“为什么治理”和“治理链地图”两种职责
- 后半 06 / 07 / 08 的边界是否已经可稳定派发

---

## 回合 3：拆后半机制，并完成卷尾收束

### 建议方式
**先串行，再半并行。**

### 本回合包含文章
6. projection / collapse：系统治理的不是 transcript 本身，而是当前可工作的视图
7. compact / compaction：主动减负机制本体
8. restore / session recovery：系统怎么把这条线重新接活

### 推荐拆法
不要 06 / 07 / 08 一起全并。

建议拆成：

#### 第一步
- 串行写 06｜projection / collapse

#### 第二步
- 再并行 07｜compact / compaction
- 与 08｜restore / session recovery

### 为什么这样拆
06 是后半卷的边界校正篇。

它先要立住一个最关键的判断：

> **系统治理的不是 transcript 本身，而是当前可工作的视图。**

如果 06 还没定，07 和 08 很容易跑偏：

- 07 会把“减负”讲成“删历史”
- 08 会把“恢复”讲成“重新读档”

所以更准确地说，回合 3 是：

> **先串行校正治理对象，再对后两篇做半并行。**

### 哪两篇适合后续并行
- 07｜compact / compaction
- 08｜restore / session recovery

原因：
- 06 定完之后，07 的职责更清楚地收缩为“主动减负机制本体”
- 08 的职责更清楚地收缩为“恢复与续接 + 卷尾收束”
- 它们已经不再争“治理对象”这一层

### 本回合结束后要检查什么
- 06 是否已经把“视图治理”从“历史文本本体治理”里分开
- 07 是否已经守住主动减负机制，不再吞并 projection / collapse / recovery
- 08 是否既完成 recovery 说明，又完成卷尾收束
- 卷四是否最终压回“持续工作总图”，而不是停在机制百科

## 四、不推荐的做法

### 1. 不建议一开始就并发 01 / 02 / 03
这样最容易把卷四前半写成：
- 一个总论
- 一个术语页
- 一个 prompt / context 拼装页

结果看起来都像在讲“上下文”，但职责没有真正拆开。

### 2. 不建议把 04 / 05 分配给不同 agent 同时主写
这两篇分别承担：
- 为什么必须治理
- 治理链总图怎样成立

如果同时写，最容易互抢“治理总论”的角色。

### 3. 不建议 06 / 07 / 08 一起并行首稿
后半三篇虽然看上去像机制分工，但 06 的边界校正作用太强。

在 06 还没定之前，07 / 08 很容易带着错误治理对象开写。

### 4. 不建议把卷四写成“长上下文功能索引”
卷四真正要建立的是：
- 持续工作问题
- 当前工作面
- 治理必要性
- 治理链地图
- 恢复与续接

而不是罗列：
- compact 是什么
- projection 是什么
- collapse 是什么
- restore 是什么

## 五、playbook 化执行规则

这份文件不只是讨论摘要，而是卷四当前的**执行协议**。

后续如果要稳定跑卷四，默认按下面的 playbook 执行：

### 1. 启动格式
当主会话或执行会话收到类似下面的指令时：
- 跑卷四回合 1
- 继续卷四下一回合
- 按卷四 playbook 执行

默认含义是：
- 读取本文件
- 读取卷四 spec
- 读取卷四写作卡片
- 按本文件定义的顺序、串并行边界和回合检查来执行
- 不重新发散顺序，不临时改回合拆法

### 2. 每篇默认流水线
卷四每篇默认都走这条流水线：
1. 读取卷级 spec 与写作卡片
2. 按 skeleton-first 规则先建新骨架
3. 再从旧文抽素材
4. 需要时补源码锚点
5. 跑 `reader-friction-reviewer`
6. 本地 commit
7. 除非明确要求，否则不在单篇阶段单独 push

### 3. 每回合的交付格式
每回合结束后，执行会话应统一回报：
- 本回合完成了哪些文章
- 每篇文件路径
- 每篇一句核心判断
- 每篇提交 hash
- 每篇是否完成 reader review
- 本回合横向风险
- 是否允许进入下一回合

### 4. 回合级停止条件
即使用户要求尽量自动执行，卷四 playbook 仍然应在这些情况暂停：
- 前半骨架篇之间出现明显串线，导致后续文章无法稳定开写
- “当前可工作上下文”与“治理链”之间的边界没有立住
- 某篇持续滑向卷二、卷三、卷五或卷六
- 并行批次之间出现明显职责冲突
- git / push / 文件缺失 / 环境异常等技术阻塞

如果没有这些情况，就继续按 playbook 执行。

### 5. push 策略
默认不要求每篇单独 push。

更稳的默认方式是：
- 单篇完成后先本地 commit
- 每回合结束做一次横向检查
- 回合级确认结构没有明显串线后，再统一 push

### 6. 推荐执行形态
卷四默认推荐：
- **主会话负责定方向、接结果、做回合级判断**
- **一个主执行 subagent 负责按 playbook 连续生产整卷**
- **只有在 playbook 明确允许的阶段，主执行 subagent 才能继续派发子 subagent**

原因是：
- 卷四前半耦合高
- 后半又存在半并行结构
- 主会话不适合在长生产过程中被新消息频繁打断

### 7. 主执行 subagent 与子 subagent 的边界
推荐采用：

> **单主执行 subagent + 受控子 subagent**

具体约束如下：

#### 主执行 subagent 必须亲自完成的部分
- 回合 1：`01 / 02 / 03`
- 回合 2：`04 / 05`
- 回合 3 的第 1 步：`06`

这些部分不得下放主写，因为它们承担：
- 卷四总问题建立
- 核心对象边界校准
- 当前工作面构造
- 治理必要性建立
- 治理链总图建立
- 后半治理对象校正

#### 允许派发子 subagent 的部分
只有在 `06` 已完成且边界已稳定后，才允许派发子 subagent 并行处理：
- `07｜compact / compaction`
- `08｜restore / session recovery`

#### 子 subagent 默认负责什么
子 subagent 可以负责：
- 单篇起稿
- 单篇 `reader-friction-reviewer`
- 单篇至少一轮收紧
- 单篇本地 commit

#### 主执行 subagent 最终必须负责什么
即使使用子 subagent，主执行 subagent 仍必须负责：
- 检查 `07 / 08` 的边界是否清楚
- 检查卷尾是否真正回到“持续工作总图”
- 检查导航、上一页/下一页、卷内位置是否一致
- 统一做横向收口
- 统一提交最终修订
- 统一 push

---

## 六、一句话结论

卷四最稳的生产方式不是“直接并发试试看”，而是：

> **先串行打稳前 5 篇，先立持续工作骨架与治理链地图；再串行写 06 校正治理对象，最后对 07 / 08 做半并行收束。**
