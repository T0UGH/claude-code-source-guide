# 卷五 14｜runAgent 为什么像一条 agent runtime 装配线

## 导读

- **所属卷**：卷五：外部扩展与多代理能力
- **卷内位置**：14 / 25
- **上一篇**：[卷五 13｜Claude Code 是怎样长出更多执行者的](./13-how-claude-code-grows-more-executors.md)
- **下一篇**：[卷五 15｜为什么主 agent 还要派生 subagent](./15-why-the-main-agent-needs-to-spawn-subagents.md)

## 这篇要回答的问题

到了 Agent 主轴，如果只停在“系统有更多执行者”，还不够。

更关键的问题很快会冒出来：

> **这些执行者到底是怎样被装成可运行体的？**

这正是第 14 篇要回答的事。

如果这里没讲清，agent 仍然会像一个抽象名词；而 15 / 16 / 17 的 worker 后半段也会失去 runtime 主干。

所以第 14 篇必须把 runAgent 的位置立住：

> **runAgent 不是轻薄包装层，而是把 agent 定义、上下文、工具池、权限、skills、MCP、hooks 等能力面组装成可运行执行体的装配线。**

## 先给结论

### 结论一：runAgent 的价值不在“把 agent 跑起来”这么简单，而在“把执行者装起来”

如果只从函数名看，runAgent 很容易被理解成：

- 一个启动入口
- 一个调度包装层
- 一个把 agent 调起来的薄壳

但更稳的理解是：

> **它是 agent runtime 的装配主干。**

因为 agent 在进入真正执行前，要先被拼成一个可运行体。

### 结论二：agent 不是天然存在的成品，它要经过上下文、能力面和治理边界的正式组装

Claude Code 里的 agent 不是一句 prompt，也不是一个自由漂浮的角色名。

它真正进入 runtime 时，要连着很多东西一起成立：

- 它是谁
- 它继承什么上下文
- 它能看见哪些工具和外部能力
- 它受什么权限约束
- 它要不要带上 skills、MCP、hooks 等协作层

runAgent 的关键，正是在把这些分散要素装成一个真正可运行的执行者。

### 结论三：只有先把 runAgent 的位置立住，后面的 worker / fork 才不会像额外技巧

15 / 16 / 17 讲的是：

- 为什么还要派生 subagent
- forkSubagent 为什么不是再开一个 agent
- 主 agent 与 worker agent 怎样收边界、回流结果

这些都默认建立在第 14 篇的前提上：

> **先有一条 agent runtime 装配主干，后面才谈得上受控分叉。**

## 为什么说 runAgent 更像装配线

### 第一，因为它面对的不是单个动作，而是一整个执行体

工具执行更像“调用一次动作”。

runAgent 处理的则不是一个动作，而是一个执行体的成立。

这意味着它要回答的，不是“这个动作怎么跑”，而是：

- 这个 agent 的运行形态是什么
- 它当前携带哪些上下文
- 它能接哪些能力面
- 它怎样正式进入 Claude Code 的运行秩序

所以它天然更像装配，而不是调用。

### 第二，因为它要把分散要素收成一条可运行主干

Claude Code 的 agent 不是靠单一材料成立的。

它至少会碰到这些层：

- agent 定义
- 当前上下文与历史状态
- 工具池与可见能力面
- skills 带来的方法组织
- MCP 带来的外部能力面
- hooks 带来的运行时接缝
- 权限与约束边界

这些东西分开都能理解，但如果没有 runAgent 把它们编到一条主干里，agent 就仍然只是概念拼图。

runAgent 的系统价值，就在于它把这些零件正式收成一条执行装配线。

### 第三，因为它决定“agent 如何成为 Claude Code 里的执行者”，而不只是“agent 这个概念存在”

很多系统也会讲 agent。

但 Claude Code 这里更值得写的，不是有这个词，而是：

> **它怎样让 agent 真正在 runtime 里变成一个可治理、可运行、可继续分叉的执行者。**

runAgent 正好处在这个从“定义”到“运行”的转换处。

## runAgent 到底装了什么

这里不按源码函数顺序细讲，只按卷五所需的系统关系来讲。

### 1. 装 agent 定义：先确定“是谁在接这段工作”

首先得成立的，是 agent 本身的定义。

也就是：

- 这个执行者是什么角色
- 它承担什么任务类型
- 它的提示与职责边界是什么

这一层不成立，后面所有上下文和能力面都没有挂点。

### 2. 装上下文：把“在什么任务语境里工作”接进去

agent 不是裸跑的。

它要带着当前任务所处的语境工作。

这意味着 runAgent 必须处理：

- 当前会话相关上下文
- 必要的历史材料
- 承接上游执行状态所需的环境信息

这一步很关键，因为后面 fork worker 时，继承和分叉的讨论也都建立在这里。

### 3. 装能力面：把 tools / skills / MCP 接进当前执行体

agent 真正有用，不能只靠一段角色说明。

它还必须拥有可工作的能力面。

在卷五这张图里，这一步至少会连上：

- **tools**：动作执行能力
- **skills**：工作方法组织
- **MCP**：外部能力源与资源系统

这也正说明 runAgent 为什么不能写成一个薄壳：

它不是简单“调用 agent”，而是在把多层能力面装给当前执行者。

### 4. 装治理边界：把权限、hooks 和运行约束一起挂进去

Claude Code 不是让 agent 自由漂在系统外面做事。

它在 runtime 里仍然要接受治理：

- 权限边界
- 运行时接缝
- 可观察与可干预的位置
- 生命周期中的清理与退出

从卷五的写法看，这一层至少意味着：

- **hooks** 不只是旁观，而是 agent 运行时可接入的结构化接缝
- 权限不只是 tool 的问题，也会决定执行者能怎样工作

所以 runAgent 装的不是单纯能力，还有治理约束。

## 为什么第 14 篇必须放在 15 之前

因为如果没有第 14 篇，subagent / worker 很容易被写成“后来额外加出来的分叉功能”。

但真实的逻辑应该反过来：

- 先有 agent runtime 装配主干
- 然后才谈某个执行者在这条主干上继续分叉出 worker

这样第 15-17 篇才会自然成为同一条主线的后半段，而不是另一组并列对象。

## runAgent 为什么是 Agent 主轴的 runtime 主干

### 因为它把“更多执行者”从概念推进到机制

第 13 篇已经立住：Claude Code 会长出更多执行者。

第 14 篇要做的，就是把这件事从概念推进到 runtime 机制：

- 执行者不是抽象名词
- 执行者要被装起来
- 装配主干就是 runAgent

### 因为它给后半段的 worker 分叉提供了共同底座

后面的 subagent、fork、回流，都不是飘在空中的结构。

它们之所以能成立，是因为已经有一条共同的 agent runtime 主干。

从这个角度看，runAgent 不只是第 14 篇的主角，也是 15 / 16 / 17 的底座。

## 这篇不展开什么

### 1. 不把 fork worker 的特殊继承细节在这里讲完

那是第 16 篇的任务。

### 2. 不把主 agent 与 worker 的回流关系在这里讲完

那是第 17 篇的任务。

### 3. 不重写 skills / MCP / hooks 各自的独立正文

这里只说明它们如何被装进 agent 执行体，不重讲它们自己的对象主线。

## 和前后文的边界

### 它承接第 13 篇

第 13 篇回答：为什么 Claude Code 会长出更多执行者。

第 14 篇接着回答：这些执行者怎样被 runtime 正式装起来。

### 它导向 15 / 16 / 17

只有先把 runAgent 立成装配主干，后面的 subagent、fork worker、信息回流，才会自然被读成这条执行者主线的后半段。

## 一句话收口

> **runAgent 之所以像一条 agent runtime 装配线，不是因为它名字里有 run，而是因为 Claude Code 里的 agent 并不是天然可运行的成品；它必须把 agent 定义、上下文、工具池、skills、MCP、hooks、权限与运行约束正式装成一个执行体，而 runAgent 正是这条从“执行者定义”走向“执行者运行”的主干。**
