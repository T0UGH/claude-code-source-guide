# 卷五 07｜什么样的 skill 才算好的 runtime skill

## 这篇要回答的问题

前两篇已经把 skills 组最重要的两件事立住了：

- 第 05 篇说明 skills 接进来的是用户经验、流程和角色结构
- 第 06 篇说明 skills 会被 runtime 正式装进执行链

那接下来最自然的问题就是：

> **既然 skill 已经是 runtime 里的正式对象，那什么样的 skill 才算写得好？**

这里的“好”，不是文笔漂不漂亮，也不是篇幅长不长，而是：

- 它能不能稳定承担一个职责
- 它会不会把边界写糊
- 它进入 runtime 之后会不会变形
- 它是不是长期真的好用

## 先给结论

### 结论一：好的 runtime skill，不是写得花，而是职责稳定

一份 skill 真正值钱，首先不是因为它辞藻丰富，而是因为它能长期承担一个清楚的职责。

也就是说，看到这份 skill 时，系统和人都应该大致知道：

- 它解决什么问题
- 它不解决什么问题
- 它什么时候该出现
- 它完成后应交回什么结果

如果这些都说不清，再华丽的 prompt 也很难算好 skill。

### 结论二：好的 runtime skill，必须守住边界，而不是不断膨胀成“万能方法包”

很多写坏的 skill，问题不在能力太弱，而在野心太大。

它想同时承担：

- 解释
- 规划
- 执行
- 验收
- 风格指导
- 角色切换

最后就会变成什么都沾一点，什么都不稳。

而好的 runtime skill 更像一个边界清楚的工作单元。

### 结论三：好的 runtime skill，要在工作流里持续可用，而不是只在一次对话里看起来聪明

有些 prompt 第一次用会显得很惊艳，但第二次、第三次就开始漂：

- 触发条件不稳
- 输出差异很大
- 权限给得过胖
- 该 inline 还是 fork 经常混乱

这说明它还不是好的 runtime skill。

好的 runtime skill 应该追求的是长期稳定性，而不是一次性的“效果很好看”。

## 好 skill 的第一标准：职责要单一而明确

这是我认为最该先保住的一条。

一个好的 skill，最好能用一句话说清：

- 它是来干什么的
- 它为什么值得单独存在

比如它可能是：

- 在动手修 bug 前先做系统化排查
- 在做大改动前先做 brainstorming
- 在文章重写前先做 skeleton-first 骨架搭建
- 在声称“修好了”前先强制做验证

这些 skill 的共同点是：

- 都有清楚职责
- 都不需要把整个世界吞进去
- 都能稳定插入某一类工作流节点

职责一旦单一，skill 才容易：

- 被发现
- 被触发
- 被复用
- 被验收

## 第二标准：边界要清，不要和相邻对象互相吞并

卷五写到这里，边界感非常关键。

### 1. 不要把 skill 写成工具集合

skill 可以调用工具，但它自己不是工具集合目录。

如果一份 skill 的主要卖点变成“我能调很多工具”，那它通常已经开始偏了。

skill 更该负责的是：

- 什么时候用这些工具
- 先后顺序是什么
- 用完之后怎样组织结果

### 2. 不要把 skill 写成迷你 agent 宣言

skill 可以组织角色分工，甚至触发 fork，但它本身仍然不是执行者本体。

如果一份 skill 已经开始承担：

- 长时间独立自治
- 自己决定完整任务边界
- 把上下文主权整体接走

那它就很容易开始侵入 agent 的位置。

### 3. 不要把 skill 写成项目总宪法

还有一种常见问题，是把很多项目规则一股脑塞进一个 skill，仿佛它想管理一切。

这种 skill 读起来很完整，跑起来却很重。

因为 runtime 更喜欢的是：

- 明确职责
- 稳定边界
- 在某一类节点上可靠插入

而不是一份包打天下的总章程。

## 第三标准：正文要像工作流，不要像散文

好 skill 的正文，最怕两种极端：

- 全是理念，没有动作
- 全是口号，没有顺序

真正稳定的 runtime skill，正文通常会很像工作流。

至少应该告诉模型：

1. 目标是什么
2. 先做什么
3. 再做什么
4. 缺信息怎么办
5. 最后产物长什么样

为什么这件事这么重要？

因为 skill 一旦进入 runtime，模型真正需要的是：

- 可执行的动作框架
- 可判断的完成标准

而不是“请认真思考”“尽量全面分析”这种作者自我安慰式话术。

## 第四标准：高影响字段要克制

一份 skill 真正危险的地方，常常不在正文，而在它对 runtime 行为的改动过重。

例如：

- 权限给得过宽
- 动不动就切 fork
- 轻易改 model / effort
- 把 hooks、paths 都先挂上

这些设置不是装饰，而是真的会改变运行行为。

所以好的 runtime skill 通常有一种共同气质：

> **只声明真正必要的运行时影响。**

### 这背后的判断很简单

- 能 inline 就别轻易 fork
- 能用最小权限就别预支更多权限
- 不需要条件激活，就别乱加 paths
- 不需要显式改模型，就别乱写 model / effort

runtime 里很多不稳定，往往不是因为 skill 不够强，而是因为它写得太贪。

## 第五标准：要有可验收的完成判据

一份 skill 如果没有完成判据，就很容易出现这种情况：

- 模型交了一个“看起来差不多”的结果
- 但主线程其实不知道这算不算完成
- 下一步也不知道该继续还是回退

所以好的 runtime skill，最好都能回答：

- 这一步做到什么算完成
- 最终要交回什么
- 哪些不确定点必须显式保留

这种完成判据，未必非要写成死板 checklist，但一定要能被执行链感知到。

否则 skill 再漂亮，也更像一段意见，而不像可稳定调用的工作单元。

## 第六标准：要经得起重复使用，而不是只在第一次显灵

我会把这一点当成真正的 runtime 标准。

因为很多 skill 第一次看没问题，但重复用几次就暴露问题：

- 作用范围太飘
- 不同场景下开始乱抢职责
- 输出风格忽胖忽瘦
- 用户稍微换个说法就不稳

而一个好的 runtime skill，应该有足够的抗波动性：

- 换一篇文章，仍然能稳
- 换一个相邻任务，仍然不越界
- 换一次上下文密度，仍然能守住主职责

这才叫“能进 runtime”。

## 从卷五结构看，第 07 篇为什么重要

skills 组如果只有第 05 篇和第 06 篇，读者会知道：

- skill 接进来的是什么
- skill 怎样进入执行链

但还缺一层最实际的判断：

> **不是所有 skill 都同样适合 runtime。**

第 07 篇要补的，就是这层质量判断。

它要把“好 skill”从写作审美，抬升成运行时判断：

- 是否职责稳定
- 是否边界清楚
- 是否路径克制
- 是否长期可复用

这样第 08 篇去切 skill / tool / agent 边界时，地基才会稳。

## 这篇不展开什么

### 1. 不把 frontmatter 字段讲成逐项手册

那会滑向卷六接口层。

### 2. 不把 skillify 等样本细讲成源码分析文

这里要的是质量判断，不是案例拆解。

### 3. 不替第 08 篇完成边界总收口

这里只讲“好 skill”，不把 skill / tool / agent 的边界一次讲完。

## 和前后文的边界

### 它承接第 06 篇

第 06 篇说明 skill 已经是执行链中的正式对象。

第 07 篇进一步回答：既然如此，什么样的 skill 才配得上这个位置。

### 它导向第 08 篇

一旦质量标准立住，第 08 篇就更容易把边界切清：

- tool 是什么
- skill 是什么
- agent 是什么

## 一句话收口

> **好的 runtime skill，不是写得最漂亮的 prompt，而是一个职责单一、边界清楚、运行影响克制、正文可执行、完成判据明确、并且能在工作流里长期稳定复用的能力单元。**
