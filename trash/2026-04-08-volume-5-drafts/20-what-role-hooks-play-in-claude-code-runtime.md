# 卷五 20｜Claude Code 的 hooks 在 runtime 里到底扮演什么角色

- **上一篇**：[卷五 19｜为什么平台层不仅要有能力对象，还要有运行时接缝](./19-why-platform-layer-needs-runtime-seams-not-just-capability-objects.md)
- **下一篇**：[卷五 21｜Claude Code 里的各类 hooks 分别在拦什么、接什么、改什么](./21-what-different-hooks-intercept-connect-and-modify-in-claude-code.md)

## 这篇要回答的问题

到了 hooks 组，最容易犯的错有两种：

- 把 hooks 写成“用户可以配的一些自动化脚本”
- 把 hooks 写成“围着 tool 跑的几个回调点”

这两种说法都抓到了一点表面，但都还不够。

因为卷五这里真正要回答的是：

> **hooks 在 Claude Code runtime 里，到底扮演什么角色？**

如果这个问题不先立住，第 21 篇就很容易滑成类型列表；而 hooks 整组也会被写轻。

所以第 20 篇必须先把 hooks 的系统位置立稳：

> **hooks 不只是额外插入动作，而是平台层留给运行时观察、注入和干预的结构化接缝。**

## 先给结论

### 结论一：hooks 不是在系统外面挂几个脚本，而是在 runtime 里面留出正式可插手的事件点

一个系统如果只有能力对象，通常意味着：

- 有工具可调用
- 有方法可组织
- 有执行者可承担
- 有外部能力可接入

但这还没有回答另一类问题：

- runtime 在哪些节点允许额外逻辑看见正在发生什么
- 哪些节点允许补上下文、改写输入、影响权限、修正输出
- 哪些地方允许系统在继续前被额外检查或干预

hooks 解决的正是这一层。

### 结论二：hooks 的重点不在“又多了一个功能”，而在“平台层对运行过程开放了接缝”

如果只把 hooks 看成功能点，读者最后会留下：

- 哦，可以在某些地方跑脚本
- 哦，可以在工具前后做点事

但卷五要留下来的模型更强：

> **Claude Code 不只是拥有能力对象，它还把运行过程本身做成了可观察、可注入、可干预的结构。**

这正是 hooks 的系统意义。

### 结论三：hooks 站在运行时接缝层，不等于 skills、MCP 或 plugins

从卷五的对象地图看：

- skills 组织方法
- MCP 接入外部能力源
- agent 组织执行者结构
- hooks 打开运行时接缝
- plugins 负责更完整的封装与治理

所以 hooks 的独立价值，不在它也能“做点事”，而在它负责的是：

> **runtime 在哪里允许被插手。**

## 先把角色说清：这篇讲什么，不讲什么

这篇是 hooks 组的 **锚点篇**。

它要完成三件事：

1. 讲清 hooks 在 runtime 里的系统位置
2. 讲清它为什么是接缝层，而不是附属脚本功能
3. 给第 21 篇的分类细拆铺稳入口

这篇**不**展开：

- 不写成 hook 类型列表
- 不细讲每个配置字段
- 不提前吃掉 plugins 的封装层问题
- 不转去讲卷六命令入口

## 为什么 hooks 必须被理解成“运行时接缝层”

### 第一，因为平台化不只需要对象，还需要在运行过程里开放正式插口

Claude Code 前面已经长出了很多对象：

- tool
- skill
- MCP
- agent

这些对象回答的是“系统有什么能力、怎么组织、谁来承担”。

但平台一旦继续长，就会遇到一个新问题：

- 运行过程本身能不能被看见和影响？

如果这个问题没有答案，系统仍然偏封闭。

hooks 正是对这个问题的回答。

### 第二，因为 hooks 面向的不是某个能力对象，而是 runtime 中正在发生的事件

skills 的主语更像工作方法。
MCP 的主语更像外部能力源。
agent 的主语更像执行者。

hooks 的主语不同。

它关心的是：

- 某个输入即将进入处理时
- 某个工具即将执行或刚执行完时
- 某个会话进入或结束时
- 某次继续、停止、恢复或补偿发生时

这说明 hooks 天然不是对象目录里的另一类能力，而是围绕**事件点**建立的运行时接缝。

### 第三，因为 hooks 的输出不是随便漂在系统外面，而是会重新进入 runtime 语义

如果 hooks 只是外面跑一段脚本，然后打印点日志，它仍然很像外围增强功能。

但 Claude Code 这里更重要的一点是：

- hook 的结果会被系统吸收
- 它可能补上下文
- 可能改写输入
- 可能影响权限判断
- 可能修改工具输出
- 可能影响是否继续某段运行

这说明 hooks 的作用，不是旁路观察，而是正式进入 runtime 后处理与控制结构。

## hooks 在 runtime 里到底扮演什么角色

### 1. 它是观察点：让系统运行过程变得可看见

先看最轻的一层。

Claude Code 在很多关键时刻，不只是自己往前跑，还允许 hook 看见“当前正在发生什么”。

这层价值是：

- 把原本封闭的运行时节点显露出来
- 让外部逻辑能知道当前到了哪个阶段
- 让平台层不只剩对象调用，还带有事件语义

如果没有这层，runtime 仍然更像黑箱。

### 2. 它是注入点：让额外上下文和判断可以在关键节点进入主线

hooks 更重要的一层，不只是看见，而是注入。

也就是说，在某些运行节点上，系统允许：

- 补充额外上下文
- 修整当前输入
- 把外部判断带进当前流程

这让 Claude Code 的运行主线不再只能依赖固定内建逻辑，而能在关键接缝处吸收额外信息。

### 3. 它是干预点：让 runtime 的部分决策可以被修改、阻断或校正

这层是 hooks 真正重量所在。

在 Claude Code 里，hooks 不是只能“顺手做点事”，而是可能真正影响：

- 某一步是否继续
- 某个输入是否需要被改写后再进入系统
- 某个权限决策如何落定
- 某次工具结果是否要被修整后再回主线

所以 hooks 不是装饰层，而是正式的运行时干预层。

## 为什么这不是“工具前后回调”那么简单

### 因为 hooks 不只卡在工具执行主链上

工具前后确实是 hooks 很容易被看见的地方。

但如果只停在这里，hooks 会被写得太窄。

卷五这里必须让读者留下更完整的理解：

- 它不只碰工具链
- 它也碰输入、权限、会话生命周期、继续/停止等运行节点
- 它面对的是整个 runtime 的若干接缝，而不是单一工具链两侧

所以“工具回调”只是 hooks 的一部分，不是它的总定义。

### 因为 hooks 的语义是结构化的，不是松散外挂

Claude Code 对 hooks 的处理，并不是“用户配了什么脚本就随便跑”。

更重要的是，它会把：

- 什么事件允许挂 hook
- hook 可以返回什么类型的影响
- 这些影响怎样被 runtime 吸收

都收成正式语义。

这正说明 hooks 是结构化接缝，而不是配置区杂物间。

## hooks 和其它对象的关系，为什么这里必须先讲一遍

### hooks 不是 skill

skill 关心的是这类工作怎么组织得更稳。

hooks 关心的是某个运行节点允许怎样被观察、注入和干预。

前者是方法层，后者是接缝层。

### hooks 不是 MCP

MCP 解决的是外部能力源怎样正式进入系统。

hooks 解决的是 runtime 在哪里允许插手。

前者接能力边界，后者开运行时接缝。

### hooks 也不是 plugin

plugin 讲的是更完整的封装、安装、分发和治理单元。

hooks 本身并不等于这种统一封装层。

它当然可以被 plugin 承载，但它自身解决的是接缝问题，不是封装问题。
