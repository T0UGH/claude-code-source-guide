## 这篇要回答的问题

第 19 / 20 篇已经先把 hooks 组的两个前提立住了：

到了第 21 篇，就不能再停在总论。

这篇必须继续往下拆：

> **Claude Code 里的不同 hooks 类型，到底分别在拦什么、接什么、改什么？**

如果这篇只说“hooks 很重要”，那 hooks 组还是不够细。

但如果把它写成 API 手册，又会直接跑偏。

所以第 21 篇的任务，是在不掉进配置索引的前提下，把 hooks 的几类典型接缝职责讲清楚。

## 先给结论

### 结论一：不同 hooks 的差异，不在名字，而在它们各自卡住的是 runtime 的哪一段接缝

同样都叫 hook，但它们不在同一个位置上发力。

有的更靠输入前后，
有的更靠工具执行前后，
有的更靠会话进入和结束，
有的更靠运行收口和继续判断。

所以 hooks 真正的价值，不在“有很多 hook”，而在：

> **不同接缝点上允许的是不同种类的观察、注入和干预。**

### 结论二：有些 hooks 主要在“拦”，有些主要在“接”，有些主要在“改”

为了便于读者留下模型，可以先把几类 hooks 粗分成三种动作倾向：

- **拦**：在继续前先检查、先裁定、先决定能不能过
- **接**：在某个节点补上下文、接回信息、把额外逻辑接入主线
- **改**：对输入、结果或后续行为做正式改写

现实里某个 hook 可能同时做两三件事，但这三个词足够帮助读者抓住差异。

### 结论三：Claude Code 的 hooks 不是同一种脚本在不同时间触发，而是一组面向不同 runtime 语义的正式接缝

这篇最后要留下来的最稳判断是：

> **hooks 不是一组同质配置，而是一套按不同 runtime 节点分工的接缝系统。**

## 先搭一个分类骨架

为了不写成索引，我会先按卷五可保留的抽象层来分。

Claude Code 里的 hooks，大致可以从四类接缝来看：

1. **输入接缝**：用户输入或采样前后发生的接缝
2. **工具执行接缝**：某个工具即将执行、刚执行完时的接缝
3. **会话生命周期接缝**：会话进入、恢复、结束等时刻的接缝
4. **运行收口 / 继续判断接缝**：系统决定是否继续、停止、补偿时的接缝

这四类足够覆盖 hooks 在卷五该留下来的细度。

## 第一类：输入接缝——主要在“接”和“改”

### 它们主要拦在什么地方

输入接缝，卡住的是：

- 某次输入正式进入主循环之前
- 某些采样前后的关键点
- 系统准备把当前输入继续送进后续运行前

也就是说，它们面对的是“这一轮将要怎么开始”。

### 它们主要接什么

这一类 hooks 很适合接进来的，是：

- 额外上下文
- 环境信息
- 对当前输入的补充解释
- 某种应在进入主线前就附着上的判断

所以从“接什么”的角度看，它们更像是在给这一轮运行补 runtime 语境。

### 它们主要改什么

输入接缝最典型的动作之一，就是改输入。

例如卷四旧材料里已经暴露出的语义词：

- `updatedInput`

这类语义很关键，因为它说明 Claude Code 并不是把 hook 输出当松散日志，而是允许它在合法边界内正式改写将要进入主线的输入。

所以输入接缝最短可以记成：

> **在开跑前先补语境、修输入。**

## 第二类：工具执行接缝——主要在“拦”和“改”

### 它们主要拦在什么地方

这一类 hooks 最容易被看到，因为位置很直观：

- 工具执行前
- 工具执行后

也就是常说的 PreToolUse / PostToolUse 一类接缝。

### 它们主要接什么

工具接缝接进来的，不只是简单日志。

更关键的是：

- 工具执行时的运行状态
- 当前工具调用所处的判断上下文
- 某些与本次调用强相关的额外信息

这一层让工具调用不再像彻底封闭的黑箱动作。

### 它们主要改什么

工具执行接缝特别值得讲的一点，是它们不只在旁边看，还可能改：

- 是否允许继续执行
- 权限裁定如何落下
- 工具输出是否需要修整后再回主线

旧材料里已经能看到一些很典型的语义：

- `permissionDecision`
- `permissionDecisionReason`
- `updatedMCPToolOutput`

这些词本身已经说明，工具接缝不是“围观工具”，而是在工具执行主链上正式插手。

所以工具执行接缝最短可以记成：

> **在工具链上先审、再放、必要时再修。**

## 第三类：会话生命周期接缝——主要在“接”和“收”

### 它们主要拦在什么地方

会话生命周期接缝，卡住的是：

- SessionStart
- resume / recovery 再进入
- SessionStop 或结束收口

也就是说，它们处理的不是某一个局部动作，而是会话阶段的进入和退出。

### 它们主要接什么

这一类 hooks 很适合接入的，是：

- 会话开始时需要补的上下文
- 恢复时要重新带回的运行时语义
- 某些在阶段边界处必须重新声明的环境信息

从卷三恢复线能看到，这类 hooks 甚至会和 resume 重新进入主循环、补回 hook 附加信息一类问题连在一起。

所以它们的“接”，更像是在给整个会话阶段补运行语义。

### 它们主要改什么

生命周期 hooks 不一定总是在“改文本”，但它们会改变：

- 一个会话阶段带着什么上下文进入
- 一个阶段结束时系统怎样收口
- 恢复时哪些附加信息需要重新被吸收

因此这一类更像：

> **在阶段边界上补语境、做收口。**

## 第四类：运行收口 / 继续判断接缝——主要在“拦”

### 它们主要拦在什么地方

这是 hooks 里最能体现重量的一类。

它们卡住的不是单个输入或单个工具，而是：

- 系统准备继续还是停止的判断点
- 某轮执行要不要在这里收束
- 某些补偿或后处理完成后能不能真正结束

换句话说，它们碰到的是 runtime 的收口处。

### 它们主要接什么

这一类 hooks 不一定主要补上下文，但它们会接进来额外判断：

- 当前是否允许结束
- 是否还需要继续后处理
- 某些停止条件是否应被覆盖或延后

这说明 hooks 不只是挂在过程里，它还深入到了“何时算这一轮真正完成”的地方。

### 它们主要改什么

这类 hooks 最典型的改动，不是改字面输入输出，而是改运行走向：

- 让本来要停的地方继续
- 让本来能过的地方先别过
- 让系统在真正结束前再经过额外检查或收口

所以这一类最短可以记成：

> **在收口点决定这一轮能不能真的收。**

## 再把“拦 / 接 / 改”三件事压成一张小图

如果只保留卷五最需要的记忆点，不同 hooks 类型可以先这样压缩：

- **输入接缝**：主要接上下文、改输入
- **工具执行接缝**：主要拦执行、裁权限、修结果
- **会话生命周期接缝**：主要接阶段语义、做进入与退出收口
- **运行收口接缝**：主要拦继续/停止判断，影响一轮是否真正结束

这样读者就不会把 hooks 只记成“前后跑脚本”。

## 为什么第 21 篇不能写成 API 手册

因为卷五的任务不是告诉读者每个字段怎么填，而是让读者看见：

> **不同 hooks 类型分别卡在不同 runtime 接缝上，因此承担的是不同干预职责。**

如果这里写成配置索引，读者最后只会记住：

- hook 有几类
- 各自配置项长什么样

却记不住最重要的系统模型。

所以第 21 篇必须以“接缝职责”来分类，而不是以“配置名字”来分类。

## hooks 为什么因此不是插件组的提前版

这也需要顺手收一下边界。

第 21 篇虽然已经开始细拆 hooks 类型，但它仍然只在回答：

- 接缝在哪
- 各类接缝干什么
- 它们怎样影响 runtime

它还没有去回答：

- 这些扩展如何被统一封装
- 怎样安装、启停、治理和分发

那些是 plugins 组的职责。

所以 hooks 细拆得再细，也仍然是接缝层，不是封装层。

## 这篇不展开什么

### 1. 不写每个 hook 的完整配置参考

这里保留的是职责模型，不是 API 文档。

### 2. 不把 plugins 的打包治理问题提前讲完

这里只说明 hooks 自己的分类，不讲它们怎样被统一承载。

### 3. 不把每一类 hook 的源码细节链路全展开

本篇重点是类型职责，不是源码考据。

## 一句话收口

> **Claude Code 里的不同 hooks 类型，真正差别不在名称，而在它们各自卡住 runtime 的哪一段接缝：有的在输入前后补语境、改输入，有的在工具链上拦执行、裁权限、修结果，有的在会话边界接入阶段语义、做进入退出收口，还有的在继续/停止判断点决定这一轮能不能真正结束；hooks 的价值，正是在这些不同接缝上承担不同的干预职责。**
